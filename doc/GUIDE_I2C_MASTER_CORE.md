# Проектирование I2C Master Core с нуля

## Введение

Этот документ — подробное руководство по проектированию аппаратного I2C мастер-контроллера на уровне бит и байт. Мы пройдём путь от спецификации протокола до готового Verilog-модуля `i2c_master_core`, объясняя каждое проектное решение.

**Что делает этот модуль:** принимает высокоуровневые команды (START, WRITE byte, READ byte, STOP, RESTART) и генерирует корректные сигналы SCL/SDA на шине I2C, соблюдая все требования протокола.

**Чего он НЕ делает:** не содержит регистров управления, прескалера, прерываний. Это чистое «ядро» — его оборачивают в AXI, Avalon или другой интерфейс.

**Уровень:** для тех, кто понимает основы Verilog и конечных автоматов, но не проектировал I2C ранее.

## Часть 1. Что нам нужно реализовать

### 1.1. Спецификация I2C (NXP UM10204) — ключевые требования

Из спецификации I2C вытекают жёсткие правила, которые определяют архитектуру ядра:

**Правило 1. Open-drain выходы.**
Мастер может только **притянуть линию к 0** или **отпустить** (high-Z). Подать логическую единицу на линию он не может — за это отвечает внешний pull-up резистор.

**Правило 2. SDA меняется только при SCL=0.**
Во время передачи данных SDA может переключаться только когда SCL удерживается мастером в 0. Если SDA меняется при SCL=1 — это условие START или STOP.

**Правило 3. START = SDA 1→0 при SCL=1. STOP = SDA 0→1 при SCL=1.**

**Правило 4. Данные передаются MSB first (старший бит первым).**

**Правило 5. Каждый байт = 8 бит данных + 1 бит ACK/NACK (всего 9 бит-слотов).**

**Правило 6. Clock stretching.** Slave может удерживать SCL в 0, чтобы попросить мастера подождать. Мастер обязан проверять, что SCL действительно поднялся, прежде чем продолжать.

**Правило 7. Arbitration.** Если несколько мастеров на шине, мастер, который отпустил SDA (ожидает 1), но видит 0 — потерял арбитраж и должен немедленно освободить шину.

### 1.2. Интерфейс команд — что видит «внешний мир»

Наше ядро должно предоставить простой командный интерфейс:

```
┌──────────────────┐       ┌────────────────────┐
│  Вышестоящий     │       │  i2c_master_core    │
│  контроллер      │       │                     │
│                  ├──────►│ cmd_valid_i          │
│  (секвенсер,     │ cmd   │ cmd_i[2:0]          │
│   CPU,           ├──────►│ din_i[7:0]          │       ┌──────┐
│   тестовый FSM)  │       │                     │ SDA ◄─►│Slave │
│                  │◄──────┤ ready_o              │ SCL ──►│      │
│                  │◄──────┤ dout_o[7:0]          │       └──────┘
│                  │◄──────┤ rx_ack_o             │
└──────────────────┘       └────────────────────┘
```

**Команды (5 штук):**

| Код | Команда | Смысл |
|-----|---------|-------|
| 1 | START | Генерировать условие START |
| 2 | WRITE | Передать байт из `din_i`, получить ACK |
| 3 | READ | Принять байт от slave, отправить ACK/NACK |
| 4 | STOP | Генерировать условие STOP |
| 5 | RESTART | Генерировать повторный START (без предварительного STOP) |

**Протокол рукопожатия:**

```
         ┌─────┐                   ┌─────┐
ready_o  │     │                   │     │
     ────┘     └───────────────────┘     └────
                                   ↑
         ┌─────────────────┐       │ Команда завершена,
cmd_valid│                 │       │ можно читать dout_o/rx_ack_o
     ────┘                 └───────┘

              ↑ Ядро приняло команду
              │ (ready_o упал)
```

1. Внешний контроллер ждёт `ready_o = 1`
2. Выставляет `cmd_valid_i = 1`, `cmd_i`, `din_i`
3. Ядро видит запрос, сбрасывает `ready_o = 0` — начинает работу
4. Контроллер может убрать `cmd_valid_i`
5. Ядро генерирует I2C-сигналы...
6. Ядро поднимает `ready_o = 1` — команда завершена
7. `dout_o` содержит принятый байт (для READ), `rx_ack_o` — полученный ACK

## Часть 2. Ключевое проектное решение — 4-фазная модель бита

### 2.1. Зачем нужны фазы

Один бит I2C — это один полный период SCL. Но внутри этого периода происходят разные вещи: подготовка данных, установка SCL, чтение данных, снятие SCL. Нам нужно разделить бит на подэтапы.

Самое естественное деление — **на 4 четверти** периода SCL:

```
         Фаза 0      Фаза 1      Фаза 2      Фаза 3
        ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
        │         │ │         │ │         │ │         │
SCL:  ──┘ LOW     └─┘  HIGH   └─┘  HIGH   └─┘  LOW    └──
                      ↑                        ↑
SDA:  ══╤════════════╤═══════════╤════════════╤═════════
        │            │           │            │
    Установить    Slave может  Данные        Подготовка
    SDA (данные   растянуть    стабильны,    к следующему
    или release)  клок тут     семплируем    биту
```

**Почему именно 4 фазы?**

- **Фаза 0 (SCL=LOW, setup):** Безопасно менять SDA — SCL низкий, изменение SDA не создаст ложный START/STOP.
- **Фаза 1 (SCL→HIGH, sample):** Отпускаем SCL. Проверяем, что SCL действительно поднялся (clock stretching!). Семплируем SDA.
- **Фаза 2 (SCL=HIGH, hold):** Держим SCL высоким. Данные стабильны. Дополнительное время для slave.
- **Фаза 3 (SCL→LOW, shift):** Притягиваем SCL к 0. Продвигаем сдвиговый регистр. Инкрементируем счётчик бит.

Это деление применяется единообразно ко ВСЕМ типам бит (данные, ACK, START, STOP, RESTART), что упрощает архитектуру.

### 2.2. Тактирование фаз — сигнал ena_i

Ядро тактируется основным клоком FPGA (например, 50 МГц), но фазы продвигаются по сигналу `ena_i` — одноцикловому импульсу от внешнего прескалера.

```
clk_i:  ┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐...  (50 MHz)

ena_i:  _│‾│__________________________________│‾│______...
         ↑                                     ↑
       Фаза N                               Фаза N+1
       (125 тактов между ними для SCL = 100 кГц)
```

**Почему так?** Это разделяет задачи:
- Прескалер (за пределами ядра) — отвечает за скорость I2C
- Ядро — отвечает за логику протокола

Ядро не знает и не заботится о конкретной частоте SCL. Оно просто продвигается на один шаг при каждом `ena_i`.

## Часть 3. Архитектура FSM — конечный автомат ядра

### 3.1. Состояния верхнего уровня

```
                    ┌──────────┐
            ┌──────►│  IDLE    │◄──────┐
            │       └────┬─────┘       │
            │            │             │
            │    cmd_valid_i?          │
            │     │    │    │    │     │
            │   START WRITE READ STOP  RESTART
            │     │    │    │    │     │
            │     ▼    ▼    ▼    ▼     ▼
            │  ┌──────┐ ┌──────┐ ┌───────┐ ┌──────┐ ┌─────────┐
            │  │START │ │ DATA │ │ DATA  │ │ STOP │ │ RESTART │
            │  │(4 ф) │ │(9×4ф)│ │(9×4ф) │ │(4 ф) │ │ (4 ф)  │
            │  └──┬───┘ └──┬───┘ └──┬────┘ └──┬───┘ └────┬────┘
            │     │        │        │         │          │
            └─────┴────────┴────────┴─────────┴──────────┘
                            ready_o = 1
```

Всего **5 состояний** (не считая IDLE):

| Состояние | Фаз | Бит-слотов | Итого фаз | Назначение |
|-----------|-----|-----------|-----------|------------|
| ST_IDLE | — | — | — | Ожидание команды |
| ST_START | 4 | 1 | 4 | Условие START |
| ST_DATA | 4 | 9 | 36 | Передача/приём байта + ACK |
| ST_STOP | 4 | 1 | 4 | Условие STOP |
| ST_RESTART | 4 | 1 | 4 | Повторный START |

### 3.2. Почему WRITE и READ используют одно состояние ST_DATA?

Это ключевое упрощение. На уровне шины WRITE и READ отличаются только одним: **кто управляет SDA** в каждом бит-слоте.

```
WRITE (мастер передаёт):
  Бит 0-7: мастер выдаёт → SDA = tx_shift_r[8]  (MSB first)
  Бит 8:   slave отвечает → SDA = release (high-Z), читаем ACK

READ (slave передаёт):
  Бит 0-7: slave выдаёт → SDA = release (high-Z), читаем данные
  Бит 8:   мастер отвечает → SDA = 0(ACK) или 1(NACK)
```

Это можно выразить одним сигналом `sda_input_mode`:

```verilog
wire sda_input_mode = (state_r == ST_DATA) && (
    (cmd_r == CMD_READ  && bit_cnt_r < 4'd8) ||  // READ биты 0-7: slave гонит
    (cmd_r == CMD_WRITE && bit_cnt_r == 4'd8)     // WRITE бит 8: slave ACK
);
```

Когда `sda_input_mode = 1` — мастер отпускает SDA и слушает. Когда `0` — мастер выдаёт бит из сдвигового регистра.

Один FSM-стейт вместо двух = меньше логики, меньше багов, проще верификация.

## Часть 4. Проектирование каждого состояния

### 4.1. ST_IDLE — ожидание команды

**Что делает:** ждёт, пока вышестоящий контроллер не выставит команду.

**Тонкость #1: управление линиями в IDLE.**

Когда ядро в IDLE, ему нужно решить, что делать с SCL и SDA. Есть три ситуации:

```
1. Шина свободна (после STOP или при включении):
   → Отпустить обе линии (высокий через pull-up)
   → scl_oen = 1, sda_oen = 1

2. Шина занята, команда ожидается (между START и STOP):
   → Держать линии как есть! НЕ отпускать!
   → Если отпустить SDA при SCL=1, получится ложный STOP

3. Новая команда пришла:
   → Защёлкнуть команду, перейти в нужное состояние
```

Это выражается так:

```verilog
ST_IDLE: begin
    if (cmd_valid_i && !arb_lost_o) begin
        ready_o <= 1'b0;       // Начинаем работу
        case (cmd_i)
            CMD_START:   ... → ST_START
            CMD_WRITE:   ... → ST_DATA
            CMD_READ:    ... → ST_DATA
            CMD_STOP:    ... → ST_STOP
            CMD_RESTART: ... → ST_RESTART
        endcase
    end else if (!busy_o) begin
        scl_oen_o <= 1'b1;    // Шина свободна — отпустить
        sda_oen_o <= 1'b1;
    end
    // Если busy_o=1 но нет команды — ничего не делаем, держим линии
end
```

**Ошибка, которую легко допустить:** безусловно отпускать линии в IDLE. Это создаёт ложные STOP между командами составной транзакции (START + WRITE + WRITE + STOP). Между WRITE и WRITE ядро ненадолго попадает в IDLE (ждёт следующую команду), и если отпустит SDA при SCL=1 — slave увидит STOP.

**Тонкость #2: загрузка сдвигового регистра.**

При приёме команды WRITE или READ сразу загружаем данные в сдвиговый регистр:

```verilog
CMD_WRITE: begin
    tx_shift_r <= {din_i, 1'b0};    // 8 бит данных + 0 (мастер ACK не нужен)
    cmd_r      <= CMD_WRITE;
    bit_cnt_r  <= 4'd0;
end
CMD_READ: begin
    tx_shift_r <= {8'hFF, din_i[0]}; // FF=отпустить SDA + бит NACK
    cmd_r      <= CMD_READ;
    bit_cnt_r  <= 4'd0;
end
```

Для WRITE: `tx_shift_r = {D7,D6,D5,D4,D3,D2,D1,D0, 0}` — 8 бит данных, потом 0 (будем слушать ACK).

Для READ: `tx_shift_r = {1,1,1,1,1,1,1,1, NACK}` — 8 единиц (отпустить SDA, slave гонит данные), потом бит NACK/ACK который мастер пошлёт.

### 4.2. ST_START — генерация условия START

**Определение START:** SDA переходит из 1 в 0, пока SCL = 1.

**Тайминг по фазам:**

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        START!
SDA: ────────────────────────────────╲
                                      ╲────────────────────────
SCL: ────────────────────────────────────────────────╲
                                                      ╲────────

     SDA=1, SCL=1      SDA=1, SCL=1   SDA=0, SCL=1   SDA=0, SCL=0
     (ждём SCL high     (hold)         (START!)        (готово)
      если stretching)
```

**Реализация:**

```verilog
ST_START: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b1;         // SDA отпущена (high)
            scl_oen_o <= 1'b1;         // SCL отпущена (high)
            if (scl_i) phase_r <= 2'd1; // Ждём, пока SCL реально станет HIGH
        end
        2'd1: begin
            sda_oen_o <= 1'b1;         // SDA всё ещё high
            scl_oen_o <= 1'b1;         // SCL high — hold time
            phase_r   <= 2'd2;
        end
        2'd2: begin
            sda_oen_o <= 1'b0;         // SDA → LOW !!! ← ВОТ ОН, START
            scl_oen_o <= 1'b1;         // SCL всё ещё high
            phase_r   <= 2'd3;
        end
        2'd3: begin
            sda_oen_o <= 1'b0;         // SDA остаётся low
            scl_oen_o <= 1'b0;         // SCL → LOW (подготовка к данным)
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;         // Команда выполнена
        end
    endcase
end
```

**Почему в фазе 0 проверяем `scl_i`?** Clock stretching. Если предыдущая команда оставила SCL низким (slave держит), нужно дождаться, пока slave отпустит. `scl_i` — это реальный уровень на пине. Пока slave держит SCL=0, мы стоим в фазе 0.

### 4.3. ST_DATA — передача/приём 9 бит

Это самое сложное состояние. Оно обрабатывает 9 бит-слотов: 8 бит данных + 1 бит ACK. Каждый бит-слот проходит через 4 фазы.

**Фаза 0: Setup SDA (SCL=LOW)**

```verilog
2'd0: begin
    scl_oen_o <= 1'b0;               // SCL LOW — безопасно менять SDA
    if (sda_input_mode)
        sda_oen_o <= 1'b1;           // Slave гонит → отпускаем SDA
    else
        sda_oen_o <= tx_shift_r[8];  // Мастер гонит → выдаём MSB
    phase_r <= 2'd1;
end
```

**Что такое `tx_shift_r[8]`?** Это старший бит сдвигового регистра. При `oen = 0` (бит = 0) мастер притягивает SDA к 0 (передаём «0»). При `oen = 1` (бит = 1) мастер отпускает SDA, pull-up подтягивает к 1 (передаём «1»).

По сути, `sda_oen_o = tx_shift_r[8]` — это XOR-инверсия: мы НЕ выдаём данные напрямую, а управляем тристейт-буфером. `oen = 1` → линия отпущена → pull-up → единица. `oen = 0` → линия притянута → ноль.

**Фаза 1: Release SCL, sample SDA**

```verilog
2'd1: begin
    scl_oen_o <= 1'b1;               // Отпускаем SCL → HIGH (через pull-up)
    // SDA продолжаем удерживать как в фазе 0

    if (scl_i) begin                  // SCL реально HIGH? (clock stretching!)
        rx_shift_r <= {rx_shift_r[7:0], sda_i}; // Семплируем SDA
        phase_r    <= 2'd2;
    end
    // Если scl_i = 0 → slave держит SCL → стоим и ждём
end
```

**Критический момент:** мы читаем `sda_i` только когда `scl_i = 1`. Спецификация I2C требует, чтобы данные были стабильны на SDA при SCL=1. Семплирование при SCL=0 бессмысленно.

**Фаза 2: Hold SCL HIGH**

```verilog
2'd2: begin
    scl_oen_o <= 1'b1;  // SCL всё ещё HIGH
    // SDA удерживаем
    phase_r <= 2'd3;
end
```

Зачем нужна эта фаза? Для соблюдения минимального времени высокого уровня SCL (tHIGH). Спецификация I2C требует tHIGH ≥ 4 мкс для Standard Mode. Две фазы SCL=HIGH (фаза 1 + фаза 2) обеспечивают это.

**Фаза 3: Pull SCL LOW, advance**

```verilog
2'd3: begin
    scl_oen_o <= 1'b0;               // SCL → LOW

    if (bit_cnt_r == 4'd8) begin      // Все 9 бит переданы (0..8)
        dout_o   <= rx_shift_r[8:1];  // Принятый байт (биты 8..1)
        rx_ack_o <= rx_shift_r[0];    // ACK/NACK (бит 0 = последний принятый)
        state_r  <= ST_IDLE;
        ready_o  <= 1'b1;             // Команда завершена
    end else begin
        bit_cnt_r  <= bit_cnt_r + 1;
        tx_shift_r <= {tx_shift_r[7:0], 1'b0};  // Сдвиг влево
        phase_r    <= 2'd0;            // Следующий бит
    end
end
```

**Важный нюанс: SDA НЕ меняется в фазе 3.**

Заманчиво было бы начать устанавливать SDA для следующего бита прямо в фазе 3. Но это опасно: SCL переходит в LOW одновременно с изменением SDA, и в зависимости от задержек slave может увидеть это как ложный START или STOP.

Безопасный подход: менять SDA только в фазе 0, когда SCL уже гарантированно LOW.

**Сдвиговые регистры:**

```
tx_shift_r (9 бит): [8][7][6][5][4][3][2][1][0]
                      ↑
                   Выдаётся на SDA

Каждый бит-слот: сдвиг влево, MSB уходит на SDA

Для WRITE: изначально = {D7,D6,D5,D4,D3,D2,D1,D0, 0}
           бит 0: выдаёт D7, бит 1: D6, ... бит 7: D0, бит 8: 0 (release для ACK)

rx_shift_r (9 бит): [8][7][6][5][4][3][2][1][0]
                                              ↑
                                         Принимается с SDA

Каждый бит-слот: сдвиг влево, LSB = sda_i

После 9 бит: rx_shift_r = {D7,D6,D5,D4,D3,D2,D1,D0, ACK}
             dout_o   = rx_shift_r[8:1] = {D7..D0}
             rx_ack_o = rx_shift_r[0]   = ACK
```

### 4.4. ST_STOP — генерация условия STOP

**Определение STOP:** SDA переходит из 0 в 1, пока SCL = 1.

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        STOP!
SDA: ─────────────────────────────────────────╱
                                           __╱──────────────────
SCL: ────────────╱
              __╱────────────────────────────────────────────────

     SDA=0, SCL=0      SDA=0, SCL=1   SDA=1, SCL=1   (hold)
     (подготовка)       (ждём SCL)     (STOP!)        (готово)
```

```verilog
ST_STOP: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b0;    // SDA = LOW (подготовка)
            scl_oen_o <= 1'b0;    // SCL = LOW
            phase_r   <= 2'd1;
        end
        2'd1: begin
            sda_oen_o <= 1'b0;    // SDA остаётся LOW
            scl_oen_o <= 1'b1;    // Отпускаем SCL → HIGH
            if (scl_i) phase_r <= 2'd2;  // Clock stretching
        end
        2'd2: begin
            sda_oen_o <= 1'b1;    // SDA → HIGH !!! ← ВОТ ОН, STOP
            scl_oen_o <= 1'b1;    // SCL остаётся HIGH
            phase_r   <= 2'd3;
        end
        2'd3: begin
            // Hold — обе линии высокие, шина свободна
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;
        end
    endcase
end
```

**Почему фаза 0 нужна?** После предыдущей команды (WRITE/READ) SCL = LOW, но SDA может быть в любом состоянии. Нам нужно гарантировать SDA = LOW перед STOP, иначе переход SDA 0→1 при SCL=1 не произойдёт. Фаза 0 устанавливает начальное условие.

### 4.5. ST_RESTART — повторный START

RESTART — это START, который выполняется без предварительного STOP. Разница: перед обычным START обе линии уже высокие (шина свободна). Перед RESTART SCL = LOW (мы посередине транзакции).

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        START!
SDA: ─────────╱
           __╱──────────────────────╲
SCL: ────────────────╱               ╲──────────────╲
                  __╱──────────────────              ╲──────────

     SDA=1, SCL=0   SDA=1, SCL=1    SDA=0, SCL=1   SDA=0, SCL=0
     (отпустить SDA) (ждём SCL)      (START!)        (готово)
```

```verilog
ST_RESTART: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b1;    // Отпустить SDA → HIGH
            scl_oen_o <= 1'b0;    // SCL остаётся LOW
            phase_r   <= 2'd1;
        end
        2'd1: begin
            sda_oen_o <= 1'b1;    // SDA HIGH
            scl_oen_o <= 1'b1;    // Отпускаем SCL → HIGH
            if (scl_i) phase_r <= 2'd2;  // Clock stretching
        end
        2'd2: begin
            sda_oen_o <= 1'b0;    // SDA → LOW !!! ← START
            scl_oen_o <= 1'b1;    // SCL HIGH
            phase_r   <= 2'd3;
        end
        2'd3: begin
            sda_oen_o <= 1'b0;    // SDA LOW
            scl_oen_o <= 1'b0;    // SCL → LOW
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;
        end
    endcase
end
```

**Порядок в фазе 0 критичен:** сначала отпускаем SDA (фаза 0), и только потом SCL (фаза 1). Если отпустить SCL раньше SDA, при SCL=1 и SDA 0→1 получится ложный STOP.

## Часть 5. Мониторинг шины

### 5.1. Отслеживание занятости шины (busy)

Ядро должно знать, занята ли шина. Это нужно для:
- Определения, какую команду выдать: START (шина свободна) или RESTART (шина занята)
- Управления линиями в IDLE (отпускать или держать)

Детектируем START и STOP **по реальным сигналам на шине**:

```verilog
// Детектор фронтов SDA
reg sda_d_r;
always @(posedge clk_i) sda_d_r <= sda_i;

wire sda_rising  =  sda_i & ~sda_d_r;  // SDA: 0 → 1
wire sda_falling = ~sda_i &  sda_d_r;  // SDA: 1 → 0

// START = SDA падает при SCL=1
wire start_on_bus = sda_falling & scl_i;

// STOP = SDA растёт при SCL=1
wire stop_on_bus = sda_rising & scl_i;

// Флаг занятости
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i)          busy_o <= 1'b0;
    else if (start_on_bus) busy_o <= 1'b1;
    else if (stop_on_bus)  busy_o <= 1'b0;
end
```

**Важно:** это работает на основном клоке `clk_i`, а не на `ena_i`. Фронты SDA/SCL могут быть очень короткими (наносекунды), и мы не должны их пропустить.

### 5.2. Обнаружение потери арбитража

**Когда мастер теряет арбитраж?** Когда он отпустил SDA (ожидает, что линия будет HIGH), но другой мастер или slave тянет SDA к LOW.

Проверяем это в двух ситуациях:

**Ситуация 1: во время передачи данных (ST_DATA, фаза 1)**

```verilog
wire al_data = (state_r == ST_DATA) &&
               (phase_r == 2'd1) &&    // SCL только поднялся
               scl_i &&                 // SCL реально HIGH
               !sda_input_mode &&       // Мастер должен управлять SDA
               sda_oen_o &&            // Мастер отпустил SDA (ожидает 1)
               !sda_i;                 // Но SDA = 0! Кто-то тянет!
```

**Ситуация 2: во время START/RESTART**

```verilog
wire al_start =
    (state_r == ST_START   && phase_r == 2'd0 && sda_oen_o && scl_i && !sda_i) ||
    (state_r == ST_RESTART && phase_r == 2'd1 && sda_oen_o && scl_i && !sda_i);
```

**Реакция на потерю арбитража — немедленное освобождение шины:**

```verilog
if (al_event && state_r != ST_IDLE) begin
    state_r   <= ST_IDLE;
    scl_oen_o <= 1'b1;   // Отпустить SCL
    sda_oen_o <= 1'b1;   // Отпустить SDA
    ready_o   <= 1'b1;   // Сообщить контроллеру
end
```

Флаг `arb_lost_o` — sticky (липкий): устанавливается при потере арбитража и не сбрасывается автоматически. Вышестоящий контроллер должен явно сбросить его через `arb_lost_clear_i` перед новой попыткой.

## Часть 6. Clock Stretching

### 6.1. Что это такое

Clock stretching — механизм, позволяющий slave (или другому мастеру) замедлить шину. Slave удерживает SCL в LOW, когда ему нужно больше времени.

```
Мастер отпускает SCL:     scl_oen_o <= 1'b1;
Ожидает SCL = HIGH:       if (scl_i) ...
Slave держит SCL LOW:     scl_i = 0  → мастер ждёт
Slave отпускает:          scl_i = 1  → мастер продолжает
```

### 6.2. Где реализовано

Clock stretching обрабатывается в **фазе 1** каждого состояния — там, где мастер отпускает SCL и ожидает, что он поднимется:

```verilog
// ST_START, фаза 0:
if (scl_i) phase_r <= 2'd1;   // Ждём SCL HIGH

// ST_DATA, фаза 1:
if (scl_i) begin               // Ждём SCL HIGH
    rx_shift_r <= {rx_shift_r[7:0], sda_i};
    phase_r    <= 2'd2;
end

// ST_STOP, фаза 1:
if (scl_i) phase_r <= 2'd2;   // Ждём SCL HIGH

// ST_RESTART, фаза 1:
if (scl_i) phase_r <= 2'd2;   // Ждём SCL HIGH
```

Паттерн один: `if (scl_i) advance; else stay`.

Нет таймаута — по спецификации I2C, мастер обязан ждать бесконечно. Таймаут, если нужен, реализуется на уровне выше (в контроллере).

## Часть 7. Пошаговый рецепт проектирования

Если бы вы проектировали это ядро с нуля, вот порядок шагов:

### Шаг 1. Определить интерфейс

Нарисуйте порты модуля. Решите:
- Как вышестоящий контроллер будет давать команды?
- Какой набор команд нужен? (START, WRITE, READ, STOP — минимум)
- Какой протокол рукопожатия? (ready/valid — самый простой)

### Шаг 2. Выбрать модель тактирования

Решите: будет ли ядро работать на SCL-частоте или на основном клоке с enable?

**Рекомендация:** основной клок + enable (`ena_i`). Это проще для синтеза, надёжнее для CDC (clock domain crossing), и позволяет менять скорость I2C без изменения ядра.

### Шаг 3. Определить фазы бита

Разбейте один I2C-бит на фазы. 4 фазы — оптимальный баланс между простотой и соблюдением тайминга.

Нарисуйте тайминг-диаграмму для каждого типа операции (START, data bit, STOP, RESTART) — это ваш «чертёж».

### Шаг 4. Спроектировать FSM верхнего уровня

Определите состояния. Для каждого состояния нарисуйте таблицу: «фаза → SCL_oen, SDA_oen, действие, условие перехода».

Пример для ST_START:

| Фаза | SCL_oen | SDA_oen | Действие | Переход |
|------|---------|---------|----------|---------|
| 0 | 1 | 1 | — | scl_i=1 → фаза 1 |
| 1 | 1 | 1 | hold | → фаза 2 |
| 2 | 1 | **0** | START! | → фаза 3 |
| 3 | **0** | 0 | — | → IDLE, ready=1 |

### Шаг 5. Спроектировать сдвиговые регистры

Определите:
- Ширину (9 бит — 8 данных + ACK)
- Начальные значения для WRITE и READ
- Направление сдвига (влево, MSB first)
- Когда семплировать входные данные

### Шаг 6. Добавить clock stretching

В каждом месте, где вы отпускаете SCL, добавьте проверку `if (scl_i)`. Это буквально одна строка в каждой фазе, но без неё ядро не будет работать с медленными slave.

### Шаг 7. Добавить мониторинг шины

Реализуйте:
- Детектор START/STOP на шине → флаг busy
- Детектор потери арбитража → флаг arb_lost + немедленный выход в IDLE

### Шаг 8. Обработать граничные случаи в IDLE

Это самая коварная часть. Продумайте:
- Что если в IDLE приходит команда, но шина занята другим мастером?
- Что если арбитраж потерян — принимать ли новые команды?
- Когда безопасно отпускать линии, а когда нет?

### Шаг 9. Написать тестбенч

Верифицируйте каждый сценарий:
1. Одиночный WRITE + ACK
2. Одиночный READ + NACK
3. Полная транзакция: START + WRITE addr + WRITE data + STOP
4. Repeated START (RESTART)
5. NACK от slave (устройство не отвечает)
6. Clock stretching (slave задерживает SCL)
7. Arbitration lost (если нужно multi-master)
8. Reset во время транзакции

## Часть 8. Типичные ошибки при проектировании

### Ошибка 1: SDA и SCL меняются одновременно

```
Неправильно: в одной фазе одновременно меняем SCL→LOW и SDA→HIGH
             Slave может увидеть STOP (SDA 0→1 при SCL=1) до того,
             как SCL успеет упасть.
```

**Решение:** всегда разделять изменения SCL и SDA на разные фазы.

### Ошибка 2: забыть clock stretching

```
Неправильно: scl_oen_o <= 1'b1;
             phase_r <= 2'd2;  // Сразу переходим дальше

Правильно:   scl_oen_o <= 1'b1;
             if (scl_i) phase_r <= 2'd2;  // Ждём реального HIGH
```

Без этого ядро будет работать только с «идеальными» slave.

### Ошибка 3: безусловный release линий в IDLE

```
Неправильно:
    ST_IDLE: begin
        scl_oen_o <= 1'b1;  // Всегда отпускаем
        sda_oen_o <= 1'b1;

Правильно:
    ST_IDLE: begin
        if (!busy_o) begin   // Только если шина свободна
            scl_oen_o <= 1'b1;
            sda_oen_o <= 1'b1;
        end
```

### Ошибка 4: семплирование SDA при SCL=LOW

```
Неправильно: (в фазе 0)
    rx_shift_r <= {rx_shift_r[7:0], sda_i};  // SCL ещё LOW!

Правильно: (в фазе 1, после scl_i check)
    if (scl_i)
        rx_shift_r <= {rx_shift_r[7:0], sda_i};  // SCL HIGH ✓
```

### Ошибка 5: не сбрасывать ready при приёме команды

```
Неправильно:
    if (cmd_valid_i) begin
        // Забыли ready_o <= 0;
        state_r <= ...;
    end

Правильно:
    if (cmd_valid_i) begin
        ready_o <= 1'b0;    // Сигнализируем: команда принята
        state_r <= ...;
    end
```

Без этого контроллер выше может повторно выдать ту же команду.
