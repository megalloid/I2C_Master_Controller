# Проектирование I2C Master Core с нуля

## Введение

Этот документ — подробное руководство по проектированию аппаратного I2C мастер-контроллера на уровне бит и байт. Мы пройдём путь от спецификации протокола до готового Verilog-модуля `i2c_master_core`, объясняя каждое проектное решение — не только «как», но и «почему именно так, а не иначе».

**Что делает этот модуль:** принимает высокоуровневые команды (START, WRITE byte, READ byte, STOP, RESTART) и генерирует корректные сигналы SCL/SDA на шине I2C, соблюдая все требования протокола.

**Чего он НЕ делает:** не содержит регистров управления, прескалера, прерываний. Это чистое «ядро» — его оборачивают в AXI, Avalon или другой интерфейс.

**Уровень:** для тех, кто понимает основы Verilog и конечных автоматов, но не проектировал I2C ранее.

## Часть 1. Что нам нужно реализовать

### 1.1. Спецификация I2C (NXP UM10204) — ключевые требования

Прежде чем проектировать, нужно извлечь из спецификации I2C те правила, которые непосредственно влияют на архитектуру цифровой логики. Не все разделы UM10204 нам релевантны — нас интересуют только те, которые диктуют поведение нашего FSM и управление пинами.

**Правило 1. Open-drain выходы.**
Мастер может только **притянуть линию к 0** или **отпустить** (high-Z). Подать логическую единицу на линию он не может — за это отвечает внешний pull-up резистор.

> **Почему это важно для архитектуры:** это означает, что наш модуль управляет не самими сигналами SCL/SDA, а **output-enable** (OEN) тристейт-буферов. Мы не можем иметь порт `sda_o` с произвольным значением — мы можем иметь только `sda_oen_o`, где `1` = отпустить (высокий через pull-up), `0` = притянуть к земле. Это решение пронизывает весь код: вместо «выдать 1» мы «отпускаем линию», вместо «выдать 0» — «притягиваем».

**Правило 2. SDA меняется только при SCL=0.**
Во время передачи данных SDA может переключаться только когда SCL удерживается мастером в 0. Если SDA меняется при SCL=1 — это условие START или STOP.

> **Почему это важно:** это накладывает жёсткое ограничение на порядок операций в нашем FSM. Мы никогда не можем менять SDA и SCL одновременно (в одном такте) — это создало бы гонку сигналов, и slave мог бы интерпретировать изменение SDA при ещё-высоком SCL как ложный START/STOP. Отсюда вытекает необходимость **разделения каждого бита на несколько фаз**, где SDA и SCL меняются строго по очереди.

**Правило 3. START = SDA 1→0 при SCL=1. STOP = SDA 0→1 при SCL=1.**

> **Архитектурное следствие:** START и STOP — это **не просто значения** на линиях, а **переходы** (фронты). Это значит, что для генерации START нужно сначала гарантировать SDA=1, потом при SCL=1 перевести SDA в 0. Нельзя просто «установить SDA=0 и SCL=1» — нужна последовательность из нескольких шагов.

**Правило 4. Данные передаются MSB first (старший бит первым).**

> **Следствие для регистров:** сдвиговый регистр должен сдвигать **влево**, а текущий выдаваемый бит берётся из **старшего разряда**. Если бы протокол использовал LSB first, архитектура регистра была бы зеркальной.

**Правило 5. Каждый байт = 8 бит данных + 1 бит ACK/NACK (всего 9 бит-слотов).**

> **Следствие:** счётчик битов считает от 0 до 8 (9 значений), а сдвиговый регистр имеет ширину 9 бит, а не 8. Девятый бит-слот принципиально отличается: при WRITE мастер отпускает SDA и слушает ACK от slave, при READ мастер сам выдаёт ACK/NACK.

**Правило 6. Clock stretching.** Slave может удерживать SCL в 0, чтобы попросить мастера подождать. Мастер обязан проверять, что SCL действительно поднялся, прежде чем продолжать.

> **Следствие:** нельзя просто «отпустить SCL и двигаться дальше». Нужен вход `scl_i` (реальное значение на пине) и условие `if (scl_i)` перед продвижением к следующей фазе. Это добавляет вход в модуль и ветвление в каждое состояние, где SCL поднимается.

**Правило 7. Arbitration.** Если несколько мастеров на шине, мастер, который отпустил SDA (ожидает 1), но видит 0 — потерял арбитраж и должен немедленно освободить шину.

> **Следствие:** нужен мониторинг SDA в определённые моменты (когда SCL=1 и мастер ожидает SDA=1). При обнаружении конфликта — аварийный выход из любого состояния в IDLE с освобождением линий. Это добавляет «аварийный путь» в FSM, который обходит нормальную логику переходов.

### 1.2. Интерфейс команд — что видит «внешний мир»

Наше ядро должно предоставить простой командный интерфейс:

```
                             ┌─────────────────────────────────┐
                             │        i2c_master_core          │
                             │                                 │
  ┌──────────────────┐       │  Тактирование:                 │
  │  Вышестоящий     │       │                                 │
  │  контроллер      │ clk ─►│ clk_i                          │
  │                  │ rst ─►│ rstn_i                         │
  │  (секвенсер,     │ ena ─►│ ena_i                          │
  │   CPU,           │       │                                 │
  │   тестовый FSM)  │       │  Командный интерфейс:          │
  │                  ├──────►│ cmd_valid_i                    │      ┌──────────────────┐
  │                  ├──────►│ cmd_i[2:0]                     │      │  I2C Bus         │
  │                  ├──────►│ din_i[7:0]                     │      │                  │
  │                  │       │                                 │      │  ┌────┐  ┌─────┐│
  │                  │◄──────┤ ready_o                        │      │  │Pull│  │Slave││
  │                  │◄──────┤ dout_o[7:0]                   │      │  │ Up │  │     ││
  │                  │◄──────┤ rx_ack_o                      │      │  └──┬─┘  └──┬──┘│
  │                  │       │                                 │      │     │       │   │
  │                  │       │  Статус:                        │ scl_i│◄────┼───────┤   │
  │                  │◄──────┤ arb_lost_o                    ├──────►│ scl_oen_o    │   │
  │                  ├──────►│ arb_lost_clear_i              │      │     │       │   │
  │                  │◄──────┤ busy_o                        │ sda_i│◄────┼───────┤   │
  │                  │       │                                ├──────►│ sda_oen_o    │   │
```

#### Полный список портов модуля

Все порты разделены на четыре логические группы: тактирование, командный интерфейс, статус/мониторинг и подключение к шине I2C.

**Группа 1 — Тактирование и сброс**

| Порт | Направление | Ширина | Описание |
|------|-------------|--------|----------|
| `clk_i` | вход | 1 | Основной тактовый сигнал FPGA. Вся логика ядра синхронна по нарастающему фронту этого клока. Частота `clk_i` не связана с частотой SCL — она определяет только скорость реакции на внутренние события (фронты SDA/SCL, арбитраж). Типичные значения: 50–200 МГц. |
| `rstn_i` | вход | 1 | Асинхронный сброс, активный LOW. При `rstn_i = 0` все регистры ядра сбрасываются: FSM → IDLE, SCL и SDA отпускаются (`oen = 1`), `ready_o = 1`. Асинхронный сброс выбран потому, что при power-on клок может ещё не быть стабильным, а шину I2C необходимо гарантированно освободить. |
| `ena_i` | вход | 1 | Одноцикловый импульс-разрешение (clock enable). FSM продвигается на одну фазу только в те такты `clk_i`, когда `ena_i = 1`. Генерируется внешним прескалером. Частота `ena_i` = 4 × fSCL (4 фазы на один период SCL). Например, для SCL = 100 кГц нужен `ena_i` с частотой 400 кГц, что при `clk_i` = 50 МГц означает один импульс каждые 125 тактов. |

> **Почему сброс асинхронный?** Синхронный сброс (`if (rst) ...` внутри `always @(posedge clk)`) требует работающего клока для срабатывания. Но в момент включения FPGA клок может отсутствовать или быть нестабильным. I2C-шина с pull-up резисторами при включении находится в состоянии HIGH (idle), и ядро должно **немедленно** перейти в состояние, когда оно не мешает шине (отпустить SCL/SDA). Асинхронный сброс гарантирует это независимо от наличия клока. В FPGA-реализациях (Xilinx, Intel) асинхронный сброс хорошо поддерживается аппаратно (dedicated reset path в flip-flops).

**Группа 2 — Командный интерфейс (от вышестоящего контроллера)**

| Порт | Направление | Ширина | Описание |
|------|-------------|--------|----------|
| `cmd_valid_i` | вход | 1 | Уровневый сигнал: `1` = вышестоящий контроллер выставил валидную команду. Держится высоким, пока ядро не примет команду (сбросит `ready_o`). Можно опустить после приёма, можно держать — ядро защёлкивает команду по фронту приёма. |
| `cmd_i[2:0]` | вход | 3 | Код команды (см. таблицу ниже). Должен быть стабилен одновременно с `cmd_valid_i`. 3 бита позволяют кодировать до 8 команд; используется 5 (0 = NOP, 1–5 = START/WRITE/READ/STOP/RESTART). |
| `din_i[7:0]` | вход | 8 | Входные данные. Для WRITE — байт для передачи на шину. Для READ — бит `din_i[0]` задаёт ACK/NACK, который мастер пошлёт slave после приёма байта (`0` = ACK = «давай ещё», `1` = NACK = «хватит»). Для START/STOP/RESTART — игнорируется. Должен быть стабилен в момент приёма команды. |
| `dout_o[7:0]` | выход | 8 | Принятый байт данных. Валиден после завершения команды READ (когда `ready_o` возвращается в `1`). После WRITE содержит «эхо» переданных данных (игнорируется контроллером). После START/STOP — не определён. |
| `rx_ack_o` | выход | 1 | Принятый бит ACK/NACK от slave. `0` = ACK (slave подтвердил), `1` = NACK (slave не ответил или отказал). Валиден после завершения WRITE или READ. Вышестоящий контроллер проверяет этот сигнал для определения успешности операции. |
| `ready_o` | выход | 1 | Готовность ядра. `1` = ядро свободно, принимает команды. `0` = ядро занято выполнением текущей команды. Переход `1→0` происходит при приёме команды. Переход `0→1` — при завершении. В этот момент `dout_o` и `rx_ack_o` содержат результаты. |

> **Почему `cmd_valid_i` — уровневый, а не импульсный?** Импульсный сигнал (длительностью 1 такт) легко потерять: если ядро в этот такт не проверило вход (например, `ena_i = 0`), импульс будет пропущен. Уровневый сигнал безопаснее — он держится, пока ядро не отреагирует. Ядро «видит» запрос по комбинации `cmd_valid_i = 1 && ready_o = 1` и может обработать его при следующем `ena_i`.

> **Почему `din_i` имеет двойное назначение (данные для WRITE, ACK для READ)?** Можно было бы добавить отдельный порт `nack_i` для управления ACK/NACK при READ. Но это лишний порт, который используется только при READ, и только для одного бита. Переиспользование `din_i[0]` экономит порт и полностью описывает сценарий: при WRITE контроллер загружает 8 бит данных, при READ — загружает значение ACK/NACK в младший бит. Семантика определяется текущей командой.

**Группа 3 — Статус и мониторинг шины**

| Порт | Направление | Ширина | Описание |
|------|-------------|--------|----------|
| `arb_lost_o` | выход | 1 | Sticky-флаг потери арбитража. Устанавливается в `1`, когда ядро обнаружило конфликт: мастер отпустил SDA (ожидал HIGH), но прочитал LOW. Не сбрасывается автоматически — контроллер должен явно сбросить через `arb_lost_clear_i`. Пока флаг установлен, ядро **не принимает** новые команды (защита от повторных попыток без обработки ошибки). |
| `arb_lost_clear_i` | вход | 1 | Одноцикловый импульс для сброса `arb_lost_o`. Вышестоящий контроллер выдаёт этот импульс после обработки ситуации потери арбитража (обычно перед повторной попыткой транзакции). |
| `busy_o` | выход | 1 | Флаг занятости шины I2C. `1` = на шине обнаружен START (любым мастером), STOP ещё не было. `0` = шина свободна. Определяется по реальным сигналам на пинах SCL/SDA, а не по внутреннему состоянию FSM. Используется внутри ядра (управление линиями в IDLE) и может использоваться вышестоящим контроллером для принятия решений (ждать освобождения шины). |

> **Почему `arb_lost_o` — sticky?** Сценарий без sticky: ядро обнаружило арбитраж, установило `arb_lost_o = 1`, через один такт сбросило. Контроллер в это время мог быть занят другой логикой и пропустить однотактовый импульс. Результат: контроллер не знает об ошибке и выдаёт следующую команду, которая тоже провалится. Sticky-флаг гарантирует, что информация «дождётся» обработки.

**Группа 4 — Подключение к шине I2C (pad-интерфейс)**

| Порт | Направление | Ширина | Описание |
|------|-------------|--------|----------|
| `scl_i` | вход | 1 | Текущее значение линии SCL, прочитанное с пина FPGA. Должен быть **предварительно синхронизирован** 2-стадийным синхронизатором (в обёртке, не в ядре). Используется для: (a) обнаружения clock stretching, (b) детекции START/STOP на шине, (c) проверки арбитража. |
| `scl_oen_o` | выход | 1 | Output-enable для SCL. `1` = отпустить (high-Z, pull-up подтянет к VCC). `0` = притянуть к GND (drive low). **Это не значение SCL**, а управление тристейт-буфером. Шина I2C — open-drain: мастер может только притянуть линию к 0 или отпустить. |
| `sda_i` | вход | 1 | Текущее значение линии SDA, прочитанное с пина FPGA. Аналогично `scl_i`, должен быть предварительно синхронизирован. Используется для: (a) приёма данных (READ), (b) чтения ACK/NACK, (c) детекции START/STOP, (d) проверки арбитража. |
| `sda_oen_o` | выход | 1 | Output-enable для SDA. Семантика та же, что и у `scl_oen_o`: `1` = отпустить, `0` = притянуть к 0. При передаче данных `sda_oen_o` напрямую отражает значение бита: для передачи «1» мастер отпускает линию (`oen = 1`), для «0» — притягивает (`oen = 0`). |

> **Почему интерфейс к шине — OEN, а не прямой data_out?** Протокол I2C основан на open-drain. Мастер **физически не может** выдать логическую единицу на SDA/SCL — он может только притянуть к земле или отпустить. Уровень HIGH создаётся внешним pull-up резистором. Если бы интерфейс был `scl_o` / `sda_o` (прямое значение), возникла бы двусмысленность: что значит `sda_o = 1`? Активно выдать HIGH (push-pull)? Или отпустить? OEN-интерфейс однозначен: `oen = 1` → high-Z → pull-up → HIGH. `oen = 0` → drive → LOW.

> **Почему синхронизация `scl_i` / `sda_i` вынесена из ядра?** Двухстадийный синхронизатор (два последовательных flip-flop для подавления метастабильности) — стандартная практика для асинхронных входов. Но его размещение — архитектурный выбор:
>
> - **Внутри ядра:** проще для пользователя (подключил пин — и забыл). Но если несколько модулей используют одну шину (например, ядро + внешний монитор), каждый имеет свою копию синхронизатора, и они могут давать **разные значения** в одном такте (из-за метастабильности). Это создаёт несогласованность.
> - **Снаружи (наш выбор):** один синхронизатор на пин, его выход разводится ко всем потребителям. Все видят одно и то же значение. Это надёжнее и стандартнее. Минус: пользователь обёртки должен не забыть добавить синхронизатор, но это документировано и типично для FPGA-дизайна.

#### Подключение к тристейт-буферам (типичная схема)

На уровне top-level модуля OEN-сигналы подключаются к тристейт-буферам:

```verilog
// В обёртке (top-level):
assign sda_io = sda_oen ? 1'bz : 1'b0;   // oen=1 → high-Z, oen=0 → drive LOW
assign scl_io = scl_oen ? 1'bz : 1'b0;

// Синхронизация входов:
reg [1:0] sda_sync;
always @(posedge clk) sda_sync <= {sda_sync[0], sda_io};
wire sda_s = sda_sync[1];  // → подаётся на sda_i ядра
```

> `1'b0` в тристейт-присваивании — потому что open-drain: когда мы «активны» (oen=0), мы всегда тянем линию к земле, никогда к VCC. Значение `1'bz` — это high-impedance, при котором pull-up резистор подтягивает линию к VCC.

**Команды (5 штук):**

| Код | Команда | Смысл |
|-----|---------|-------|
| 1 | START | Генерировать условие START |
| 2 | WRITE | Передать байт из `din_i`, получить ACK |
| 3 | READ | Принять байт от slave, отправить ACK/NACK |
| 4 | STOP | Генерировать условие STOP |
| 5 | RESTART | Генерировать повторный START (без предварительного STOP) |

> **Почему именно 5 команд?** Это минимальный полный набор, покрывающий все I2C-сценарии. Рассмотрим альтернативы:
>
> - **3 команды (WRITE, READ, STOP)** — можно было бы встроить START в команду WRITE (первый WRITE автоматически генерирует START). Но это ломает гибкость: вышестоящий контроллер теряет контроль над тем, когда именно выдать START, и не может разделить START и передачу адреса на отдельные этапы. Для отладки и для сложных протоколов (SMBus, PMBus) это критично.
>
> - **4 команды (без RESTART)** — можно было бы реализовать RESTART как STOP+START. Но это неправильно: спецификация I2C различает RESTART (repeated start) и STOP+START, потому что RESTART не освобождает шину (busy не сбрасывается, другой мастер не может захватить шину). Многие slave-устройства (в частности, EEPROM при random read) требуют именно RESTART.
>
> - **6+ команд (составные)** — можно было бы добавить STA+WR, RD+NACK+STO и другие комбинации. Но это усложняет ядро без необходимости. Составные команды эффективнее реализовать в слое выше (в «секвенсере» обёртки AXI/Avalon), оставив ядро атомарным.
>
> Таким образом, 5 команд — это **минимально полный** набор: ничего нельзя убрать без потери функциональности, и ничего не нужно добавлять для полноценной работы.

**Протокол рукопожатия:**

```
         ┌─────┐                   ┌─────┐
ready_o  │     │                   │     │
     ────┘     └───────────────────┘     └────
                                   ↑
         ┌─────────────────┐       │ Команда завершена,
cmd_valid│                 │       │ можно читать dout_o/rx_ack_o
     ────┘                 └───────┘

              ↑ Ядро приняло команду
              │ (ready_o упал)
```

1. Внешний контроллер ждёт `ready_o = 1`
2. Выставляет `cmd_valid_i = 1`, `cmd_i`, `din_i`
3. Ядро видит запрос, сбрасывает `ready_o = 0` — начинает работу
4. Контроллер может убрать `cmd_valid_i`
5. Ядро генерирует I2C-сигналы...
6. Ядро поднимает `ready_o = 1` — команда завершена
7. `dout_o` содержит принятый байт (для READ), `rx_ack_o` — полученный ACK

> **Почему именно ready/valid, а не FIFO или request/acknowledge?**
>
> Мы рассматривали три варианта рукопожатия:
>
> **Вариант A: FIFO-интерфейс (write-enable + data, read-enable + data).** Плюс: позволяет буферизовать несколько команд. Минус: существенно усложняет ядро (нужна логика FIFO, указатели чтения/записи, обработка переполнения), увеличивает латентность из-за стадии чтения из FIFO, и при ошибке (NACK, arb lost) возникает проблема — что делать с командами, уже находящимися в FIFO? Нужен механизм flush, что ещё сложнее.
>
> **Вариант B: request/acknowledge (req → ack → deassert req → deassert ack).** Это 4-фазное рукопожатие. Плюс: надёжно, асинхронно-безопасно. Минус: 4 фазы рукопожатия создают overhead по тактам, и это избыточно — ядро и контроллер работают в одном клоковом домене.
>
> **Вариант C: ready/valid (наш выбор).** Контроллер выставляет valid+data, ядро подтверждает через ready. Одна фаза — один такт на приём команды. Плюс: минимальная латентность, простая реализация (ready — это просто флаг «ядро свободно»), естественно вписывается в FSM (ready=1 в IDLE, ready=0 во время работы). Минус: нет буферизации, но нам она и не нужна — I2C-команды по своей природе последовательны (нельзя начать STOP, не закончив WRITE).
>
> Выбран вариант C как самый простой и достаточный.

## Часть 2. Ключевое проектное решение — 4-фазная модель бита

### 2.1. Зачем нужны фазы

Один бит I2C — это один полный период SCL. Но внутри этого периода происходят разные вещи: подготовка данных, установка SCL, чтение данных, снятие SCL. Нам нужно разделить бит на подэтапы.

> **Почему нельзя обойтись без фаз?** Можно было бы попытаться обработать весь бит в одном такте — выставить SDA, поднять SCL, прочитать данные, опустить SCL. Но это невозможно по двум причинам:
>
> 1. **Тайминг:** спецификация I2C требует минимальных задержек между событиями. SCL должен быть HIGH как минимум 4 мкс (Standard Mode). Нельзя поднять и опустить SCL за один такт — slave не успеет среагировать.
>
> 2. **Clock stretching:** после того как мастер отпускает SCL, нужно проверить, что SCL реально поднялся. Это может занять неопределённое время (slave удерживает SCL). Значит, нужно состояние ожидания — а это уже несколько тактов.
>
> 3. **Разделение SDA и SCL изменений:** правило 2 из спецификации требует, чтобы SDA менялось только при SCL=0. Если в одном такте менять оба сигнала, на реальной плате из-за разницы задержек (routing, буферы) slave может увидеть изменение SDA при ещё-высоком SCL.
>
> Итого: каждый бит неизбежно требует нескольких тактов. Вопрос только в количестве.

Самое естественное деление — **на 4 четверти** периода SCL:

```
         Фаза 0      Фаза 1      Фаза 2      Фаза 3
        ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
        │         │ │         │ │         │ │         │
SCL:  ──┘ LOW     └─┘  HIGH   └─┘  HIGH   └─┘  LOW    └──
                      ↑                        ↑
SDA:  ══╤════════════╤═══════════╤════════════╤═════════
        │            │           │            │
    Установить    Slave может  Данные        Подготовка
    SDA (данные   растянуть    стабильны,    к следующему
    или release)  клок тут     семплируем    биту
```

> **Почему именно 4 фазы, а не 2, 3 или 8?**
>
> **2 фазы (SCL LOW / SCL HIGH):** кажутся достаточными на первый взгляд. Но в фазе SCL LOW нужно сделать два дела: (1) установить SDA и (2) подготовиться к следующему биту (сдвиг регистра). А в фазе SCL HIGH нужно: (1) дождаться реального SCL=1 (stretching) и (2) дать время для стабилизации данных (hold time). Две фазы не дают достаточно гранулярности для разделения этих подзадач. Конкретная проблема: в фазе 0 (SCL LOW) мы сдвигаем регистр и выставляем SDA. Но если объединить сдвиг и выдачу, нужно точно знать, какой бит уже на SDA, а какой ещё нет. При 2 фазах это приводит к тонким ошибкам на последнем бите (бит 8 — ACK).
>
> **3 фазы (setup / SCL HIGH / shift):** работоспособный вариант, и некоторые реализации I2C так и делают. Но спецификация требует минимальный tHIGH (время SCL в HIGH). При 3 фазах SCL HIGH занимает только 1/3 периода, что при граничных частотах может быть впритык. При 4 фазах SCL HIGH занимает 2/4 = 50% периода, что даёт хороший запас по таймингу и симметричный сигнал.
>
> **8 фаз или больше:** дают ещё более точный контроль тайминга, но избыточны. Каждая дополнительная фаза — это дополнительный бит в счётчике фаз, усложнение case-блоков и замедление итерации (больше ena_i тиков на бит). Для Standard Mode (100 кГц) и даже Fast Mode (400 кГц) 4 фазы обеспечивают все необходимые тайминги.
>
> **4 фазы — оптимальный компромисс:** минимально достаточное количество для соблюдения всех тайминговых требований, симметричный SCL (50% duty cycle), и простой 2-битный счётчик фаз (`phase_r[1:0]`), который дёшево ложится на FPGA.

- **Фаза 0 (SCL=LOW, setup):** Безопасно менять SDA — SCL низкий, изменение SDA не создаст ложный START/STOP.
- **Фаза 1 (SCL→HIGH, sample):** Отпускаем SCL. Проверяем, что SCL действительно поднялся (clock stretching!). Семплируем SDA.
- **Фаза 2 (SCL=HIGH, hold):** Держим SCL высоким. Данные стабильны. Дополнительное время для slave.
- **Фаза 3 (SCL→LOW, shift):** Притягиваем SCL к 0. Продвигаем сдвиговый регистр. Инкрементируем счётчик бит.

> **Почему семплирование SDA именно в фазе 1, а не в фазе 2?** Обе фазы — SCL HIGH, данные должны быть стабильны. Но фаза 1 — это момент *сразу после* подъёма SCL. Если slave ответил ACK (притянул SDA к 0), он сделал это ещё в фазе 0 предыдущего бита или раньше. К фазе 1 данные уже стабилизировались. Технически можно семплировать и в фазе 2 — результат будет тот же. Мы выбрали фазу 1 потому что: (a) если SCL задержался из-за stretching, мы семплируем сразу, как только SCL поднялся — это минимизирует задержку; (b) фаза 2 остаётся «свободной», что упрощает код (нет побочных эффектов).

Это деление применяется единообразно ко ВСЕМ типам бит (данные, ACK, START, STOP, RESTART), что упрощает архитектуру.

> **Почему единообразие важно?** Мы могли бы оптимизировать START и STOP — им не нужен сдвиговый регистр, у них нет 9 бит-слотов, и теоретически можно обойтись 2-3 фазами. Но разная длительность фаз для разных состояний означает, что прескалер (генератор ena_i) должен знать, в каком состоянии ядро, чтобы генерировать тики с разной частотой. Это ломает модульность. При единообразных 4 фазах прескалер — это просто счётчик, который не зависит от внутреннего состояния ядра. START и STOP занимают 4 такта ena_i (вместо возможных 2-3), но это микроскопическая «потеря» на фоне 36 тактов на байт данных.

### 2.2. Тактирование фаз — сигнал ena_i

Ядро тактируется основным клоком FPGA (например, 50 МГц), но фазы продвигаются по сигналу `ena_i` — одноцикловому импульсу от внешнего прескалера.

```
clk_i:  ┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐...  (50 MHz)

ena_i:  _│‾│__________________________________│‾│______...
         ↑                                     ↑
       Фаза N                               Фаза N+1
       (125 тактов между ними для SCL = 100 кГц)
```

> **Почему прескалер вынесен за пределы ядра?** Мы рассматривали два варианта:
>
> **Вариант A: встроенный прескалер.** Ядро имеет параметр `PRESCALE` и внутренний счётчик. Плюс: одномодульное решение, проще подключить. Минус: при изменении скорости I2C (например, переключение Standard → Fast Mode «на лету») нужно менять внутренний регистр ядра и синхронизировать изменение с текущим состоянием FSM. Это добавляет сложность: что если прескалер меняется посередине бита?
>
> **Вариант B: внешний прескалер (наш выбор).** Ядро получает однобитный `ena_i` и не знает о частоте. Плюс: чистое разделение ответственности — ядро отвечает за протокол, внешний модуль — за тайминг. Смена скорости I2C не затрагивает ядро вообще. Переиспользование ядра на другой частоте FPGA требует только замены прескалера, а не модификации ядра. Минус: нужен внешний модуль, но это тривиальный счётчик в 5 строк.
>
> Выбран вариант B. Это классический принцип «single responsibility»: ядро знает I2C-протокол и ничего больше.

**Почему так?** Это разделяет задачи:
- Прескалер (за пределами ядра) — отвечает за скорость I2C
- Ядро — отвечает за логику протокола

Ядро не знает и не заботится о конкретной частоте SCL. Оно просто продвигается на один шаг при каждом `ena_i`.

> **Расчёт ena_i для конкретной частоты.** Поскольку 4 фазы = 1 SCL-период, то частота ena_i = 4 × fSCL. Для Standard Mode (fSCL = 100 кГц): fena = 400 кГц. При clk = 50 МГц: прескалер = 50 000 000 / 400 000 = 125. То есть ena_i = 1 раз в 125 тактов clk. Прескалер — это простой счётчик от 0 до 124 с выдачей импульса при переполнении.

## Часть 3. Архитектура FSM — конечный автомат ядра

### 3.1. Состояния верхнего уровня

```
                    ┌──────────┐
            ┌──────►│  IDLE    │◄──────┐
            │       └────┬─────┘       │
            │            │             │
            │    cmd_valid_i?          │
            │     │    │    │    │     │
            │   START WRITE READ STOP  RESTART
            │     │    │    │    │     │
            │     ▼    ▼    ▼    ▼     ▼
            │  ┌──────┐ ┌──────┐ ┌───────┐ ┌──────┐ ┌─────────┐
            │  │START │ │ DATA │ │ DATA  │ │ STOP │ │ RESTART │
            │  │(4 ф) │ │(9×4ф)│ │(9×4ф) │ │(4 ф) │ │ (4 ф)  │
            │  └──┬───┘ └──┬───┘ └──┬────┘ └──┬───┘ └────┬────┘
            │     │        │        │         │          │
            └─────┴────────┴────────┴─────────┴──────────┘
                            ready_o = 1
```

Всего **5 состояний** (не считая IDLE):

| Состояние | Фаз | Бит-слотов | Итого фаз | Назначение |
|-----------|-----|-----------|-----------|------------|
| ST_IDLE | — | — | — | Ожидание команды |
| ST_START | 4 | 1 | 4 | Условие START |
| ST_DATA | 4 | 9 | 36 | Передача/приём байта + ACK |
| ST_STOP | 4 | 1 | 4 | Условие STOP |
| ST_RESTART | 4 | 1 | 4 | Повторный START |

> **Почему 5 состояний, а не больше или меньше?**
>
> **Можно ли убрать ST_RESTART?** Нет. RESTART отличается от START начальными условиями: перед START шина свободна (SCL=1, SDA=1), перед RESTART SCL=0 (мы посередине транзакции). Последовательность управления линиями разная. Если использовать одно состояние START для обоих случаев, нужен дополнительный флаг «откуда мы пришли», что эквивалентно второму состоянию, но менее читаемо.
>
> **Можно ли добавить отдельные состояния для WRITE и READ?** Можно, но это избыточно (см. 3.2 ниже).
>
> **Можно ли объединить START и STOP в одно состояние «шинная условие»?** Теоретически — да, с параметром «тип условия». Но START и STOP имеют разные последовательности фаз (в START сначала SCL HIGH, потом SDA→LOW; в STOP сначала SDA→LOW, потом SCL HIGH, потом SDA→HIGH). Объединение приведёт к вложенному case (case по типу внутри case по фазе), что запутывает код больше, чем два отдельных состояния.
>
> **Итог:** 5 состояний — это 1:1 отображение I2C-операций на FSM-стейты. Каждое состояние имеет свою уникальную последовательность фаз. Меньше нельзя без потери корректности, больше — без потери простоты.

### 3.2. Почему WRITE и READ используют одно состояние ST_DATA?

Это ключевое упрощение. На уровне шины WRITE и READ отличаются только одним: **кто управляет SDA** в каждом бит-слоте.

```
WRITE (мастер передаёт):
  Бит 0-7: мастер выдаёт → SDA = tx_shift_r[8]  (MSB first)
  Бит 8:   slave отвечает → SDA = release (high-Z), читаем ACK

READ (slave передаёт):
  Бит 0-7: slave выдаёт → SDA = release (high-Z), читаем данные
  Бит 8:   мастер отвечает → SDA = 0(ACK) или 1(NACK)
```

> **Анализ общности:** посмотрим, что происходит в каждом бит-слоте для обоих случаев:
>
> | Бит-слот | WRITE: SDA | READ: SDA |
> |----------|-----------|-----------|
> | 0-7 | tx_shift_r[8] (мастер гонит) | release (slave гонит) |
> | 8 | release (slave ACK) | tx_shift_r[8] (мастер ACK/NACK) |
>
> Видим паттерн: в каждом бит-слоте SDA либо управляется мастером (`sda_oen_o = tx_shift_r[8]`), либо отпускается (`sda_oen_o = 1`). Разница только в том, **какие** бит-слоты мастер управляет, а какие — отпускает. Это один бит решения, а не два разных FSM.

Это можно выразить одним сигналом `sda_input_mode`:

```verilog
wire sda_input_mode = (state_r == ST_DATA) && (
    (cmd_r == CMD_READ  && bit_cnt_r < 4'd8) ||  // READ биты 0-7: slave гонит
    (cmd_r == CMD_WRITE && bit_cnt_r == 4'd8)     // WRITE бит 8: slave ACK
);
```

Когда `sda_input_mode = 1` — мастер отпускает SDA и слушает. Когда `0` — мастер выдаёт бит из сдвигового регистра.

> **Почему это важно для ресурсов?** Два отдельных состояния (ST_WRITE, ST_READ) с почти идентичной логикой фаз означают дублирование case-блоков для фаз 0, 1, 2, 3. На FPGA это: (a) удвоение LUT-ов для управления SCL/SDA, (b) удвоение числа переходов в FSM, (c) удвоение поверхности для ошибок при модификации. Один сигнал `sda_input_mode` заменяет всё это одним мультиплексором.

Один FSM-стейт вместо двух = меньше логики, меньше багов, проще верификация.

## Часть 4. Проектирование каждого состояния

### 4.1. ST_IDLE — ожидание команды

**Что делает:** ждёт, пока вышестоящий контроллер не выставит команду.

**Тонкость #1: управление линиями в IDLE.**

Когда ядро в IDLE, ему нужно решить, что делать с SCL и SDA. Есть три ситуации:

```
1. Шина свободна (после STOP или при включении):
   → Отпустить обе линии (высокий через pull-up)
   → scl_oen = 1, sda_oen = 1

2. Шина занята, команда ожидается (между START и STOP):
   → Держать линии как есть! НЕ отпускать!
   → Если отпустить SDA при SCL=1, получится ложный STOP

3. Новая команда пришла:
   → Защёлкнуть команду, перейти в нужное состояние
```

> **Почему в IDLE нужно различать «шина свободна» и «шина занята»?** При типичной I2C-транзакции (START → WRITE addr → WRITE data → STOP) ядро возвращается в IDLE **три раза**: после START, после WRITE addr, после WRITE data (перед STOP). В каждом из этих «промежуточных IDLE» шина занята — SCL удерживается в LOW (мастер не отпускал его после последней фазы 3 предыдущей команды). Если бы IDLE безусловно отпускал SCL и SDA, после каждой команды slave бы видел:
>
> 1. SCL переходит в HIGH (мастер отпустил)
> 2. SDA в неопределённом состоянии → возможен STOP
> 3. Slave сбрасывает свой FSM → транзакция разрушена
>
> Именно поэтому мы проверяем `busy_o`: если шина занята, линии остаются как есть. Если свободна — безопасно отпускаем.

Это выражается так:

```verilog
ST_IDLE: begin
    if (cmd_valid_i && !arb_lost_o) begin
        ready_o <= 1'b0;       // Начинаем работу
        case (cmd_i)
            CMD_START:   ... → ST_START
            CMD_WRITE:   ... → ST_DATA
            CMD_READ:    ... → ST_DATA
            CMD_STOP:    ... → ST_STOP
            CMD_RESTART: ... → ST_RESTART
        endcase
    end else if (!busy_o) begin
        scl_oen_o <= 1'b1;    // Шина свободна — отпустить
        sda_oen_o <= 1'b1;
    end
    // Если busy_o=1 но нет команды — ничего не делаем, держим линии
end
```

> **Почему проверяем `!arb_lost_o`?** Если арбитраж потерян, новые команды не принимаются до явного сброса флага. Это предотвращает ситуацию, когда контроллер пытается повторить транзакцию, не узнав о потере арбитража. Без этой проверки ядро могло бы начать новый START на шине, которая уже захвачена другим мастером.

**Ошибка, которую легко допустить:** безусловно отпускать линии в IDLE. Это создаёт ложные STOP между командами составной транзакции (START + WRITE + WRITE + STOP). Между WRITE и WRITE ядро ненадолго попадает в IDLE (ждёт следующую команду), и если отпустит SDA при SCL=1 — slave увидит STOP.

**Тонкость #2: загрузка сдвигового регистра.**

При приёме команды WRITE или READ сразу загружаем данные в сдвиговый регистр:

```verilog
CMD_WRITE: begin
    tx_shift_r <= {din_i, 1'b0};    // 8 бит данных + 0 (мастер ACK не нужен)
    cmd_r      <= CMD_WRITE;
    bit_cnt_r  <= 4'd0;
end
CMD_READ: begin
    tx_shift_r <= {8'hFF, din_i[0]}; // FF=отпустить SDA + бит NACK
    cmd_r      <= CMD_READ;
    bit_cnt_r  <= 4'd0;
end
```

> **Почему загрузка происходит в IDLE, а не в ST_DATA?** Мы могли бы загружать tx_shift_r в фазе 0 первого бита ST_DATA. Но это создало бы проблему: в фазе 0 мы уже должны выставить SDA (из tx_shift_r[8]), значит загрузка и чтение регистра должны произойти в один такт. Это возможно в Verilog (неблокирующее присваивание использует старое значение для чтения), но ведёт к неинтуитивному коду и потенциальным ошибкам при модификации.
>
> Загрузка в IDLE (при приёме команды) гарантирует, что к моменту перехода в ST_DATA регистр уже содержит правильные данные. Чисто и безопасно.

Для WRITE: `tx_shift_r = {D7,D6,D5,D4,D3,D2,D1,D0, 0}` — 8 бит данных, потом 0 (будем слушать ACK).

Для READ: `tx_shift_r = {1,1,1,1,1,1,1,1, NACK}` — 8 единиц (отпустить SDA, slave гонит данные), потом бит NACK/ACK который мастер пошлёт.

> **Почему для READ загружаем 8'hFF?** Потому что `sda_oen_o = tx_shift_r[8]`, и `oen = 1` означает «отпустить». Загружая `1` в старшие 8 бит, мы гарантируем, что в бит-слотах 0-7 мастер отпустит SDA (slave будет управлять ей). А девятый бит (бит 0 регистра) — это значение ACK/NACK, которое мастер хочет послать: `din_i[0] = 0` = ACK (продолжить чтение), `din_i[0] = 1` = NACK (последний байт).

### 4.2. ST_START — генерация условия START

**Определение START:** SDA переходит из 1 в 0, пока SCL = 1.

**Тайминг по фазам:**

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        START!
SDA: ────────────────────────────────╲
                                      ╲────────────────────────
SCL: ────────────────────────────────────────────────╲
                                                      ╲────────

     SDA=1, SCL=1      SDA=1, SCL=1   SDA=0, SCL=1   SDA=0, SCL=0
     (ждём SCL high     (hold)         (START!)        (готово)
      если stretching)
```

> **Почему START занимает 4 фазы, а не 2 (SDA→0, затем SCL→0)?**
>
> 1. **Фаза 0 — гарантия начальных условий.** Перед START обе линии должны быть высокими. Если это самый первый START (шина свободна), это выполнено по умолчанию. Но спецификация требует выделить отдельный setup time (tBUF) после предыдущего STOP. Фаза 0 обеспечивает этот промежуток. Кроме того, здесь мы ждём clock stretching (`if (scl_i)`) — возможно, предыдущая операция оставила SCL в LOW.
>
> 2. **Фаза 1 — hold.** Дополнительное время гарантирует, что и SCL, и SDA стабильно HIGH перед тем, как мы уроним SDA. Без этого, при быстрых FPGA-клоках, промежуток между отпусканием SCL (фаза 0) и падением SDA (фаза 2) мог бы быть слишком коротким для slave.
>
> 3. **Фаза 2 — собственно START.** SDA → LOW при SCL = HIGH. Это единственный «полезный» такт.
>
> 4. **Фаза 3 — подготовка.** SCL → LOW. Это необходимо, потому что после START следующая операция — передача данных. Данные передаются при SCL=LOW. Если не опустить SCL, мастер начнёт менять SDA (для первого бита) при SCL=HIGH, что slave интерпретирует как ещё один START или непонятную последовательность.

**Реализация:**

```verilog
ST_START: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b1;         // SDA отпущена (high)
            scl_oen_o <= 1'b1;         // SCL отпущена (high)
            if (scl_i) phase_r <= 2'd1; // Ждём, пока SCL реально станет HIGH
        end
        2'd1: begin
            sda_oen_o <= 1'b1;         // SDA всё ещё high
            scl_oen_o <= 1'b1;         // SCL high — hold time
            phase_r   <= 2'd2;
        end
        2'd2: begin
            sda_oen_o <= 1'b0;         // SDA → LOW !!! ← ВОТ ОН, START
            scl_oen_o <= 1'b1;         // SCL всё ещё high
            phase_r   <= 2'd3;
        end
        2'd3: begin
            sda_oen_o <= 1'b0;         // SDA остаётся low
            scl_oen_o <= 1'b0;         // SCL → LOW (подготовка к данным)
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;         // Команда выполнена
        end
    endcase
end
```

**Почему в фазе 0 проверяем `scl_i`?** Clock stretching. Если предыдущая команда оставила SCL низким (slave держит), нужно дождаться, пока slave отпустит. `scl_i` — это реальный уровень на пине. Пока slave держит SCL=0, мы стоим в фазе 0.

### 4.3. ST_DATA — передача/приём 9 бит

Это самое сложное состояние. Оно обрабатывает 9 бит-слотов: 8 бит данных + 1 бит ACK. Каждый бит-слот проходит через 4 фазы.

> **Почему 9 × 4 = 36 фаз в одном состоянии, а не 9 последовательных состояний по 4 фазы?** Каждый из 9 бит-слотов выполняет абсолютно одну и ту же последовательность действий (setup SDA → raise SCL → sample → lower SCL). Различие только в том, какой бит выдаётся/принимается (определяется `bit_cnt_r` и `sda_input_mode`). Создавать 9 отдельных состояний для одинаковой логики — это копипаста, увеличивающая FSM с 5 до 13 состояний без какой-либо пользы. Вместо этого мы используем **двухуровневую вложенность**: внешний уровень — `bit_cnt_r` (0..8), внутренний — `phase_r` (0..3).

**Фаза 0: Setup SDA (SCL=LOW)**

```verilog
2'd0: begin
    scl_oen_o <= 1'b0;               // SCL LOW — безопасно менять SDA
    if (sda_input_mode)
        sda_oen_o <= 1'b1;           // Slave гонит → отпускаем SDA
    else
        sda_oen_o <= tx_shift_r[8];  // Мастер гонит → выдаём MSB
    phase_r <= 2'd1;
end
```

**Что такое `tx_shift_r[8]`?** Это старший бит сдвигового регистра. При `oen = 0` (бит = 0) мастер притягивает SDA к 0 (передаём «0»). При `oen = 1` (бит = 1) мастер отпускает SDA, pull-up подтягивает к 1 (передаём «1»).

По сути, `sda_oen_o = tx_shift_r[8]` — это XOR-инверсия: мы НЕ выдаём данные напрямую, а управляем тристейт-буфером. `oen = 1` → линия отпущена → pull-up → единица. `oen = 0` → линия притянута → ноль.

> **Почему OEN = данные, а не OEN = ~данные?** Это следствие open-drain: единственный способ выдать «0» — притянуть линию к земле (`oen = 0`, drive low). Единственный способ выдать «1» — отпустить линию (`oen = 1`, release). Таким образом, значение OEN *совпадает* с передаваемым битом: OEN=1 → линия = 1, OEN=0 → линия = 0. Инверсия не нужна. Это удобное совпадение, которое упрощает код.

**Фаза 1: Release SCL, sample SDA**

```verilog
2'd1: begin
    scl_oen_o <= 1'b1;               // Отпускаем SCL → HIGH (через pull-up)
    // SDA продолжаем удерживать как в фазе 0

    if (scl_i) begin                  // SCL реально HIGH? (clock stretching!)
        rx_shift_r <= {rx_shift_r[7:0], sda_i}; // Семплируем SDA
        phase_r    <= 2'd2;
    end
    // Если scl_i = 0 → slave держит SCL → стоим и ждём
end
```

**Критический момент:** мы читаем `sda_i` только когда `scl_i = 1`. Спецификация I2C требует, чтобы данные были стабильны на SDA при SCL=1. Семплирование при SCL=0 бессмысленно.

> **Почему SDA семплируется безусловно (и для WRITE, и для READ)?** Для READ это очевидно — мы принимаем данные. Но зачем семплировать SDA при WRITE? Потому что:
>
> 1. **Бит 8 (ACK):** при WRITE мастер отпускает SDA и slave отвечает ACK/NACK. Семплирование в бите 8 необходимо.
> 2. **Арбитраж:** даже когда мастер гонит данные (биты 0-7), он должен проверять, совпадает ли значение на SDA с тем, что он выдал. Если мастер выдал 1 (отпустил SDA), а на SDA оказался 0 — значит другой мастер тянет линию. Это детектируется отдельной логикой арбитража, но rx_shift_r собирает данные в любом случае.
> 3. **Единообразие:** один и тот же путь данных для WRITE и READ упрощает код. `dout_o` после WRITE будет содержать «мусор» (то, что мастер сам выдал), но вышестоящий контроллер просто игнорирует dout_o для WRITE.

**Фаза 2: Hold SCL HIGH**

```verilog
2'd2: begin
    scl_oen_o <= 1'b1;  // SCL всё ещё HIGH
    // SDA удерживаем
    phase_r <= 2'd3;
end
```

Зачем нужна эта фаза? Для соблюдения минимального времени высокого уровня SCL (tHIGH). Спецификация I2C требует tHIGH ≥ 4 мкс для Standard Mode. Две фазы SCL=HIGH (фаза 1 + фаза 2) обеспечивают это.

> **Не является ли фаза 2 «пустой»?** По коду — да, тело фазы 2 почти пустое (держим SCL=1, ничего не делаем). Но это не баг, а фича:
>
> 1. Фаза 2 обеспечивает **50% duty cycle** SCL (2 фазы LOW + 2 фазы HIGH). Спецификация рекомендует, чтобы tHIGH и tLOW были примерно равны.
> 2. Фаза 2 даёт дополнительный запас для **медленных slave-устройств**, которым нужно больше времени для стабилизации SDA.
> 3. Фаза 2 сохраняет **единообразие**: все состояния (START, DATA, STOP, RESTART) используют 4 фазы. Если убрать фазу 2 для DATA, нужно менять прескалер или вводить исключения.

**Фаза 3: Pull SCL LOW, advance**

```verilog
2'd3: begin
    scl_oen_o <= 1'b0;               // SCL → LOW

    if (bit_cnt_r == 4'd8) begin      // Все 9 бит переданы (0..8)
        dout_o   <= rx_shift_r[8:1];  // Принятый байт (биты 8..1)
        rx_ack_o <= rx_shift_r[0];    // ACK/NACK (бит 0 = последний принятый)
        state_r  <= ST_IDLE;
        ready_o  <= 1'b1;             // Команда завершена
    end else begin
        bit_cnt_r  <= bit_cnt_r + 1;
        tx_shift_r <= {tx_shift_r[7:0], 1'b0};  // Сдвиг влево
        phase_r    <= 2'd0;            // Следующий бит
    end
end
```

> **Почему завершение байта определяется как `bit_cnt_r == 4'd8`, а не `bit_cnt_r == 4'd9`?** Счётчик bit_cnt_r начинается с 0. Бит-слоты нумеруются 0, 1, 2, ..., 8. Фаза 3 бита N инкрементирует счётчик: после бита 7 счётчик становится 8, и мы переходим к фазе 0 бита 8 (ACK). После бита 8 мы проверяем `bit_cnt_r == 8` и выходим. Это работает потому, что проверка происходит **в фазе 3 бита 8** — когда ACK-бит уже полностью обработан (SDA семплирована в фазе 1 бита 8).

**Важный нюанс: SDA НЕ меняется в фазе 3.**

Заманчиво было бы начать устанавливать SDA для следующего бита прямо в фазе 3. Но это опасно: SCL переходит в LOW одновременно с изменением SDA, и в зависимости от задержек slave может увидеть это как ложный START или STOP.

Безопасный подход: менять SDA только в фазе 0, когда SCL уже гарантированно LOW.

> **Почему сдвиг tx_shift_r происходит в фазе 3, а не в фазе 0?** Сдвиг подготавливает следующий бит в позиции tx_shift_r[8]. В фазе 0 следующего бита мы уже читаем tx_shift_r[8] для управления SDA. Если бы мы сдвигали в фазе 0, то в том же такте пытались бы записать (сдвинуть) и прочитать (выдать) tx_shift_r[8]. В Verilog при неблокирующем присваивании это корректно (читается старое значение), но это нелогично: мы бы выдавали бит, который ещё не сдвинулся. При сдвиге в фазе 3 к моменту фазы 0 следующего бита tx_shift_r[8] уже содержит правильный новый бит.

**Сдвиговые регистры:**

```
tx_shift_r (9 бит): [8][7][6][5][4][3][2][1][0]
                      ↑
                   Выдаётся на SDA

Каждый бит-слот: сдвиг влево, MSB уходит на SDA

Для WRITE: изначально = {D7,D6,D5,D4,D3,D2,D1,D0, 0}
           бит 0: выдаёт D7, бит 1: D6, ... бит 7: D0, бит 8: 0 (release для ACK)

rx_shift_r (9 бит): [8][7][6][5][4][3][2][1][0]
                                              ↑
                                         Принимается с SDA

Каждый бит-слот: сдвиг влево, LSB = sda_i

После 9 бит: rx_shift_r = {D7,D6,D5,D4,D3,D2,D1,D0, ACK}
             dout_o   = rx_shift_r[8:1] = {D7..D0}
             rx_ack_o = rx_shift_r[0]   = ACK
```

> **Почему оба регистра 9 бит, а не 8?** Девятый бит-слот (ACK) обрабатывается точно так же, как биты данных — через тот же сдвиг. Для tx_shift_r бит 0 при WRITE загружается нулём: в бит-слоте 8 (ACK) мастер будет отпускать SDA (`oen = tx_shift_r[8]`, к этому моменту бит 0 сдвинулся в позицию 8, и это 0 — мастер не управляет SDA, слушает ACK). Для READ бит 0 загружается значением `din_i[0]` — это ACK/NACK, который мастер хочет послать.
>
> Единый 9-битный регистр и единый 4-фазный цикл для всех 9 бит-слотов — это ключ к простоте ST_DATA. Никаких специальных случаев для ACK-бита в FSM.

### 4.4. ST_STOP — генерация условия STOP

**Определение STOP:** SDA переходит из 0 в 1, пока SCL = 1.

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        STOP!
SDA: ─────────────────────────────────────────╱
                                           __╱──────────────────
SCL: ────────────╱
              __╱────────────────────────────────────────────────

     SDA=0, SCL=0      SDA=0, SCL=1   SDA=1, SCL=1   (hold)
     (подготовка)       (ждём SCL)     (STOP!)        (готово)
```

```verilog
ST_STOP: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b0;    // SDA = LOW (подготовка)
            scl_oen_o <= 1'b0;    // SCL = LOW
            phase_r   <= 2'd1;
        end
        2'd1: begin
            sda_oen_o <= 1'b0;    // SDA остаётся LOW
            scl_oen_o <= 1'b1;    // Отпускаем SCL → HIGH
            if (scl_i) phase_r <= 2'd2;  // Clock stretching
        end
        2'd2: begin
            sda_oen_o <= 1'b1;    // SDA → HIGH !!! ← ВОТ ОН, STOP
            scl_oen_o <= 1'b1;    // SCL остаётся HIGH
            phase_r   <= 2'd3;
        end
        2'd3: begin
            // Hold — обе линии высокие, шина свободна
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;
        end
    endcase
end
```

**Почему фаза 0 нужна?** После предыдущей команды (WRITE/READ) SCL = LOW, но SDA может быть в любом состоянии. Нам нужно гарантировать SDA = LOW перед STOP, иначе переход SDA 0→1 при SCL=1 не произойдёт. Фаза 0 устанавливает начальное условие.

> **Подробнее о фазе 0 STOP.** Рассмотрим ситуацию: предыдущая команда — WRITE, последний бит данных = 1. После фазы 3 ST_DATA: SCL=LOW, SDA=HIGH (мастер отпустил). Если сразу перейти к фазе 1 STOP (поднять SCL), мы получим: SCL=1, SDA=1 → SDA уже HIGH, и мы не можем сделать переход 0→1. Более того, если в этот момент SDA изменится (slave отпустит ACK), slave может увидеть STOP раньше времени.
>
> Фаза 0 решает это: принудительно устанавливаем SDA=LOW при SCL=LOW. Теперь в фазе 2, когда мы отпускаем SDA при SCL=HIGH, переход 0→1 гарантирован.
>
> **Альтернатива:** можно было бы в ST_DATA фазе 3 принудительно опускать SDA (при cmd_r == CMD_WRITE). Но это «просачивание знания» о следующей команде в текущее состояние, что нарушает модульность. ST_DATA не должно знать, что после него будет STOP.

### 4.5. ST_RESTART — повторный START

RESTART — это START, который выполняется без предварительного STOP. Разница: перед обычным START обе линии уже высокие (шина свободна). Перед RESTART SCL = LOW (мы посередине транзакции).

```
        Фаза 0         Фаза 1         Фаза 2         Фаза 3
                                        START!
SDA: ─────────╱
           __╱──────────────────────╲
SCL: ────────────────╱               ╲──────────────╲
                  __╱──────────────────              ╲──────────

     SDA=1, SCL=0   SDA=1, SCL=1    SDA=0, SCL=1   SDA=0, SCL=0
     (отпустить SDA) (ждём SCL)      (START!)        (готово)
```

> **Почему нельзя использовать ST_START для RESTART?** При обычном START мы начинаем с SDA=1, SCL=1 (шина свободна). При RESTART мы начинаем с SCL=LOW (мастер удерживает после предыдущей передачи данных), SDA может быть в любом состоянии. Ключевое отличие:
>
> - **ST_START фаза 0:** и SCL, и SDA уже HIGH, нужно только дождаться SCL (stretching).
> - **ST_RESTART фаза 0:** SCL LOW, нужно сначала отпустить SDA (HIGH), и только потом поднимать SCL.
>
> Если бы мы использовали ST_START для RESTART, в фазе 0 мы бы отпустили SCL (`scl_oen = 1`). Но SDA может быть LOW. Получается: SCL→HIGH при SDA=LOW → slave видит «данные», а не подготовку к START. Потом SDA→HIGH при SCL=HIGH → slave видит STOP! Не START. Именно поэтому RESTART имеет другой порядок: сначала SDA→HIGH при SCL=LOW (фаза 0), потом SCL→HIGH при SDA=HIGH (фаза 1), потом SDA→LOW при SCL=HIGH (фаза 2) — это правильный START.

```verilog
ST_RESTART: begin
    case (phase_r)
        2'd0: begin
            sda_oen_o <= 1'b1;    // Отпустить SDA → HIGH
            scl_oen_o <= 1'b0;    // SCL остаётся LOW
            phase_r   <= 2'd1;
        end
        2'd1: begin
            sda_oen_o <= 1'b1;    // SDA HIGH
            scl_oen_o <= 1'b1;    // Отпускаем SCL → HIGH
            if (scl_i) phase_r <= 2'd2;  // Clock stretching
        end
        2'd2: begin
            sda_oen_o <= 1'b0;    // SDA → LOW !!! ← START
            scl_oen_o <= 1'b1;    // SCL HIGH
            phase_r   <= 2'd3;
        end
        2'd3: begin
            sda_oen_o <= 1'b0;    // SDA LOW
            scl_oen_o <= 1'b0;    // SCL → LOW
            state_r   <= ST_IDLE;
            ready_o   <= 1'b1;
        end
    endcase
end
```

**Порядок в фазе 0 критичен:** сначала отпускаем SDA (фаза 0), и только потом SCL (фаза 1). Если отпустить SCL раньше SDA, при SCL=1 и SDA 0→1 получится ложный STOP.

## Часть 5. Мониторинг шины

### 5.1. Отслеживание занятости шины (busy)

Ядро должно знать, занята ли шина. Это нужно для:
- Определения, какую команду выдать: START (шина свободна) или RESTART (шина занята)
- Управления линиями в IDLE (отпускать или держать)

Детектируем START и STOP **по реальным сигналам на шине**:

```verilog
// Детектор фронтов SDA
reg sda_d_r;
always @(posedge clk_i) sda_d_r <= sda_i;

wire sda_rising  =  sda_i & ~sda_d_r;  // SDA: 0 → 1
wire sda_falling = ~sda_i &  sda_d_r;  // SDA: 1 → 0

// START = SDA падает при SCL=1
wire start_on_bus = sda_falling & scl_i;

// STOP = SDA растёт при SCL=1
wire stop_on_bus = sda_rising & scl_i;

// Флаг занятости
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i)          busy_o <= 1'b0;
    else if (start_on_bus) busy_o <= 1'b1;
    else if (stop_on_bus)  busy_o <= 1'b0;
end
```

**Важно:** это работает на основном клоке `clk_i`, а не на `ena_i`. Фронты SDA/SCL могут быть очень короткими (наносекунды), и мы не должны их пропустить.

> **Почему детектор работает на clk_i, а не на ena_i?** ena_i приходит раз в ~125 тактов (при 100 кГц SCL). Фронт SDA при SCL=HIGH (START или STOP) длится ≈ один период ena_i (один четвертьпериод SCL). Если детектор работает на ena_i, он может **пропустить** короткий фронт: например, если START произошёл ровно между двумя импульсами ena_i. Детектирование на clk_i гарантирует, что мы увидим любой фронт SDA длительностью ≥ 1 период clk_i (20 нс при 50 МГц).
>
> **Почему не отдельный асинхронный детектор на edge-triggered latch?** Асинхронная логика плохо синтезируется на FPGA (нет гарантии метастабильности, зависимость от routing delays). Синхронный детектор фронтов на основном клоке — стандартный и надёжный паттерн.

> **Почему busy_o отслеживает START/STOP на шине, а не просто является флагом FSM?** Можно было бы установить `busy_o = 1` при входе в ST_START и сбросить при выходе из ST_STOP. Но тогда busy_o отражал бы состояние **нашего** мастера, а не **шины**. В multi-master конфигурации другой мастер может генерировать START/STOP, и мы должны это видеть. Кроме того, если мы потеряли арбитраж и вышли в IDLE, но другой мастер продолжает транзакцию — шина всё ещё busy, и мы не должны отпускать линии.

### 5.2. Обнаружение потери арбитража

**Когда мастер теряет арбитраж?** Когда он отпустил SDA (ожидает, что линия будет HIGH), но другой мастер или slave тянет SDA к LOW.

Проверяем это в двух ситуациях:

**Ситуация 1: во время передачи данных (ST_DATA, фаза 1)**

```verilog
wire al_data = (state_r == ST_DATA) &&
               (phase_r == 2'd1) &&    // SCL только поднялся
               scl_i &&                 // SCL реально HIGH
               !sda_input_mode &&       // Мастер должен управлять SDA
               sda_oen_o &&            // Мастер отпустил SDA (ожидает 1)
               !sda_i;                 // Но SDA = 0! Кто-то тянет!
```

> **Почему проверка именно в фазе 1?** Потому что фаза 1 — это момент, когда SCL только что стал HIGH, и данные на SDA стабильны. Это единственный момент, когда проверка арбитража имеет смысл: мастер выдал бит (в фазе 0), поднял SCL (в фазе 1), и теперь может сравнить реальное значение SDA с ожидаемым.
>
> **Почему только при `!sda_input_mode`?** Если в данный бит-слот данные гонит slave (READ биты 0-7 или WRITE ACK), то низкий уровень SDA — это нормальная передача от slave, а не конфликт арбитража.
>
> **Почему только при `sda_oen_o == 1`?** Мастер проверяет арбитраж только когда он **отпустил** SDA (выдаёт «1»). Если мастер сам притянул SDA к 0 (`sda_oen_o = 0`), то SDA = 0 — это его собственная работа, конфликта нет. Потеря арбитража возможна только при передаче бита «1»: мастер отпустил линию, ожидая HIGH, но другой мастер тянет к LOW.

**Ситуация 2: во время START/RESTART**

```verilog
wire al_start =
    (state_r == ST_START   && phase_r == 2'd0 && sda_oen_o && scl_i && !sda_i) ||
    (state_r == ST_RESTART && phase_r == 2'd1 && sda_oen_o && scl_i && !sda_i);
```

> **Почему проверяем при START?** Два мастера могут одновременно начать генерацию START. В фазе 0 ST_START оба отпускают SDA и SCL. Если один из них увидит SDA=0 при ожидании SDA=1, значит другой мастер уже опустил SDA — уже «начал» START раньше. Первый мастер должен уступить.
>
> **Почему при RESTART проверяем в фазе 1, а не в фазе 0?** В фазе 0 RESTART SCL=LOW (мастер ещё не отпустил). При SCL=LOW изменение SDA не является START/STOP, и арбитраж по SDA не имеет смысла. В фазе 1 мастер отпустил и SCL, и SDA — вот теперь можно проверять.

**Реакция на потерю арбитража — немедленное освобождение шины:**

```verilog
if (al_event && state_r != ST_IDLE) begin
    state_r   <= ST_IDLE;
    scl_oen_o <= 1'b1;   // Отпустить SCL
    sda_oen_o <= 1'b1;   // Отпустить SDA
    ready_o   <= 1'b1;   // Сообщить контроллеру
end
```

> **Почему «немедленное» освобождение, а не «дождаться текущего бита»?** Спецификация I2C однозначна: мастер, потерявший арбитраж, **должен немедленно** прекратить управление шиной. Продолжение передачи даже одного бита может помешать выигравшему мастеру. Поэтому мы безусловно отпускаем обе линии и возвращаемся в IDLE, не дожидаясь окончания текущей фазы.

Флаг `arb_lost_o` — sticky (липкий): устанавливается при потере арбитража и не сбрасывается автоматически. Вышестоящий контроллер должен явно сбросить его через `arb_lost_clear_i` перед новой попыткой.

> **Почему флаг sticky, а не auto-clear?** Если бы arb_lost_o сбрасывался автоматически (скажем, при следующем ena_i), вышестоящий контроллер мог бы не успеть его прочитать — и начать новую транзакцию, не зная об ошибке. Sticky-флаг гарантирует, что информация о потере арбитража не будет потеряна, пока контроллер её не обработает.

## Часть 6. Clock Stretching

### 6.1. Что это такое

Clock stretching — механизм, позволяющий slave (или другому мастеру) замедлить шину. Slave удерживает SCL в LOW, когда ему нужно больше времени.

```
Мастер отпускает SCL:     scl_oen_o <= 1'b1;
Ожидает SCL = HIGH:       if (scl_i) ...
Slave держит SCL LOW:     scl_i = 0  → мастер ждёт
Slave отпускает:          scl_i = 1  → мастер продолжает
```

> **Почему clock stretching обязателен, а не опционален?** Спецификация I2C *требует* от мастера поддержки clock stretching. Многие реальные устройства (EEPROM при page write, сложные датчики) активно используют stretching для сигнализации «не готов». Мастер без поддержки stretching нарушит тайминг при работе с такими устройствами и получит повреждённые данные.
>
> **Можно ли добавить таймаут?** Спецификация I2C не определяет максимальное время stretching — теоретически slave может держать SCL сколь угодно долго. Однако SMBus (расширение I2C) определяет таймаут 35 мс. Мы **не** реализуем таймаут в ядре, потому что:
>
> 1. Ядро — универсальное. Не все применения используют SMBus.
> 2. Таймаут — это политика, а не протокол. Решение «что делать при зависании» (reset шины? перезапуск? уведомить CPU?) зависит от системы.
> 3. Таймаут тривиально реализуется в обёртке (AXI/Avalon слой) поверх ядра.

### 6.2. Где реализовано

Clock stretching обрабатывается в **фазе 1** каждого состояния — там, где мастер отпускает SCL и ожидает, что он поднимется:

```verilog
// ST_START, фаза 0:
if (scl_i) phase_r <= 2'd1;   // Ждём SCL HIGH

// ST_DATA, фаза 1:
if (scl_i) begin               // Ждём SCL HIGH
    rx_shift_r <= {rx_shift_r[7:0], sda_i};
    phase_r    <= 2'd2;
end

// ST_STOP, фаза 1:
if (scl_i) phase_r <= 2'd2;   // Ждём SCL HIGH

// ST_RESTART, фаза 1:
if (scl_i) phase_r <= 2'd2;   // Ждём SCL HIGH
```

Паттерн один: `if (scl_i) advance; else stay`.

> **Почему в ST_START stretching в фазе 0, а не в фазе 1?** В ST_START фаза 0 — это момент, когда мы отпускаем SCL (`scl_oen = 1`) и ждём его реального подъёма. Это аналогично фазе 1 в других состояниях. Нумерация фаз смещена из-за того, что последовательность для START отличается: нам нужно дождаться SCL=HIGH *до* того, как мы будем менять SDA. В ST_DATA/ST_STOP/ST_RESTART мы сначала устанавливаем SDA (фаза 0), а потом поднимаем SCL (фаза 1). В ST_START мы сразу начинаем с высокого SCL (фаза 0).

Нет таймаута — по спецификации I2C, мастер обязан ждать бесконечно. Таймаут, если нужен, реализуется на уровне выше (в контроллере).

## Часть 7. Пошаговый рецепт проектирования

Если бы вы проектировали это ядро с нуля, вот порядок шагов:

> **Почему именно этот порядок?** Шаги организованы по принципу «от внешнего к внутреннему» и «от простого к сложному»:
>
> 1. Сначала определяем **что** модуль делает (интерфейс) — это контракт с внешним миром.
> 2. Потом определяем **как** тактируется (модель тактирования) — это базовое ограничение для всей логики.
> 3. Затем проектируем **микроархитектуру** (фазы бита) — базовый строительный блок.
> 4. Строим **FSM** поверх фаз — высокоуровневая логика.
> 5. Добавляем **регистры данных** — путь данных.
> 6. Добавляем **обработку исключений** (stretching, арбитраж) — defensive logic.
> 7. Шлифуем **граничные случаи** — это всегда самое сложное и последнее.
>
> Попытка проектировать в обратном порядке (сначала FSM, потом интерфейс) приводит к переделкам, когда интерфейс не стыкуется с уже реализованной логикой.

### Шаг 1. Определить интерфейс

Нарисуйте порты модуля. Решите:
- Как вышестоящий контроллер будет давать команды?
- Какой набор команд нужен? (START, WRITE, READ, STOP — минимум)
- Какой протокол рукопожатия? (ready/valid — самый простой)

> **Почему начинаем с интерфейса?** Интерфейс — это контракт модуля с окружением. Изменение интерфейса после реализации ядра каскадно затрагивает все модули, которые его используют (AXI wrapper, testbench, документацию). Фиксация интерфейса на раннем этапе экономит огромное количество переделок.
>
> На этом шаге мы выбрали ready/valid рукопожатие вместо FIFO или req/ack (см. обоснование в разделе 1.2), 5 атомарных команд вместо составных, и open-drain OEN-интерфейс для SCL/SDA вместо прямого управления (потому что open-drain — требование протокола).

### Шаг 2. Выбрать модель тактирования

Решите: будет ли ядро работать на SCL-частоте или на основном клоке с enable?

**Рекомендация:** основной клок + enable (`ena_i`). Это проще для синтеза, надёжнее для CDC (clock domain crossing), и позволяет менять скорость I2C без изменения ядра.

> **Почему не отдельный SCL-клок?** Создание отдельного клокового домена для SCL (через PLL или делитель) кажется интуитивным — ядро работает на «родной» I2C-частоте. Но это вызывает ряд проблем:
>
> 1. **CDC:** все сигналы между основным клоком и SCL-клоком нуждаются в синхронизаторах. cmd_valid_i, cmd_i, din_i, ready_o, dout_o — каждый требует 2-стадийного синхронизатора или FIFO для пересечения доменов. Это многократно увеличивает сложность.
> 2. **Jitter:** PLL/делитель вносит джиттер, который в случае I2C не критичен, но создаёт дополнительные заботы при STA (static timing analysis).
> 3. **Ресурсы:** отдельный клоковый домен на FPGA требует глобальной клоковой сети, которых ограниченное количество.
> 4. **Clock stretching:** если SCL «клок» управляется внешним slave (stretching), PLL не может за ним следовать.
>
> Enable-подход (`ena_i`) решает все эти проблемы: один клоковый домен, нет CDC, нет PLL, stretching обрабатывается обычным if.

### Шаг 3. Определить фазы бита

Разбейте один I2C-бит на фазы. 4 фазы — оптимальный баланс между простотой и соблюдением тайминга.

Нарисуйте тайминг-диаграмму для каждого типа операции (START, data bit, STOP, RESTART) — это ваш «чертёж».

> **Совет:** тайминг-диаграммы лучше рисовать на бумаге или в WaveDrom, а не «в голове». Ошибки с порядком переключения SCL/SDA (создание ложных START/STOP) — самая частая причина багов в I2C-ядрах, и диаграмма позволяет их увидеть до написания кода.

### Шаг 4. Спроектировать FSM верхнего уровня

Определите состояния. Для каждого состояния нарисуйте таблицу: «фаза → SCL_oen, SDA_oen, действие, условие перехода».

Пример для ST_START:

| Фаза | SCL_oen | SDA_oen | Действие | Переход |
|------|---------|---------|----------|---------|
| 0 | 1 | 1 | — | scl_i=1 → фаза 1 |
| 1 | 1 | 1 | hold | → фаза 2 |
| 2 | 1 | **0** | START! | → фаза 3 |
| 3 | **0** | 0 | — | → IDLE, ready=1 |

> **Почему таблица, а не сразу Verilog?** Таблица позволяет проверить логику до написания кода. Каждая строка — одна фаза, каждый столбец — одно решение. Можно проверить: (1) нет ли момента, когда SCL и SDA меняются одновременно; (2) везде ли, где SCL отпускается, есть проверка scl_i для stretching; (3) правильный ли порядок фронтов для START/STOP.

### Шаг 5. Спроектировать сдвиговые регистры

Определите:
- Ширину (9 бит — 8 данных + ACK)
- Начальные значения для WRITE и READ
- Направление сдвига (влево, MSB first)
- Когда семплировать входные данные

> **Почему 9 бит, а не 8 + отдельный бит ACK?** Можно было бы иметь 8-битный регистр данных и отдельный флаг/регистр для ACK. Но тогда для бита 8 (ACK) нужна бы была специальная ветка в ST_DATA: «если bit_cnt == 8, вместо shift-регистра использовать отдельный ACK-бит». Это усложняет фазу 0 и добавляет мультиплексор. При 9-битном регистре все 9 бит-слотов обрабатываются единообразно — bit 8 «просто ещё один бит» в регистре, никаких ветвлений.

### Шаг 6. Добавить clock stretching

В каждом месте, где вы отпускаете SCL, добавьте проверку `if (scl_i)`. Это буквально одна строка в каждой фазе, но без неё ядро не будет работать с медленными slave.

> **Совет по отладке:** если ваше ядро «зависает» и не завершает транзакцию — в 90% случаев проблема в clock stretching. Либо вы забыли проверку `if (scl_i)` в одном из состояний, либо slave реально тянет SCL и ваш тестбенч не моделирует это.

### Шаг 7. Добавить мониторинг шины

Реализуйте:
- Детектор START/STOP на шине → флаг busy
- Детектор потери арбитража → флаг arb_lost + немедленный выход в IDLE

> **Почему мониторинг добавляется на позднем этапе?** Мониторинг — это «наблюдательная» логика, которая не влияет на основной путь данных. Она отслеживает побочные эффекты (чужие START/STOP на шине, конфликты SDA) и устанавливает флаги. Основной FSM можно полностью отладить без мониторинга (в single-master конфигурации без stretching), а потом добавить мониторинг поверх.

### Шаг 8. Обработать граничные случаи в IDLE

Это самая коварная часть. Продумайте:
- Что если в IDLE приходит команда, но шина занята другим мастером?
- Что если арбитраж потерян — принимать ли новые команды?
- Когда безопасно отпускать линии, а когда нет?

> **Почему этот шаг последний?** Потому что граничные случаи IDLE зависят от *всех* остальных компонентов: busy_o (из мониторинга), arb_lost_o (из арбитража), текущего состояния SCL/SDA (из FSM). Пока эти компоненты не спроектированы, невозможно корректно обработать граничные случаи.

### Шаг 9. Написать тестбенч

Верифицируйте каждый сценарий:
1. Одиночный WRITE + ACK
2. Одиночный READ + NACK
3. Полная транзакция: START + WRITE addr + WRITE data + STOP
4. Repeated START (RESTART)
5. NACK от slave (устройство не отвечает)
6. Clock stretching (slave задерживает SCL)
7. Arbitration lost (если нужно multi-master)
8. Reset во время транзакции

> **Почему тестбенч — последний шаг, а не первый (TDD)?** В FPGA-разработке TDD (test-driven development) менее практичен, чем в software, по нескольким причинам:
>
> 1. **Тестбенч зависит от интерфейса.** Писать тест до того, как определены порты модуля — нонсенс. А интерфейс определяется на шаге 1.
> 2. **I2C-тестбенч требует slave-модель.** Slave-модель — это сложный модуль, который сам по себе может содержать баги. Проектировать slave-модель параллельно с ядром — двойная когнитивная нагрузка.
> 3. **Итеративность.** На практике тестбенч пишется параллельно с реализацией: написал ST_START — протестировал START. Написал ST_DATA — добавил WRITE-тест. Формально это не «шаг 9 после шага 8», а «шаг 9 выполняется параллельно с шагами 4-8».

## Часть 8. Типичные ошибки при проектировании

### Ошибка 1: SDA и SCL меняются одновременно

```
Неправильно: в одной фазе одновременно меняем SCL→LOW и SDA→HIGH
             Slave может увидеть STOP (SDA 0→1 при SCL=1) до того,
             как SCL успеет упасть.
```

**Решение:** всегда разделять изменения SCL и SDA на разные фазы.

> **Почему это так опасно на практике?** На реальной плате сигналы SCL и SDA имеют разные задержки (разная длина дорожек, разные pull-up резисторы, разные ёмкости). Если в одном такте FPGA выдаёт SCL→LOW и SDA→HIGH, на шине SCL может упасть на 5 нс позже SDA из-за большей ёмкости SCL-линии. В эти 5 нс slave видит: SCL=HIGH, SDA 0→1 = STOP. Транзакция разрушена.

### Ошибка 2: забыть clock stretching

```
Неправильно: scl_oen_o <= 1'b1;
             phase_r <= 2'd2;  // Сразу переходим дальше

Правильно:   scl_oen_o <= 1'b1;
             if (scl_i) phase_r <= 2'd2;  // Ждём реального HIGH
```

Без этого ядро будет работать только с «идеальными» slave.

### Ошибка 3: безусловный release линий в IDLE

```
Неправильно:
    ST_IDLE: begin
        scl_oen_o <= 1'b1;  // Всегда отпускаем
        sda_oen_o <= 1'b1;

Правильно:
    ST_IDLE: begin
        if (!busy_o) begin   // Только если шина свободна
            scl_oen_o <= 1'b1;
            sda_oen_o <= 1'b1;
        end
```

### Ошибка 4: семплирование SDA при SCL=LOW

```
Неправильно: (в фазе 0)
    rx_shift_r <= {rx_shift_r[7:0], sda_i};  // SCL ещё LOW!

Правильно: (в фазе 1, после scl_i check)
    if (scl_i)
        rx_shift_r <= {rx_shift_r[7:0], sda_i};  // SCL HIGH ✓
```

### Ошибка 5: не сбрасывать ready при приёме команды

```
Неправильно:
    if (cmd_valid_i) begin
        // Забыли ready_o <= 0;
        state_r <= ...;
    end

Правильно:
    if (cmd_valid_i) begin
        ready_o <= 1'b0;    // Сигнализируем: команда принята
        state_r <= ...;
    end
```

Без этого контроллер выше может повторно выдать ту же команду.

```
│  │    │                                                    │  │
│  │  [phase_r: 0→1→2→3]  [bit_cnt_r: 0→1→...→8]          │  │
│  └─────────────────────────────────────────────────────────┘  │
│                                                               │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────────┐   │
```

### Метрики модуля

| Метрика | Значение |
|---------|----------|
| Состояний FSM | 5 (IDLE, START, DATA, STOP, RESTART) |
| Фаз на бит | 4 |
| Бит-слотов на байт | 9 (8 data + 1 ACK) |
| Фаз на полный байт | 36 |
| ena_i тиков на один байт | 36 |
| SCL-периодов на байт | 9 |
| Минимум LUT (оценка) | ~150-200 (Cyclone IV) |
| Регистров | ~40 бит |

> **Почему модуль получился компактным (~150-200 LUT)?** Именно потому, что мы последовательно избегали дублирования:
>
> - Одно состояние ST_DATA для WRITE и READ (вместо двух)
> - Один 9-битный регистр вместо 8-битного + отдельного ACK
> - Единообразные 4 фазы для всех операций
> - Внешний прескалер (нет счётчика деления внутри ядра)
> - Детектор фронтов на 2 регистрах (sda_d_r + комбинаторика)
>
> Каждое архитектурное решение, описанное в этом документе, было направлено на минимизацию логики при сохранении полного соответствия спецификации I2C.
