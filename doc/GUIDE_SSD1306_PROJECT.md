# Проектирование аппаратного контроллера SSD1306 OLED с нуля

## Введение

Этот документ подробно объясняет, как спроектировать аппаратный проект для FPGA, который инициализирует OLED-дисплей SSD1306 (128×64 пикселей) по шине I2C и отображает на нём тестовую картинку. Проект не использует процессор — вся логика реализована в железе.

**Что мы построим:**

```
Кнопка → [FPGA] → I2C → [SSD1306 OLED] → картинка на экране
                    │
                    └→ [LED + 7-сегмент] → статус и прогресс
```

**Ключевая задача:** передать по I2C более 1000 байт данных в один дисплей. Это отличается от типичных I2C-примеров, где передаётся 1-2 байта. Здесь мы отправляем:
- 32 байта команд инициализации
- 1025 байт данных фреймбуфера (1 управляющий + 1024 пикселя)

**Целевая платформа:** ALINX AX301, Cyclone IV E (EP4CE6F17C8), 50 МГц.

## Часть 1. Дисплей SSD1306 — как он работает

### 1.1. Физический уровень — I2C

SSD1306 подключается по I2C с адресом **0x3C** (7-бит; при SA0=GND) или 0x3D (SA0=VCC). В I2C-транзакции адрес передаётся со сдвигом: `0x3C << 1 = 0x78` для записи.

Подключение минимально — 4 провода:

```
FPGA            SSD1306 OLED
────────        ──────────────
3.3V ──────────── VCC
GND  ──────────── GND
SDA ──────┬────── SDA
     R(4.7kΩ)
SCL ──────┬────── SCL
     R(4.7kΩ)
```

Pull-up резисторы (4.7 кОм к 3.3 В) — обязательны для I2C. На плате AX301 они уже установлены для EEPROM, который делит ту же шину.

### 1.2. Протокол SSD1306 — управляющий байт

После адреса SSD1306 ожидает **управляющий байт** (control byte), определяющий тип последующих данных:

```
Управляющий байт: [Co] [D/C#] [0] [0] [0] [0] [0] [0]
                   bit7  bit6  bit5 ................. bit0
```

| Co | D/C# | Байт | Значение |
|----|------|------|----------|
| 0 | 0 | **0x00** | Все следующие байты — **команды** |
| 0 | 1 | **0x40** | Все следующие байты — **данные** (пиксели) |
| 1 | 0 | 0x80 | Следующий один байт — команда, потом снова control byte |
| 1 | 1 | 0xC0 | Следующий один байт — данные, потом снова control byte |

**Ключевой момент:** когда `Co=0`, после управляющего байта можно передавать **неограниченное** число байт одного типа до STOP. Это позволяет отправить весь фреймбуфер (1024 байта) за одну I2C-транзакцию:

```
I2C: START → 0x78 → 0x40 → pixel[0] → pixel[1] → ... → pixel[1023] → STOP
              addr    ctrl    ←── 1024 байта данных ──→
```

### 1.3. Организация видеопамяти

SSD1306 128×64 — это 128 столбцов × 64 строки = 8192 пикселя = 1024 байта.

Память разделена на **8 страниц** (pages), каждая высотой 8 пикселей:

```
        Column 0       Column 1       Column 2       ...     Column 127
       ┌──────────┐   ┌──────────┐   ┌──────────┐          ┌──────────┐
Page 0 │ bit0(top)│   │ bit0     │   │ bit0     │          │ bit0     │  строки 0-7
       │ bit1     │   │ bit1     │   │ bit1     │          │ bit1     │
       │ bit2     │   │ bit2     │   │ bit2     │          │ bit2     │
       │ bit3     │   │ bit3     │   │ bit3     │          │ bit3     │
       │ bit4     │   │ bit4     │   │ bit4     │          │ bit4     │
       │ bit5     │   │ bit5     │   │ bit5     │          │ bit5     │
       │ bit6     │   │ bit6     │   │ bit6     │          │ bit6     │
       │ bit7(bot)│   │ bit7     │   │ bit7     │          │ bit7     │
       └──────────┘   └──────────┘   └──────────┘          └──────────┘
Page 1 │ ...      │   │ ...      │   │ ...      │          │ ...      │  строки 8-15
       └──────────┘   └──────────┘   └──────────┘          └──────────┘
  ...
Page 7 │ ...      │   │ ...      │   │ ...      │          │ ...      │  строки 56-63
       └──────────┘   └──────────┘   └──────────┘          └──────────┘
```

Каждый байт видеопамяти — это **8 вертикальных пикселей** в одном столбце одной страницы:
- **bit 0** = верхний пиксель (ближе к верху экрана)
- **bit 7** = нижний пиксель
- `1` = пиксель горит (белый), `0` = пиксель не горит (чёрный)

**Пример:** байт `0x0F` (`00001111` в двоичном) в столбце 10 страницы 2 означает:
- Строки 16-19 (верхние 4 бита = 0) — чёрные
- Строки 20-23 (нижние 4 бита = 1) — белые

### 1.4. Режимы адресации

SSD1306 поддерживает 3 режима адресации памяти. Мы используем **горизонтальный** (Horizontal Addressing Mode, команда `0x20, 0x00`):

```
Порядок записи байт в горизонтальном режиме:

Page 0: [col0] → [col1] → [col2] → ... → [col127] → (переход на Page 1)
Page 1: [col0] → [col1] → ... → [col127] → (переход на Page 2)
...
Page 7: [col0] → ... → [col127] → (возврат на Page 0)
```

То есть байты записываются слева направо, сверху вниз — как текст. Указатель автоматически переходит на следующую страницу. Это позволяет отправить все 1024 байта подряд без дополнительных команд адресации.

### 1.5. Последовательность инициализации

SSD1306 после включения питания требует настройки через серию команд. Каждая команда — 1 или 2 байта. Вот полная последовательность:

```
Команда    Аргумент  Описание
────────── ──────── ─────────────────────────────────────────────
0xAE       —        Display OFF (выключить дисплей перед настройкой)
0xD5       0x80     Частота обновления (по умолчанию)
0xA8       0x3F     Multiplex ratio = 63 (64 строки, нумерация с 0)
0xD3       0x00     Смещение дисплея = 0
0x40       —        Начальная строка = 0
0x8D       0x14     Включить встроенный DC-DC конвертер (charge pump)
0x20       0x00     Горизонтальный режим адресации
0xA1       —        Зеркалирование столбцов (col 127 = SEG0)
0xC8       —        Зеркалирование строк (COM63 → COM0)
0xDA       0x12     Конфигурация COM-пинов (для 128×64)
0x81       0xCF     Контрастность = 207 (из 255)
0xD9       0xF1     Период предзаряда (phase1=1, phase2=15)
0xDB       0x40     Уровень VCOMH ≈ 0.77 × VCC
0xA4       —        Отображать содержимое RAM (а не все пиксели вкл.)
0xA6       —        Нормальное отображение (не инверсия)
0x21       0x00     Диапазон столбцов: начало = 0
           0x7F                         конец = 127
0x22       0x00     Диапазон страниц: начало = 0
           0x07                        конец = 7
0xAF       —        Display ON (включить дисплей)
```

Всего 31 командный байт. Они передаются в одной I2C-транзакции с префиксом `0x00`:

```
I2C: START → 0x78 → 0x00 → 0xAE → 0xD5 → 0x80 → ... → 0xAF → STOP
              addr    ctrl   ←── 31 командный байт ──→
```

**Зачем 0xA1 и 0xC8?** Это зеркалирование по горизонтали и вертикали. Без них изображение будет отображаться зеркально и/или перевёрнуто (зависит от того, как припаян дисплей к плате модуля).

**Зачем 0x8D, 0x14?** Charge pump. У SSD1306 OLED-пиксели работают от ~7-8 В, но модуль питается от 3.3 В. Встроенный преобразователь (charge pump) повышает напряжение. Без него экран останется тёмным.

## Часть 2. Архитектура проекта — от задачи к модулям

### 2.1. Декомпозиция задачи

Задача: «по кнопке инициализировать SSD1306 и показать картинку».

Разбиваем на подзадачи:

```
1. Обработать нажатие кнопки               → ax_debounce
2. Сгенерировать тактирование для I2C      → prescaler
3. Выполнить I2C-операции на уровне бит    → i2c_master_core (уже есть)
4. Автоматически отправить N байт по I2C   → i2c_burst_writer (НОВЫЙ)
5. Управлять последовательностью init+data → ssd1306_ctrl (НОВЫЙ)
6. Показать статус на дисплее и LED        → seg_scan + логика
7. Связать всё вместе                      → ssd1306_test_top
```

### 2.2. Иерархия модулей

```
ssd1306_test_top
│
├── ax_debounce              Антидребезг кнопки
│
├── prescaler (в top-level)  50 МГц → 100 кГц тик для I2C
│
├── i2c_master_core          Ядро I2C (побитовый уровень)
│     ↑ cmd_valid, cmd, din
│     ↓ ready, rx_ack
│
├── ssd1306_ctrl             Управление SSD1306
│   │
│   ├── init_rom()           ROM с командами инициализации
│   ├── gen_pattern()        Процедурный генератор картинки
│   │
│   └── i2c_burst_writer    Пакетная запись по I2C
│         ↑ start, slave_addr, byte_count
│         ↓ done, error
│         ↔ data_req / data / data_valid  (от ssd1306_ctrl)
│         ↔ cmd_valid / cmd / din / ready (к i2c_master_core)
│
├── seg_scan                 7-сегментный дисплей
│
└── LED logic                Светодиоды статуса
```

### 2.3. Потоки данных

Проследим, как байт пикселя попадает на I2C-шину:

```
Шаг 1. ssd1306_ctrl генерирует байт:
        gen_pattern(col=5, page=2) → 8'hFF

Шаг 2. burst_writer запрашивает байт:
        data_req_o = 1

Шаг 3. ssd1306_ctrl отдаёт байт:
        data_i = 8'hFF, data_valid_i = 1

Шаг 4. burst_writer выдаёт команду ядру:
        cmd_valid = 1, cmd = CMD_WRITE, din = 8'hFF

Шаг 5. i2c_master_core передаёт байт по I2C:
        9 бит-слотов × 4 фазы = 36 тиков ena_i

Шаг 6. Ядро завершает:
        ready = 1, rx_ack = 0 (ACK от SSD1306)

Шаг 7. burst_writer запрашивает следующий байт...
        (повторяется 1025 раз для фреймбуфера)
```

## Часть 3. Модуль i2c_burst_writer — пакетная запись

### 3.1. Зачем нужен этот модуль

Ядро `i2c_master_core` работает на уровне отдельных команд: один START, один WRITE, один STOP. Чтобы отправить N байт, нужно:

```
Без burst_writer (вручную):         С burst_writer (автоматически):
  CMD_START                           start_i = 1
  CMD_WRITE (addr)                    slave_addr_i = 0x3C
  CMD_WRITE (data[0])                 byte_count_i = 1025
  CMD_WRITE (data[1])                 → всё остальное автоматически
  ...                                 ← done_o = 1
  CMD_WRITE (data[1024])
  CMD_STOP
  (1028 отдельных команд)
```

### 3.2. FSM burst_writer

Конечный автомат burst_writer — линейная последовательность с ветвлениями на ошибках:

```
                ┌──────┐
    start_i ──► │ IDLE │
                └──┬───┘
                   ▼
           ┌───────────────┐
           │  START_CMD    │  cmd = CMD_START
           │  START_WAIT   │  ждём ready
           └──────┬────────┘
                  ▼
           ┌───────────────┐
           │  ADDR_CMD     │  cmd = CMD_WRITE, din = {addr, 0}
           │  ADDR_WAIT    │  ждём ready
           └──────┬────────┘
                  ▼
           ┌── NACK? ──── Да ──────────────┐
           │                                │
           Нет                              │
           ▼                                │
    ┌──► DATA_REQ ──┐                      │
    │               ▼                       │
    │    DATA_CMD   │  cmd = CMD_WRITE     │
    │    DATA_WAIT  │  ждём ready           │
    │       │       │                       │
    │       ▼       │                       │
    │  NACK или     │                       │
    │  cnt == 0? ───┼───────────────────────┤
    │       │       │                       │
    │      Нет      │                       │
    └───────┘       │                       │
                    ▼                       ▼
             ┌───────────────┐       ┌───────────────┐
             │  STOP_CMD     │◄──────┤  STOP_CMD     │
             │  STOP_WAIT    │       │  (с ошибкой)  │
             └──────┬────────┘       └───────────────┘
                    ▼
             ┌───────────────┐
             │     DONE      │ done_o = 1, error_o = nack_flag
             └───────────────┘
```

### 3.3. Протокол рукопожатия с ядром

Для каждой I2C-команды burst_writer выполняет двухфазное рукопожатие:

```
Тактовые циклы:  1    2    3    4    5    6    ...   K   K+1  K+2  ...  M   M+1
                 │    │    │    │    │    │         │    │    │        │    │
cmd_valid_o:   ──┘‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾└────────────────────────
                      ↑ Ассертим                    ↑ Деассертим
ready_i:       ‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾└────...────────┘‾‾‾‾‾
                                                     ↑ Ядро взяло    ↑ Ядро закончило
state:         ────── CMD ─────────────────────────────── WAIT ────────── next
```

**Фаза CMD:** burst_writer ставит `cmd_valid_o = 1` и ждёт, пока ядро не примет команду (ready падает). Условие: `cmd_valid_o && !ready_i`.

**Фаза WAIT:** burst_writer снимает `cmd_valid_o` и ждёт, пока ядро закончит (ready поднимается).

```verilog
S_START_CMD: begin
    cmd_o <= CMD_START;
    if (ready_i)                      // Ядро готово — ассертим
        cmd_valid_o <= 1'b1;
    if (cmd_valid_o && !ready_i) begin // Ядро приняло — переходим
        cmd_valid_o <= 1'b0;
        state       <= S_START_WAIT;
    end
end
S_START_WAIT: begin
    if (ready_i)                      // Ядро закончило
        state <= S_ADDR_CMD;
end
```

**Почему проверяем `ready_i` перед ассертом?** Если ядро ещё занято предыдущей командой, ready будет 0. Мы не хотим ассертить cmd_valid пока ядро не освободится.

**Почему `cmd_valid_o && !ready_i` для детекции приёма?** Нам нужно убедиться, что (1) мы уже подали команду (`cmd_valid_o = 1`) и (2) ядро её забрало (`ready_i` упало). Без проверки `cmd_valid_o` мы могли бы ложно среагировать на ready=0 от предыдущей команды.

### 3.4. Интерфейс источника данных

Burst_writer не знает, откуда берутся данные. Он использует простой запрос-ответ:

```
burst_writer                     Источник данных
────────────                     ───────────────
data_req_o = 1  ─────────────►  "Дай следующий байт"
                ◄─────────────   data_i = 0x42
                ◄─────────────   data_valid_i = 1
din_o ← 0x42                    "Вот он"
state → DATA_CMD
```

Для **комбинаторного** источника (ROM или генератор) `data_valid_i` можно соединить напрямую с `data_req_o` — данные доступны мгновенно.

Для **регистрового** источника (FIFO) между запросом и ответом может пройти несколько тактов — burst_writer будет ждать в состоянии `S_DATA_REQ`.

### 3.5. Обработка ошибок

Burst_writer проверяет ACK после каждого переданного байта:

```verilog
S_ADDR_WAIT: begin
    if (ready_i) begin
        if (rx_ack_i) begin     // NACK — устройство не ответило
            nack_flag <= 1'b1;
            state     <= S_STOP_CMD;  // Всё равно посылаем STOP
        end else if (cnt == 0)
            state <= S_STOP_CMD;
        else
            state <= S_DATA_REQ;
    end
end
```

При NACK burst_writer:
1. Ставит флаг `nack_flag`
2. Отправляет STOP (чтобы корректно освободить шину)
3. Сигнализирует `error_o = 1` в состоянии DONE

При потере арбитража — немедленный выход:

```verilog
if (arb_lost_i && busy_o) begin
    cmd_valid_o <= 1'b0;
    nack_flag   <= 1'b1;
    state       <= S_DONE;
end
```

## Часть 4. Модуль ssd1306_ctrl — верхнеуровневый контроллер

### 4.1. Фазовый автомат

Этот модуль управляет всей последовательностью операций:

```
             ┌──────┐  start_i    ┌───────┐
             │ IDLE ├────────────►│ DELAY │ 100 мс
             └──┬───┘             └───┬───┘
                │                     ▼
                │              ┌──────────┐
                │              │  INIT    │ bw_start = 1, count = 32
                │              └────┬─────┘
                │                   ▼
                │              ┌──────────┐
                │              │ INIT_W   │ Ждём bw_done
                │              └────┬─────┘
                │                   │
                │            ┌──────┴──────┐
                │            │             │
                │          error?        ok
                │            ▼             ▼
                │        ┌───────┐  ┌──────────┐
                │        │  ERR  │  │  DATA    │ bw_start = 1, count = 1025
                │        └───┬───┘  └────┬─────┘
                │            │           ▼
                │            │     ┌──────────┐
                │  start_i   │     │ DATA_W   │ Ждём bw_done
                │◄───────────┤     └────┬─────┘
                │            │    ┌─────┴─────┐
                │            │    │           │
                │            │  error?      ok
                │            │    ▼           ▼
                │            ├──ERR      ┌──────┐
                │            │           │  OK  │
                │            └───────────┤      │
                │◄──── start_i ──────────┘      │
                └───────────────────────────────┘
```

**Фаза DELAY:** после подачи питания SSD1306 нужно время на стабилизацию. Спецификация не требует конкретной задержки после VCC, но 100 мс — надёжный запас. При 50 МГц это 5 000 000 тактов:

```verilog
localparam [22:0] DELAY_CYCLES = (CLK_FREQ / 1000) * DELAY_MS;
// 50_000_000 / 1000 * 100 = 5_000_000 (умещается в 23 бита)

PH_DELAY: begin
    if (delay_cnt == 23'd0)
        phase <= PH_INIT;
    else
        delay_cnt <= delay_cnt - 23'd1;
end
```

**Фазы INIT и INIT_W:** запуск burst_writer для инициализации.

```verilog
PH_INIT: begin
    bw_start      <= 1'b1;           // Импульс запуска
    bw_byte_count <= {10'd0, INIT_LEN};  // 32 байта
    phase         <= PH_INITW;
end

PH_INITW: begin
    progress_o <= src_idx;           // Обновляем прогресс для дисплея
    if (bw_done) begin
        if (bw_error) begin
            err_o <= 1'b1;
            phase <= PH_ERR;
        end else
            phase <= PH_DATA;        // Инициализация OK → фреймбуфер
    end
end
```

Обратите внимание: `bw_start` — одноцикловый импульс. В PH_INIT он ставится в 1, но на следующем такте (PH_INITW) сбрасывается в 0 через `bw_start <= 1'b0` в начале always-блока.

### 4.2. Источник данных — мультиплексор

SSD1306_ctrl должен отдавать burst_writer'у правильные байты в зависимости от текущей фазы:

```
Фаза INIT_W:  init_rom(src_idx)        — байты из ROM инициализации
Фаза DATA_W:  src_idx == 0 ? 0x40      — управляющий байт «данные»
              : gen_pattern(col, page)  — байт тестовой картинки
```

Это реализуется комбинаторным мультиплексором:

```verilog
wire [7:0] init_byte = init_rom(src_idx[5:0]);
wire [7:0] data_byte = (src_idx == 11'd0) ? 8'h40
                                           : gen_pattern(pix_col, pix_page);
wire [7:0] bw_data   = (phase == PH_INITW) ? init_byte : data_byte;
```

### 4.3. Счётчик источника и адресация пикселей

Счётчик `src_idx` отслеживает, какой байт нужно отдать следующим:

```verilog
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i)
        src_idx <= 11'd0;
    else if (phase == PH_INIT || phase == PH_DATA)
        src_idx <= 11'd0;              // Сброс при входе в новую фазу
    else if (bw_data_req)
        src_idx <= src_idx + 11'd1;    // Инкремент при выдаче байта
end
```

**Преобразование индекса в координаты пикселя:**

В фазе DATA_W данные начинаются с `src_idx = 1` (src_idx = 0 — это управляющий байт 0x40). Для пикселей:

```
pixel_index = src_idx - 1      (диапазон 0..1023)
column      = pixel_index % 128 = pixel_index[6:0]
page        = pixel_index / 128 = pixel_index[9:7]
```

```verilog
wire [10:0] pix_raw  = src_idx - 11'd1;
wire [6:0]  pix_col  = pix_raw[6:0];    // 0..127
wire [2:0]  pix_page = pix_raw[9:7];    // 0..7
```

Это работает благодаря горизонтальному режиму адресации SSD1306: байты идут по столбцам слева направо, потом переходят на следующую страницу.

### 4.4. ROM инициализации

Последовательность инициализации хранится в комбинаторной функции `init_rom`:

```verilog
function [7:0] init_rom;
    input [5:0] idx;
    case (idx)
        6'd0:  init_rom = 8'h00;   // Control byte: Co=0, D/C#=0 → команды
        6'd1:  init_rom = 8'hAE;   // Display OFF
        6'd2:  init_rom = 8'hD5;   // Set clock divider
        6'd3:  init_rom = 8'h80;   //   (аргумент)
        // ... (всего 32 записи)
        6'd31: init_rom = 8'hAF;   // Display ON
        default: init_rom = 8'h00;
    endcase
endfunction
```

**Почему функция, а не `initial`-блок с массивом?** Некоторые FPGA-синтезаторы (включая Quartus для Cyclone IV) лучше оптимизируют case-функции, чем initial-блоки. Функция гарантированно синтезируется в LUT-based ROM.

**Порядок команд критичен.** Команды `0x21, 0x22` (задание диапазона адресов) идут перед `0xAF` (Display ON). Без них указатель может быть в произвольной позиции, и фреймбуфер будет записан не с начала.

## Часть 5. Генерация тестовой картинки

### 5.1. Зачем процедурная генерация

Вместо хранения 1024 байт в ROM мы вычисляем каждый байт «на лету» из координат (column, page). Преимущества:
- Ноль блочной RAM
- Легко менять паттерн
- Занимает ~50 LUT вместо 8 кбит ROM

### 5.2. Дизайн паттерна

Экран делится на 4 квадранта с рамкой по периметру:

```
 col: 0   1          63  64         126 127
      ┌───┬───────────┬───┬───────────┬───┐
pg 0: │ ▓ │ ........ │   │ ▓▓▓▓      │ ▓ │  ← верхняя рамка (bit 0)
pg 1: │ ▓ │ ████     │   │    ▓▓▓▓   │ ▓ │
pg 2: │ ▓ │     ████ │   │ ▓▓▓▓      │ ▓ │  Верт.полосы │ Шахматка
pg 3: │ ▓ │ ████     │   │    ▓▓▓▓   │ ▓ │
      ├───┤──────────┤───┤───────────┤───┤
pg 4: │ ▓ │ ░░░░░░░░ │   │ ▓▓▒▒░░   │ ▓ │
pg 5: │ ▓ │ ░░░░░░░░ │   │ ▓▓▒▒░░   │ ▓ │  Пунктир  │ Диагональ
pg 6: │ ▓ │ ░░░░░░░░ │   │ ▓▓▒▒░░   │ ▓ │
pg 7: │ ▓ │ ........ │   │ ........ │ ▓ │  ← нижняя рамка (bit 7)
      └───┴───────────┴───┴───────────┴───┘
```

### 5.3. Реализация — условия по приоритету

Функция `gen_pattern` использует каскад if-else, проверяя координаты от краёв к центру:

```verilog
function [7:0] gen_pattern;
    input [6:0] col;
    input [2:0] page;
    begin
        // 1. Левая и правая вертикальные рамки — все 8 бит = 1
        if (col == 7'd0 || col == 7'd127)
            gen_pattern = 8'hFF;

        // 2. Верхняя горизонтальная рамка — только bit 0 (верхний пиксель)
        else if (page == 3'd0)
            gen_pattern = 8'h01;

        // 3. Нижняя горизонтальная рамка — только bit 7 (нижний пиксель)
        else if (page == 3'd7)
            gen_pattern = 8'h80;

        // 4. Верхний левый квадрант: вертикальные полосы шириной 8 пикселей
        else if (page < 3'd4 && col < 7'd64)
            gen_pattern = col[3] ? 8'hFF : 8'h00;

        // 5. Верхний правый квадрант: шахматная доска 8×8
        else if (page < 3'd4)
            gen_pattern = (col[3] ^ page[0]) ? 8'hFF : 8'h00;

        // 6. Нижний левый квадрант: горизонтальный пунктир
        else if (col < 7'd64)
            gen_pattern = 8'h55;

        // 7. Нижний правый квадрант: диагональная лесенка
        else
            gen_pattern = 8'hFF >> col[2:0];
    end
endfunction
```

Разберём каждый паттерн подробно:

**Рамка (условия 1-3):**
- Столбцы 0 и 127: `8'hFF` = все 8 бит = вертикальная линия шириной 1 пиксель
- Страница 0: `8'h01` = `00000001` = только верхний бит (строка 0)
- Страница 7: `8'h80` = `10000000` = только нижний бит (строка 63)

**Вертикальные полосы (условие 4):** `col[3]` переключается каждые 8 столбцов (0-7 → 0, 8-15 → 1, 16-23 → 0, ...). Результат: чередование чёрных и белых полос шириной 8 пикселей.

**Шахматная доска (условие 5):** `col[3] ^ page[0]` — XOR бита столбца и бита страницы. Когда оба одинаковы — чёрный блок, когда разные — белый. Размер клетки: 8 столбцов × 8 строк (1 страница).

**Горизонтальный пунктир (условие 6):** `8'h55` = `01010101` — чередование горящих и потухших пикселей по вертикали в каждом столбце. На экране это выглядит как горизонтальные пунктирные линии.

**Диагональная лесенка (условие 7):**

```
8'hFF >> col[2:0]

col[2:0] = 0: 11111111 (0xFF) — все 8 пикселей
col[2:0] = 1: 01111111 (0x7F) — верхний погашен
col[2:0] = 2: 00111111 (0x3F) — верхние 2 погашены
col[2:0] = 3: 00011111 (0x1F) — верхние 3 погашены
col[2:0] = 4: 00001111 (0x0F) — верхняя половина погашена
col[2:0] = 5: 00000111 (0x07)
col[2:0] = 6: 00000011 (0x03)
col[2:0] = 7: 00000001 (0x01) — только нижний пиксель

Повторяется каждые 8 столбцов → диагональная "лесенка"
```

Визуально это создаёт наклонные линии, спускающиеся слева направо и повторяющиеся каждые 8 пикселей — характерный паттерн «тестовой карточки».

## Часть 6. Top-level — сборка проекта

### 6.1. Прескалер I2C

Ядро `i2c_master_core` работает на основном клоке (50 МГц) и продвигается по фазам при `ena_i = 1`. Частота SCL определяется частотой импульсов ena:

```
SCL_freq = clk_freq / (4 × (PRE_TOP + 1))
100 кГц  = 50 МГц  / (4 × 125)
PRE_TOP  = 124
```

Прескалер — простой счётчик:

```verilog
localparam [6:0] PRE_TOP = 7'd124;

reg [6:0] pre_cnt;
reg       core_ena;

always @(posedge clk_50m or negedge rst_n) begin
    if (!rst_n) begin
        pre_cnt  <= 7'd0;
        core_ena <= 1'b0;
    end else begin
        if (pre_cnt == PRE_TOP) begin
            pre_cnt  <= 7'd0;
            core_ena <= 1'b1;      // Один такт из 125
        end else begin
            pre_cnt  <= pre_cnt + 7'd1;
            core_ena <= 1'b0;
        end
    end
end
```

**Почему 4 × (PRE_TOP + 1)?** Потому что один I2C-бит = 4 фазы по 1 ena-тику = 4 ena-тика. Каждый ena-тик — это 1/(PRE_TOP+1) от clk_freq. Итого: один бит = 4 × (PRE_TOP+1) тактов clk = один период SCL.

### 6.2. Тристейтные буферы и синхронизаторы

I2C — шина с открытым коллектором. FPGA-пин работает как тристейт:

```verilog
// Тристейтные буферы:
// oen = 1 → high-Z (линия отпущена, pull-up подтягивает к VCC)
// oen = 0 → линия притянута к GND
assign i2c_sda = sda_oen ? 1'bz : 1'b0;
assign i2c_scl = scl_oen ? 1'bz : 1'b0;

// Обратное чтение состояния линий:
assign sda_pad_in = i2c_sda;
assign scl_pad_in = i2c_scl;
```

**2-ступенчатые синхронизаторы** предотвращают метастабильность при чтении асинхронных сигналов SDA/SCL:

```verilog
reg [1:0] sda_sync, scl_sync;
always @(posedge clk_50m or negedge rst_n) begin
    if (!rst_n) begin
        sda_sync <= 2'b11;    // По умолчанию — линии отпущены
        scl_sync <= 2'b11;
    end else begin
        sda_sync <= {sda_sync[0], sda_pad_in};  // Сдвиговый регистр
        scl_sync <= {scl_sync[0], scl_pad_in};
    end
end
wire sda_s = sda_sync[1];   // Синхронизированный SDA (задержка 2 такта)
wire scl_s = scl_sync[1];
```

Зачем 2 ступени? Один триггер может «подвиснуть» в метастабильном состоянии, если входной сигнал меняется слишком близко к фронту клока. Два последовательных триггера снижают вероятность метастабильности до пренебрежимо малой величины.

### 6.3. Антидребезг кнопки

Механическая кнопка при нажатии генерирует серию быстрых переключений (дребезг) в течение 5-20 мс. Без фильтрации одно нажатие может вызвать десятки ложных срабатываний.

```
Реальный сигнал кнопки:
        ─────┐ ┌┐┌┐ ┌──┐  ┌────────────────────────
             └─┘└┘└┘ └──┘  └
             ←── дребезг ──→
             ←── ~10 мс ──→

После антидребезга:
        ─────────────────────────────┐
                                     └──────────────
                                     ↑
                                  key_pulse (1 такт)
```

Алгоритм:
1. Фиксируем текущее состояние кнопки `key_d`
2. Если `key_d` отличается от стабильного `key_stable`, запускаем счётчик
3. Если отличие держится 20 мс (1 000 000 тактов) — принимаем новое состояние
4. При переходе 1→0 (нажатие, active-low) генерируем одноцикловый импульс

```verilog
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i) begin
        cnt <= 0;  key_d <= 1;  key_stable <= 1;  key_pulse_o <= 0;
    end else begin
        key_pulse_o <= 1'b0;          // Импульс по умолчанию = 0
        key_d       <= key_i;         // Семплируем вход

        if (key_d != key_stable) begin
            if (cnt >= CNT_MAX - 1) begin   // Отличие держится достаточно
                cnt        <= 0;
                key_stable <= key_d;        // Принимаем новое состояние
                if (!key_d)                 // Кнопка нажата (active-low)
                    key_pulse_o <= 1'b1;    // Одноцикловый импульс
            end else
                cnt <= cnt + 1;
        end else
            cnt <= 0;                       // Совпало — сброс счётчика
    end
end
```

### 6.4. Индикация на светодиодах

```verilog
assign led[0] = ssd_busy;        // Горит во время передачи
assign led[1] = ssd_done_latch;  // Горит после успешного завершения
assign led[2] = ssd_err;         // Горит при ошибке
assign led[3] = busy;            // Горит пока шина I2C занята
```

`ssd_done_latch` — защёлка, потому что `ssd_done` — это одноцикловый импульс (невозможно увидеть глазом на светодиоде):

```verilog
reg ssd_done_latch;
always @(posedge clk_50m or negedge rst_n) begin
    if (!rst_n)          ssd_done_latch <= 1'b0;
    else if (ssd_done)   ssd_done_latch <= 1'b1;  // Защёлкнуть при успехе
    else if (key_pulse)  ssd_done_latch <= 1'b0;  // Сбросить при новом нажатии
end
```

### 6.5. 7-сегментный дисплей — отображение прогресса

На 6-разрядном дисплее отображается HEX-значение счётчика переданных байт:

```
Разряд:   5   4   3   2   1   0
          ___  _  ___  ___  ___  ___
         |   || ||   ||   ||   ||   |
          ___  _  ___  ___  ___  ___

          пусто    hex[10:8]  hex[7:4]  hex[3:0]  статус
```

Разряд 0 показывает статус: `-` (в процессе), `0` (OK), `E` (ошибка).

Преобразование 4-битного значения в 7-сегментный код (active-low):

```verilog
function [7:0] seg_hex;
    input [3:0] val;
    case (val)
        4'h0: seg_hex = 8'hC0;   //  _
        4'h1: seg_hex = 8'hF9;   // | |    Формат: dp g f e d c b a
        4'h2: seg_hex = 8'hA4;   // |_|    0 = сегмент горит
        // ...                             1 = сегмент не горит
        4'hF: seg_hex = 8'h8E;
    endcase
endfunction
```

## Часть 7. Тайминг — сколько времени занимает передача

### 7.1. Расчёт для одного байта

```
1 I2C бит = 4 ena-тика = 4 × 125 = 500 тактов clk = 10 мкс при 100 кГц
1 I2C байт = 9 бит (8 data + 1 ACK) = 90 мкс
START / STOP = 4 ena-тика = 10 мкс каждый
```

### 7.2. Расчёт для init-транзакции

```
START:     1 × 10 мкс  =   10 мкс
Адрес:     1 × 90 мкс  =   90 мкс
Данные:   32 × 90 мкс  = 2880 мкс
STOP:      1 × 10 мкс  =   10 мкс
─────────────────────────────────
Итого:                  ≈ 3.0 мс
```

### 7.3. Расчёт для data-транзакции

```
START:       1 × 10 мкс  =     10 мкс
Адрес:       1 × 90 мкс  =     90 мкс
Данные:   1025 × 90 мкс  = 92 250 мкс
STOP:        1 × 10 мкс  =     10 мкс
──────────────────────────────────────
Итого:                    ≈ 92.4 мс
```

### 7.4. Полное время

```
Задержка:    100 мс
Init:          3 мс
Data:         92 мс
──────────────────
Итого:      ≈ 195 мс (менее 0.2 секунды)
```

На 7-сегментном дисплее будет видно, как счётчик стремительно нарастает от 000 до 401 (hex). Передача идёт непрерывно — burst_writer не делает пауз между байтами.

### 7.5. При 400 кГц (Fast Mode)

Если заменить `PRE_TOP = 124` на `PRE_TOP = 30` (125/4 ≈ 31, ближайшее целое):

```
SCL = 50 МГц / (4 × 31) ≈ 403 кГц

Полное время:
  Задержка: 100 мс
  Init:     ~0.75 мс
  Data:     ~23 мс
  ────────────────────
  Итого:    ≈ 124 мс
```

SSD1306 поддерживает Fast Mode (400 кГц), так что это возможно без изменения кода — только параметр прескалера.

## Часть 8. Совместная работа на одной шине

### 8.1. SSD1306 + EEPROM 24LC04

На плате AX301 к I2C-линиям (D1, E6) уже подключена EEPROM 24LC04 с адресом 0x50. SSD1306 имеет адрес 0x3C. Конфликта нет:

```
I2C Bus (SDA/SCL):
    │
    ├── EEPROM 24LC04  (адрес 0x50)
    │     ← Не реагирует на 0x3C
    │
    └── SSD1306 OLED   (адрес 0x3C)
          ← Отвечает ACK на 0x3C
```

Когда мастер отправляет адрес 0x78 (= 0x3C << 1), EEPROM видит этот адрес, сравнивает со своим (0x50), видит несовпадение и игнорирует всю транзакцию до STOP. SSD1306 видит свой адрес и отвечает ACK.

**Единственное ограничение:** нельзя одновременно обращаться к обоим устройствам (I2C — общая шина). Но в нашем проекте это не проблема — мы работаем только с SSD1306.

## Часть 9. Обработка ошибок и отладка

### 9.1. Типичные проблемы

**Проблема 1: LED[2] горит (ошибка) сразу после нажатия кнопки.**

Причина: SSD1306 не отвечает ACK на адрес. Проверьте:
- Правильность I2C-адреса (0x3C или 0x3D — зависит от модуля)
- Подключение SDA/SCL (не перепутаны ли)
- Наличие питания на SSD1306
- Pull-up резисторы

**Проблема 2: Экран SSD1306 остаётся тёмным, но LED[1] горит (успех).**

Причина: данные передаются, но дисплей не инициализирован правильно:
- Проверьте charge pump (0x8D, 0x14) — без него экран не светится
- Проверьте multiplex ratio (0xA8, 0x3F для 128×64, 0x1F для 128×32)
- Убедитесь, что финальная команда 0xAF (Display ON) присутствует

**Проблема 3: Изображение перевёрнуто или зеркально.**

Причина: команды зеркалирования не соответствуют конкретному модулю:
- `0xA1` / `0xA0` — переключение направления столбцов
- `0xC8` / `0xC0` — переключение направления строк
- Попробуйте разные комбинации

**Проблема 4: Изображение смещено или обрезано.**

Причина: режим адресации или диапазон адресов настроен неправильно:
- Убедитесь, что `0x20, 0x00` (горизонтальный режим) присутствует
- Проверьте `0x21, 0x00, 0x7F` (полный диапазон столбцов)
- Проверьте `0x22, 0x00, 0x07` (полный диапазон страниц)

### 9.2. Как использовать прогресс на 7-сегментном дисплее

Во время передачи дисплей показывает количество переданных байт:

```
Фаза инициализации:
  000 → 001 → 002 → ... → 020 (hex, = 32 dec)

Фаза данных:
  000 → 001 → ... → 401 (hex, = 1025 dec)
```

Если счётчик остановился на определённом значении, это означает, что SSD1306 перестал отвечать ACK на этом байте. Если остановился на 000 в фазе данных — проблема с первым байтом данных (управляющий 0x40). Если остановился на 001 — с первым пикселем.

## Часть 10. Пошаговый рецепт проектирования

Если бы вы создавали подобный проект с нуля для другого I2C-устройства:

### Шаг 1. Изучите устройство

Прочитайте даташит целевого устройства. Определите:
- I2C-адрес
- Формат команд (есть ли управляющий байт, как в SSD1306?)
- Последовательность инициализации
- Формат данных и их объём

### Шаг 2. Определите объём передачи

Если данных мало (< 16 байт) — можно обойтись простым FSM, как в проекте EEPROM-теста.
Если данных много (сотни-тысячи байт) — используйте `i2c_burst_writer`.

### Шаг 3. Реализуйте источник данных

Варианты:
- **ROM** (case-функция) — для фиксированных данных (инициализация)
- **Процедурный генератор** — для регулярных паттернов (тестовые картинки)
- **FIFO** — для данных от внешнего источника (UART, SPI, память)
- **Block RAM** — для произвольных изображений (.mif/.hex файл)

### Шаг 4. Соберите конвейер

```
Источник данных → burst_writer → i2c_master_core → I2C шина → Устройство
```

### Шаг 5. Добавьте управление и индикацию

- Кнопка запуска (с антидребезгом)
- Светодиоды (busy/done/error)
- 7-сегментный дисплей или UART для подробного статуса

### Шаг 6. Рассчитайте тайминг

Определите:
- Частоту SCL (100/400 кГц)
- Общее время передачи
- Подходит ли скорость для вашего сценария

### Шаг 7. Протестируйте

1. Verilator lint — проверка синтаксиса и предупреждений
2. iverilog — компиляция
3. Симуляция — если есть модель устройства
4. Железо — загрузка в FPGA и наблюдение по LED/7-сегменту
