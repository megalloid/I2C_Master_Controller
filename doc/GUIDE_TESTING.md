# Тестирование I2C Master Core — подробное руководство

## Введение: зачем тестировать и что такое тестбенч

### Что вообще значит «тестировать» цифровую схему?

Когда мы пишем Verilog-модуль, мы описываем поведение аппаратной логики — набор регистров и проводов, которые переключаются по фронтам тактового сигнала. Но до загрузки в FPGA или производства микросхемы у нас нет физической платы, осциллографа и реальных устройств на шине. Поэтому мы **моделируем** всё окружение программно — в так называемом **тестбенче** (testbench, сокращённо TB).

Тестбенч — это Verilog-модуль (обычно на SystemVerilog), который:
1. **Создаёт** тактовый сигнал и сброс
2. **Подключает** к тестируемому модулю (DUT — Design Under Test) модели окружающих устройств
3. **Посылает** последовательности команд и данных
4. **Проверяет**, что выходы DUT соответствуют ожиданиям

Результат — вердикт `PASS` или `FAIL` для каждого сценария, плюс файл осциллограмм (VCD), который можно открыть в GTKWave и увидеть каждый сигнал по тактам.

### Наше тестовое окружение

Участники:

| Компонент | Что это | Файл | Аналогия из реального мира |
|-----------|---------|------|---------------------------|
| **DUT** | Тестируемый модуль — наш I2C контроллер | `rtl/i2c_master_top.v` | Микросхема на плате |
| **AXI BFM** | Эмулятор процессора: записывает и читает регистры через AXI4-Lite | `tb/axi_lite_master_bfm.sv` | Процессор ARM, который общается с контроллером |
| **I2C Slave** | Модель устройства на I2C шине (ведёт себя как EEPROM с 256 байтами памяти) | `tb/i2c_slave_model.sv` | Чип EEPROM 24C02 на плате |
| **Pull-up** | Подтяжка линий SDA/SCL к VCC (как в реальной схеме) | `pullup` в тестбенче | Резисторы 4.7кОм на плате |
| **Сценарии** | Последовательность действий, записанная в блоке `initial` | `tb/i2c_master_tb.sv` | Ваша программа на Linux, которая дёргает i2c-tools |

### Как работает взаимодействие

Представьте, что вы сидите за Linux-терминалом и выполняете:

```bash
i2cset -y 0 0x50 0x10 0xA5    # записать байт 0xA5 в EEPROM по адресу 0x10
i2cget -y 0 0x50 0x10          # прочитать обратно
```

В реальной системе Linux-драйвер записывает в регистры I2C-контроллера, контроллер генерирует сигналы на шине, EEPROM отвечает.

В нашем тестбенче то же самое, но вместо Linux-драйвера — AXI BFM, вместо контроллера — DUT, вместо EEPROM — slave-модель:

```
Реальный мир:           Linux → драйвер → регистры → I2C шина → EEPROM
Тестбенч:               Сценарий → AXI BFM → регистры → I2C шина → Slave-модель
```

### Как передать одну команду через тестбенч

Чтобы записать байт на I2C шину, нужно выполнить несколько шагов через AXI-регистры:

```
Шаг 1. Записать адрес slave в TX_DATA:    AXI_WRITE(0x0C, 0xA0)   // 0x50 << 1 | 0 = 0xA0
Шаг 2. Дать команду START + WRITE:         AXI_WRITE(0x08, 0x09)   // STA=1, WR=1
Шаг 3. Подождать, пока TIP сбросится:     Опрашивать STATUS[0] пока не станет 0
Шаг 4. Проверить ACK:                      Прочитать STATUS[1] — если 0, slave ответил
Шаг 5. Записать данные в TX_DATA:          AXI_WRITE(0x0C, 0xA5)
Шаг 6. Дать команду WRITE + STOP:          AXI_WRITE(0x08, 0x0A)   // WR=1, STO=1
Шаг 7. Подождать TIP=0
```

Каждый `AXI_WRITE` — это вызов task-а `bfm.axi_write(addr, data)` в тестбенче. Каждый «подождать» — это цикл с опросом. Всё это моделируется такт за тактом в симуляторе.

## Как устроены проверки: PASS и FAIL

В тестбенче есть два счётчика: `test_pass` и `test_fail`. После каждого сценария проверяется результат:

```verilog
if (read_val === 8'hA5) begin
    $display("  PASS: read 0x%02h == expected 0xA5", read_val);
    test_pass = test_pass + 1;
end else begin
    $error("  FAIL: read 0x%02h != expected 0xA5", read_val);
    test_fail = test_fail + 1;
end
```

`$display` — печать в консоль (информация). `$error` — печать с пометкой «ошибка». В конце симуляции — итог:

```
  TEST SUMMARY:  PASS=10  FAIL=0
All tests PASSED
```

Если хотя бы один тест провалился, симуляция завершается с `$fatal` — это аналог `exit(1)` в программировании. CI-системы (Jenkins, GitLab CI) увидят ненулевой код возврата и пометят сборку как «красную».

## Сценарий 1. Одиночный WRITE + ACK

### Что проверяем

Самая базовая операция: мастер отправляет один байт данных slave-устройству, и slave подтверждает приём (ACK).

### Зачем это нужно

Если это не работает — не работает ничего. Это «hello world» для I2C контроллера.

### Что происходит на шине

```
  SDA: ──╲    ┌─A6─┐┌─A5─┐┌─A4─┐┌─A3─┐┌─A2─┐┌─A1─┐┌─A0─┐┌─W──┐┌ACK┐
          ╲───┤    ├┤    ├┤    ├┤    ├┤    ├┤    ├┤    ├┤    ├┤   ├─...
  SCL: ──────╲┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐  ┌┐┌┐
             ↑                                                   ↑
          START                                              Slave тянет
                                                             SDA=0 (ACK)
```

1. Мастер генерирует **START** (SDA падает при SCL=1)
2. Мастер передаёт 7 бит адреса slave + бит направления (W=0)
3. Slave узнаёт свой адрес и отвечает **ACK** — притягивает SDA к 0 на 9-м бите
4. Мастер передаёт 8 бит данных (адрес регистра внутри EEPROM)
5. Slave отвечает ACK
6. Мастер передаёт 8 бит данных (значение для записи)
7. Slave отвечает ACK
8. Мастер генерирует **STOP** (SDA поднимается при SCL=1)

### Как это выглядит в тестбенче

```verilog
// Записать 0xA5 в ячейку 0x10 slave-а 0x50
i2c_write_byte(7'h50, 8'h10, 8'hA5);
```

Внутри task-а `i2c_write_byte` происходит:

```
AXI_WRITE(TX_DATA, 0xA0)     ← 0x50 << 1 | 0 = адрес slave с битом «запись»
AXI_WRITE(CMD, STA | WR)     ← START + WRITE
wait_tip_clear()              ← ждём завершения на I2C
проверить STATUS.RXACK == 0  ← slave ответил ACK?
AXI_WRITE(TX_DATA, 0x10)     ← адрес ячейки
AXI_WRITE(CMD, WR)           ← WRITE
wait_tip_clear()
AXI_WRITE(TX_DATA, 0xA5)     ← данные
AXI_WRITE(CMD, WR | STO)     ← WRITE + STOP
wait_tip_clear()
```

### Что может пойти не так

| Симптом | Возможная причина |
|---------|-------------------|
| NACK вместо ACK | Slave-модель не распознала адрес (ошибка в формировании адресного байта) |
| Тайм-аут (TIP не сбрасывается) | FSM ядра «застрял» — ошибка в переходах состояний |
| Данные записались неверно | Сдвиговый регистр или порядок бит (MSB/LSB) неправильный |

### Критерий PASS

После записи делаем обратное чтение. Если прочитанный байт совпадает с записанным — тест пройден.

## Сценарий 2. Одиночный READ + NACK

### Что проверяем

Мастер читает один байт из slave-а и посылает NACK в конце — сигнал «больше данных не нужно».

### Зачем это нужно

READ — вторая фундаментальная операция. А NACK от мастера после READ — обязательная часть протокола при чтении последнего (или единственного) байта. Если мастер пошлёт ACK, slave будет ожидать чтение следующего байта и не отпустит шину.

### Что происходит на шине

```
     Фаза записи адреса      RESTART      Фаза чтения
     (куда читать)                          (slave отдаёт данные)
  ┌────────────────────┐    ┌──┐    ┌────────────────────────┐
  │ S ADDR+W REG_ADDR  │    │Sr│    │ ADDR+R  D7..D0  NACK P │
  └────────────────────┘    └──┘    └────────────────────────┘
                                              ↑          ↑
                                        Slave гонит   Мастер
                                        данные        посылает
                                                      NACK
```

Чтение из EEPROM — составная операция:
1. **Фаза записи:** START → адрес slave (W) → адрес ячейки → (slave ACK)
2. **Repeated START** (не STOP!) — потому что мы переключаемся с записи на чтение
3. **Фаза чтения:** адрес slave (R) → slave выдаёт 8 бит данных → мастер отвечает **NACK** → STOP

### Как это выглядит в тестбенче

```verilog
i2c_read_byte(7'h50, 8'h10, read_val);
```

Внутри task-а:

```
AXI_WRITE(TX_DATA, 0xA0)     ← адрес slave + W
AXI_WRITE(CMD, STA | WR)     ← START + WRITE
wait_tip_clear()
AXI_WRITE(TX_DATA, 0x10)     ← адрес ячейки
AXI_WRITE(CMD, WR)           ← WRITE
wait_tip_clear()
AXI_WRITE(TX_DATA, 0xA1)     ← адрес slave + R  (0x50 << 1 | 1)
AXI_WRITE(CMD, STA | WR)     ← RESTART + WRITE (адрес с новым направлением)
wait_tip_clear()
AXI_WRITE(CMD, RD | NACK | STO) ← READ + NACK + STOP
wait_tip_clear()
AXI_READ(RX_DATA, read_val)  ← забрать принятый байт
```

### Ключевой момент: NACK от мастера

Бит `CMD.NACK = 1` говорит ядру: «после приёма байта пошли NACK, а не ACK». В `i2c_master_core` это работает через 9-й бит сдвигового регистра:

```verilog
// При команде READ:
tx_shift_r <= {8'hFF, din_i[0]};   // din_i[0] = 1 → NACK
```

8'hFF — отпустить SDA на 8 бит данных (slave управляет). `din_i[0] = 1` — в 9-м бит-слоте мастер выдаёт «1» (NACK).

### Критерий PASS

Прочитанный байт совпадает с ранее записанным значением.

## Сценарий 3. Полная транзакция: START + WRITE addr + WRITE data + STOP

### Что проверяем

Полный цикл жизни I2C-транзакции записи — от начала до конца, включая все условия на шине.

### Зачем это нужно

Сценарии 1 и 2 используют высокоуровневые task-и (`i2c_write_byte`, `i2c_read_byte`). Здесь мы проверяем, что **низкоуровневая** последовательность команд работает корректно — что каждый переход FSM между START, DATA и STOP происходит правильно.

### Что мы на самом деле тестируем

Этот сценарий проверяет три перехода FSM ядра:

```
IDLE → ST_START → IDLE → ST_DATA (9 бит × 4 фазы) → IDLE → ST_DATA → IDLE → ST_STOP → IDLE
```

Между каждой командой ядро возвращается в `ST_IDLE`. Критично, что в этих промежуточных IDLE ядро **не отпускает линии** (шина занята — `busy_o = 1`).

### Что происходит на шине (побитово)

```
Время →

SDA: ─╲ ┌A6┐┌A5┐┌A4┐┌A3┐┌A2┐┌A1┐┌A0┐┌W─┐┌AK┐ ┌D7┐┌D6┐┌D5┐┌D4┐┌D3┐┌D2┐┌D1┐┌D0┐┌AK┐ ──╱─
       ╲─┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├─┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├───╱
SCL: ────╲┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐───────────
      START   Адрес slave + W    ACK    8 бит данных        ACK         STOP
```

### Как верифицировать в осциллограмме

Откройте VCD-файл в GTKWave и проверьте:

1. **START:** SDA падает при SCL=HIGH → `busy_o` устанавливается в 1
2. **Адресный байт:** 8 импульсов SCL, SDA меняется только при SCL=LOW
3. **ACK:** на 9-м SCL-импульсе slave притягивает SDA к 0
4. **Байт данных:** ещё 8+1 SCL-импульсов
5. **STOP:** SDA поднимается при SCL=HIGH → `busy_o` сбрасывается в 0

### Типичная ошибка, которую ловит этот тест

Если ядро неправильно управляет SDA в промежуточном IDLE (между WRITE addr и WRITE data), slave увидит ложный STOP и прервёт транзакцию. Тест обнаружит это через NACK на втором байте.

## Сценарий 4. Repeated START (RESTART)

### Что проверяем

Генерацию повторного START без предварительного STOP. Это нужно для переключения направления (с записи на чтение) в одной транзакции.

### Зачем это нужно

Множество I2C-устройств требуют RESTART. Самый распространённый пример — чтение из EEPROM:

```
Запись:   S  ADDR+W  REG  Sr  ADDR+R  DATA  NACK  P
          ↑                ↑                       ↑
        START          RESTART                   STOP
                   (не STOP+START!)
```

Между «отправить адрес ячейки» и «прочитать данные» нужен **RESTART**, а не STOP+START. Почему?

1. **STOP освобождает шину.** В multi-master конфигурации другой мастер может захватить шину между STOP и START, и наша составная операция прервётся.
2. **Многие slave сбрасывают состояние по STOP.** Если мы сделаем STOP после записи адреса ячейки, EEPROM может «забыть», какую ячейку мы выбрали.

### Что происходит на шине

```
         Обычный START              RESTART
         (шина была свободна)       (шина занята, SCL=LOW)

SDA: ─────╲                   ────╱────╲
           ╲──────             ╱        ╲──────
SCL: ─────────╲            ──╱──────╲
              ╲───         ╱        ╲───
      SCL=1   SDA=0     Сначала    Потом
      SDA=1→0          SDA→1       SDA→0
                       при SCL=0   при SCL=1
```

Ключевое отличие RESTART от START: перед RESTART SCL удерживается мастером в LOW (мы посередине транзакции). Поэтому нужна другая последовательность:

| Фаза | START | RESTART |
|------|-------|---------|
| 0 | SDA=1, SCL=1, ждём SCL | SDA→1, SCL=0 (сначала отпускаем SDA) |
| 1 | hold | SCL→1 (потом отпускаем SCL) |
| 2 | SDA→0 (START!) | SDA→0 (START!) |
| 3 | SCL→0 | SCL→0 |

### Как это выглядит в тестбенче

RESTART генерируется автоматически при повторной отправке `CMD_STA | CMD_WR` во время активной транзакции:

```verilog
// Фаза записи
AXI_WRITE(TX_DATA, 0xA0);      // Адрес + W
AXI_WRITE(CMD, STA | WR);      // START + WRITE
wait_tip_clear();
AXI_WRITE(TX_DATA, 0x10);      // Адрес ячейки
AXI_WRITE(CMD, WR);            // WRITE
wait_tip_clear();

// Фаза чтения (RESTART!)
AXI_WRITE(TX_DATA, 0xA1);      // Адрес + R
AXI_WRITE(CMD, STA | WR);      // ← секвенсер видит STA при busy=1 → генерирует RESTART
wait_tip_clear();
AXI_WRITE(CMD, RD | NACK | STO);
wait_tip_clear();
```

### Как верифицировать

В осциллограмме между двумя байтами адреса (запись и чтение) должен быть RESTART, а не STOP+START. Признаки:

- `busy_o` **не сбрасывается** между двумя START-условиями
- На шине нет промежутка с обеими линиями HIGH (что было бы при STOP)
- SCL переходит из LOW в HIGH, затем SDA падает — это RESTART

### Типичная ошибка

Если вместо RESTART генерируется STOP+START, slave увидит два отдельных обращения. EEPROM «забудет» выбранную ячейку и начнёт чтение с адреса 0 (или вернёт NACK).

## Сценарий 5. NACK от slave (устройство не отвечает)

### Что проверяем

Поведение контроллера, когда на шине нет устройства с указанным адресом (или устройство отказывает).

### Зачем это нужно

В реальной жизни NACK — обычная ситуация:
- Указан неверный адрес
- Устройство занято (EEPROM выполняет page write ≈5 мс)
- Устройство физически отсутствует или повреждено
- Устройство не готово к приёму данных

Контроллер **не должен** зависнуть или сломаться при NACK. Он должен корректно сообщить об ошибке.

### Что происходит на шине

```
  SDA: ──╲ ┌A6┐┌A5┐┌A4┐┌A3┐┌A2┐┌A1┐┌A0┐┌W─┐┌───┐ ──╱─
          ╲─┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤  ├┤НАК├───╱
  SCL: ────╲┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐─────────
       START    Адрес (неправильный!)   NACK    STOP
                                       SDA=1
                                       (никто
                                       не тянет)
```

Slave не узнал «свой» адрес → не притянул SDA → мастер читает SDA=1 в 9-м бит-слоте → это NACK.

### Как это выглядит в тестбенче

```verilog
$display("\n=== TEST 3: NACK on wrong slave address ===");

// Посылаем адрес 0x3F — такого slave на шине нет (есть только 0x50)
bfm.axi_write(REG_TX_DATA, {24'd0, 7'h3F, 1'b0});
bfm.axi_write(REG_CMD, CMD_STA | CMD_WR);
wait_tip_clear();

// Проверяем: STATUS.RXACK должен быть 1 (NACK)
bfm.axi_read(REG_STATUS, rd_data);
if (rd_data[1]) begin  // STATUS bit 1 = RXACK
    $display("  PASS: Got NACK for wrong address as expected");
    test_pass = test_pass + 1;
end

// Обязательно освобождаем шину!
bfm.axi_write(REG_CMD, CMD_STO);
wait_tip_clear();
```

### Важный момент: STOP после NACK

После получения NACK мастер **обязан** послать STOP для освобождения шины. Если этого не сделать:
- Шина остаётся в состоянии «busy»
- SCL удерживается в LOW
- Следующая транзакция не может начаться (ядро не примет START при busy=1 без RESTART)

В реальном коде драйвера Linux это обрабатывается автоматически. В тестбенче — нужно делать руками.

### Критерий PASS

`STATUS.RXACK == 1` после отправки неизвестного адреса. Контроллер не завис, шина корректно освобождена.

## Сценарий 6. Clock stretching (slave задерживает SCL)

### Что проверяем

Способность мастера корректно работать с медленными slave-устройствами, которые удерживают SCL в LOW.

### Зачем это нужно

Clock stretching — механизм, позволяющий slave «попросить подождать». Slave притягивает SCL к 0, не давая мастеру продолжить. Мастер обязан заметить это и остановиться.

Реальные примеры:
- EEPROM выполняет запись во Flash (5 мс) и держит SCL
- Датчик производит АЦП-преобразование
- Slave на медленном микроконтроллере обрабатывает прерывание

### Как это работает

```
Нормальная ситуация:
  SCL (мастер): ─╲    ╱───────╲    ╱────
                  ╲──╱         ╲──╱
  SCL (шина):     аналогично

Clock stretching:
  SCL (мастер отпускает): ──────── HIGH (oen=1)
  SCL (slave держит):     ──────── LOW  (притянут)
  SCL (шина, реально):    ──────── LOW! ← wired-AND: 0 побеждает 1

  Мастер проверяет scl_i... видит 0... ждёт...

  Slave отпускает:        HIGH
  SCL (шина):             HIGH ← pull-up подтянул
  Мастер видит scl_i=1:  продолжает
```

В нашем ядре clock stretching обрабатывается в **фазе 1** каждого состояния:

```verilog
// Мастер отпустил SCL (oen=1), но проверяет реальный уровень:
2'd1: begin
    scl_oen_o <= 1'b1;           // Отпускаем SCL
    if (scl_i) begin              // SCL реально HIGH?
        // Да → продолжаем
        phase_r <= 2'd2;
    end
    // Нет → стоим на месте, ждём
end
```

### Как тестировать

Для тестирования clock stretching нужна slave-модель, которая **активно задерживает SCL**. Текущая slave-модель не реализует stretching (она мгновенно отвечает), но можно добавить:

```verilog
// В slave-модели: после приёма адреса, перед отправкой ACK,
// удерживаем SCL в LOW на 100 тактов:
reg scl_stretch;
assign scl_io = scl_stretch ? 1'b0 : 1'bz;

// При переходе в S_ADDR_ACK:
scl_stretch <= 1'b1;
repeat (100) @(posedge clk);
scl_stretch <= 1'b0;
```

Без stretching-модели этот сценарий проверяется **косвенно**: если в коде ядра убрать `if (scl_i)` и заменить на безусловный переход, все остальные тесты тоже сломаются, потому что на реальной шине pull-up имеет задержку (RC-цепь), и в первые наносекунды после отпускания SCL линия ещё в неопределённом состоянии.

### Что может пойти не так без поддержки stretching

Если мастер не проверяет `scl_i` и сразу переходит к следующей фазе:
1. Мастер «думает», что SCL=HIGH, и начинает менять SDA
2. Slave всё ещё держит SCL=LOW, и изменение SDA при LOW не имеет значения... **пока slave не отпустит**
3. Когда slave отпускает SCL, на шине может оказаться совсем не то значение SDA, которое мастер хотел выставить
4. Данные повреждены

## Сценарий 7. Arbitration lost (потеря арбитража)

### Что проверяем

Поведение мастера при конфликте с другим мастером на той же шине (multi-master конфигурация).

### Зачем это нужно

I2C поддерживает несколько мастеров на одной шине. Два мастера могут одновременно начать транзакцию. Протокол гарантирует, что один из них «выиграет», а второй — корректно уступит.

### Как работает арбитраж

Арбитраж основан на свойстве open-drain: если **любое** устройство тянет линию к 0, линия = 0. Если все отпустили — линия = 1 (через pull-up).

```
Мастер 1 передаёт:  1  0  1  1  0  ...
Мастер 2 передаёт:  1  0  1  0  ← конфликт!
SDA на шине:         1  0  1  0  ← Мастер 2 «побеждает» (его 0 сильнее)

                                 ↑
                       Мастер 1 отпустил SDA (ожидает 1)
                       Но видит SDA=0 → арбитраж потерян!
                       Мастер 1 немедленно отпускает шину.
```

Побеждает тот, кто передаёт «0» — потому что 0 электрически доминирует. Проигравший обнаруживает конфликт, когда его бит = 1, а на шине = 0.

### Как это реализовано в ядре

```verilog
// В фазе 1 ST_DATA (SCL=HIGH, семплируем SDA):
wire al_data = (state_r == ST_DATA) &&
               (phase_r == 2'd1) &&      // SCL только поднялся
               scl_i &&                   // SCL реально HIGH
               !sda_input_mode &&         // Мастер управляет SDA
               sda_oen_o &&              // Мастер отпустил (ожидает 1)
               !sda_i;                   // Но SDA = 0!!! Конфликт!

// Реакция:
if (al_event) begin
    state_r   <= ST_IDLE;     // Аварийный выход
    scl_oen_o <= 1'b1;        // Отпустить всё
    sda_oen_o <= 1'b1;
    arb_lost_o <= 1'b1;       // Поднять sticky-флаг
end
```

### Как тестировать

Для полноценного теста нужны **два экземпляра** мастера на одной шине:

```
  ┌──────────┐        ┌──────────┐
  │ Master 1 ├──┬──SDA──┬──┤ Master 2 │
  │ (DUT)    ├──┼──SCL──┼──┤ (DUT 2)  │
  └──────────┘  │       │  └──────────┘
              Pull-up  Pull-up
                │       │
              ┌─┴───────┴─┐
              │  Slave     │
              └────────────┘
```

Оба мастера одновременно начинают передачу разных данных. Один из них увидит `arb_lost_o = 1` и откатится в IDLE.

В текущем тестбенче этот сценарий не реализован (один мастер), но флаг `arb_lost_o` проверяется косвенно: после каждого теста он должен быть `0`. Если бы арбитражная логика сработала ложно, он бы был `1`, и следующие команды не были бы приняты (ядро блокирует приём при `arb_lost_o = 1`).

### Что может пойти не так без обработки арбитража

1. Два мастера одновременно передают данные
2. Один отпустил SDA (бит = 1), второй тянет к 0 (бит = 0)
3. Без арбитражной логики первый мастер **не знает** о конфликте и продолжает
4. Оба мастера портят данные друг другу
5. Slave получает «мусор» — повреждение данных

## Сценарий 8. Reset во время транзакции

### Что проверяем

Способность контроллера корректно восстановиться после аппаратного сброса.

### Зачем это нужно

Сброс может произойти в любой момент:
- Watchdog timer перезагружает систему
- Пользователь нажал кнопку Reset
- Программная ошибка привела к перезагрузке CPU

Контроллер должен:
1. **Немедленно** отпустить SCL и SDA (не блокировать шину)
2. **Сбросить** все регистры в начальное состояние
3. **Быть готовым** к нормальной работе после снятия сброса

### Что происходит при сбросе

```verilog
always @(posedge clk_i or negedge rstn_i) begin
    if (!rstn_i) begin
        state_r    <= ST_IDLE;    // FSM → начальное состояние
        scl_oen_o  <= 1'b1;      // Отпустить SCL (HIGH)
        sda_oen_o  <= 1'b1;      // Отпустить SDA (HIGH)
        ready_o    <= 1'b1;      // Готов к командам
        arb_lost_o <= 1'b0;      // Нет ошибок
        busy_o     <= 1'b0;      // Шина свободна
        // ... все остальные регистры в 0
    end
```

### Как это выглядит в тестбенче

```verilog
$display("\n=== TEST 6: Reset recovery ===");

// Сброс на 10 тактов
rst_n = 0;
repeat (10) @(posedge clk);
rst_n = 1;
repeat (10) @(posedge clk);

// Переинициализация: включить ядро
bfm.axi_write(REG_CTRL, 32'h03);  // EN=1, IEN=1
repeat (5) @(posedge clk);

// Проверить, что контроллер работает
i2c_write_byte(SLAVE_ADDR, 8'h50, 8'hBB);
repeat (100) @(posedge clk);
i2c_read_byte(SLAVE_ADDR, 8'h50, read_val);

if (read_val === 8'hBB) begin
    $display("  PASS: post-reset OK");
    test_pass = test_pass + 1;
end
```

### Что может пойти не так

| Проблема | Последствие |
|----------|-------------|
| SDA/SCL не отпускаются при сбросе | Шина заблокирована, все устройства ждут бесконечно |
| FSM не сбрасывается | Ядро продолжает прерванную операцию с мусорными данными |
| Прескалер не сбрасывается | SCL-частота непредсказуема |
| `ready_o` не сбрасывается в 1 | Контроллер не принимает новые команды |
| `arb_lost_o` не сбрасывается | Ядро блокирует все команды до явного clear |

### Тонкий момент: slave после reset мастера

Когда мастер сбрасывается посередине транзакции, slave может быть в неопределённом состоянии:
- Slave ждёт следующий SCL-импульс
- Slave может держать SDA в LOW (если передавал бит «0» или ACK)

В реальной системе для восстановления используется **bus recovery**: мастер генерирует 9 импульсов SCL без START/STOP, что заставляет slave «дошагать» оставшиеся биты и отпустить SDA.

В нашем тестбенче slave-модель при обнаружении STOP (SDA→1 при SCL=1) автоматически сбрасывает своё состояние, поэтому дополнительная bus recovery не требуется.
