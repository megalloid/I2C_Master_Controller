# Создание проекта в Xilinx Vivado 2025.2 для платы ZYNQ MINI Rev B и OLED-дисплея на шине PL — пошаговый разбор

Этот гайд написан так, чтобы его понял **школьник или студент-первокурсник**, у которого есть базовые знания Verilog (модуль, `always`, разрядности). Никаких предположений «вы и так знаете, как кликнуть» здесь нет: каждое окно, каждый чек-бокс, каждое поле объясняется, и почему оно именно такое.

Источник истины по железу — файл **`datasheets/ZYNQ_MINI_REVB Schematic.pdf`**:

- лист «**第二部分**» — какой именно SoC стоит на плате;
- лист «**第五部分 ZYNQ的PL部分的BANK（FPGA部分）**» — на какие выводы FPGA выведены сигналы PL (включая OLED, кнопки, светодиоды, тактовый кварц);
- лист «**第十二部分 PL的OLED电路**» — разъём `J4 OLED12864`.

## 1. Что вообще такое Zynq и почему есть «PL» и «PS»

Чип **XC7Z010** содержит **две части в одном кристалле**:

- **PS (Processing System)** — двухъядерный ARM Cortex-A9. Это «обычный процессор», как в телефоне: на нём может работать Linux или bare-metal программа.
- **PL (Programmable Logic)** — это собственно **FPGA**, в которой мы «рисуем схему» с помощью Verilog.

Они могут работать и **по отдельности**, и **вместе**. Когда вместе — общаются по шинам **AXI**. В нашем случае:

- I2C-мастер (Verilog) живёт в **PL**.
- Управлять им можно либо «жёстко» из самого PL, либо через **AXI4-Lite** из PS (если хотите программу на ARM).

**Совет для начинающих:** для первого запуска проще выбрать **только PL** (Вариант A в разделе 5). Меньше шагов, не нужен DDR, не нужно пересобирать программу для ARM.

## 2. Куда подключается OLED — конкретные выводы FPGA

В схеме (лист 5) сигналы дисплея уже **разведены к шарикам корпуса CLG400**. Я выписал значения непосредственно из PDF:

### Вариант A — встроенный разъём `J4` (лист 12 схемы, COG SSD1306)

| Сигнал на разъёме `J4` | Шарик FPGA (`PACKAGE_PIN`) | Bank | Логическая функция (для I2C-режима SSD1306) |
|------------------------|----------------------------|------|---------------------------------------------|
| **`OLED_D0`** | **`E18`** | 35 | На большинстве модулей при перемычке в «I2C» = **SDA** (данные) |
| **`OLED_D1`** | **`E19`** | 35 | = **SCL** (тактирование шины) |
| **`OLED_DC`** | **`F16`** | 35 | Adress select / D/C; для чистого I2C SSD1306 на модуле часто фиксирован, но всё равно подведён к FPGA — управляйте как **GPIO**, выставив 0 (или согласно даташиту вашего модуля) |
| **`OLED_RST`** | **`F17`** | 35 | Аппаратный сброс дисплея — **GPIO**, держим в `1` после инициализации |

### Вариант B — внешний модуль через 40-pin GPIO-разъём `CAM1` (используется в текущем PS+PL build)

| Сигнал | Шарик FPGA | Bank | Сеть на схеме |
|--------|-----------|------|---------------|
| **SDA** | **`T20`** | 34 | `FPGA_GPIO_15P_34` (CAM1 pin 12) |
| **SCL** | **`P20`** | 34 | `FPGA_GPIO_14N_34` (CAM1 pin 18) |

Дополнительно на плате есть:

| Сигнал | `PACKAGE_PIN` | Зачем |
|--------|---------------|-------|
| **`PL_CLK_50M`** | **`K17`** (это вход MRCC bank 35) | Тактовый кварц 50 МГц — наш системный `clk` |
| **`FPGA_PL_KEY1`** | **`M20`** | Кнопка — удобно использовать как «сброс» |
| **`FPGA_PL_KEY2`** | **`M19`** | Вторая кнопка |
| **`FPGA_PL_LED1..4`** | **`T12`**, **`U12`**, **`V12`**, **`W13`** | 4 светодиода — индикация состояния автомата |

> *Bank 35* в схеме питается от **3,3 В**, поэтому стандарт ввода-вывода для всех этих портов в `XDC` будет **`LVCMOS33`**.

> ⚠️ **Перед прошивкой** ещё раз сверьте свою конкретную ревизию платы со схемой в PDF: если в магазине вам приехала более поздняя ревизия, разводка может отличаться. Источник истины — **схема в коробке**.

## 3. Запускаем Vivado и создаём пустой проект — буквально по нажатию каждой кнопки

### 3.1 Запуск среды

1. **Запуск:** в меню «Пуск» (Windows) или в терминале (Linux: `source /opt/Xilinx/Vivado/2025.2/settings64.sh && vivado &`) откройте Vivado. Появится **Welcome screen**.
2. На приветственном экране в колонке **Quick Start** нажмите **«Create Project»**. Откроется мастер из нескольких страниц.

> *Что такое «мастер» / wizard?* Это серия диалоговых окон, где вы заполняете поля и жмёте **Next**. Vivado по этим ответам создаст для вас структуру папок и базовые файлы.

### 3.2 Страница «New Project»

- Просто прочтите вступление и нажмите **Next**.

### 3.3 Страница «Project Name»

- **Project name:** введите латиницей без пробелов, например `zynq_mini_oled`. *Почему латиница?* Vivado внутри использует Tcl-скрипты, в которых пробелы и кириллица иногда ломают пути.
- **Project location:** выберите папку для проекта. Лучше короткий путь, например `D:\fpga\` или `/home/user/fpga/`. Слишком длинные пути в Windows могут привести к ошибке «filename too long».
- **Create project subdirectory:** ✅ оставьте включённой — Vivado создаст внутри `Project location` папку с именем `Project name`.
- **Next**.

### 3.4 Страница «Project Type»

- Выберите **RTL Project**. *Что это?* «Project for designing in Verilog/VHDL». Альтернативы (Post-synthesis, I/O Planning, Imported, Example) нам сейчас не нужны.
- **Do not specify sources at this time** — ✅ поставьте галочку. *Зачем?* Чтобы мастер не приставал с добавлением файлов; мы добавим их вручную позже, так нагляднее.
- **Next**.

### 3.5 Страница «Default Part»

Это самый важный экран мастера: здесь говорим Vivado, какая именно микросхема стоит на плате.

1. В левом верхнем углу выберите вкладку **«Parts»** (а не «Boards» — в каталоге Vivado этой китайской платы не будет).
2. Включите **Filter** для поиска:
   - **Family:** `Zynq-7000`.
   - **Package:** `clg400`.
   - **Speed grade:** `-1` (если на корпусе чипа стоит `-2` или `-3` — выберите ту).
3. В таблице ниже ищите строку с **`xc7z010clg400-1`** — кликните по ней.
4. **Next**, затем **Finish**. Vivado создаст проект и откроет главное окно.

> *Почему именно `xc7z010clg400`?* По схеме (`第二部分`) на плате стоит ровно эта микросхема. Если выбрать неправильную, Vivado будет ругаться, что назначенные пины не существуют в выбранном корпусе.

## 4. Знакомимся с интерфейсом Vivado

После создания проекта вы видите окно с панелями. Запомните **четыре ключевых места**:

1. **Flow Navigator** — длинный список слева. Здесь все «большие кнопки»: Add Sources, IP Catalog, Create Block Design, Run Synthesis, Run Implementation, Generate Bitstream, Open Hardware Manager.
2. **Sources** — посередине вверху. Дерево файлов: Design Sources (RTL), Constraints (XDC), Simulation Sources.
3. **Project Manager → Settings** — общие настройки проекта (язык по умолчанию, target language, симуляторы).
4. **Tcl Console** — внизу. *Зачем она нам?* В неё Vivado пишет каждое действие как Tcl-команду; вы можете копировать и вставлять команды, чтобы не тыкать мышкой.

## 5. Добавляем RTL-исходники

В этом проекте мы используем готовый **I2C-мастер** из папки `rtl/` репозитория.

1. В **Flow Navigator** → **Project Manager** → **Add Sources**.
2. В мастере: **Add or create design sources** → **Next**.
3. Кнопка **Add Files…** → выберите по очереди (Ctrl+клик):
   - `rtl/i2c_master_core.v` — низкоуровневое ядро.
   - `rtl/i2c_master_axi.v` — обёртка с регистрами AXI.
   - `rtl/i2c_master_top.v` — top с tri-state на SDA/SCL (если хотите вариант с `inout`-портами).
4. ⚠️ Если вместо AXI выберете **прямое управление ядром из PL**, добавьте также `rtl/i2c_burst_writer.v`.
5. **Copy sources into project:** ❌ снимите галочку. *Почему?* Тогда Vivado будет читать файлы по ссылке из репозитория, и ваши правки в редакторе сразу синхронизируются с git.
6. **Add directories:** не используем — мы уже добавили нужные файлы.
7. **Finish**.

В дереве **Sources → Design Sources** должны появиться все три модуля. Vivado автоматически определит, что **верхним модулем** является тот, у которого нет «родителя» в иерархии. Если он угадал неверно, кликните правой кнопкой по нужному модулю → **Set as Top**.

> *Аналогия:* «верхний модуль» — это как `main()` в C: точка входа, к которой будут привязаны физические выводы FPGA.

## 6. Создаём свой top-модуль (если идём по пути «только PL»)

Готовый `i2c_master_top.v` имеет **AXI-интерфейс на входе**. Если вы НЕ хотите подключать процессор, удобнее написать **свой top**, который:

- принимает `clk` от кварца,
- содержит простой автомат (или `i2c_burst_writer`),
- выводит наружу только пины OLED и кнопок.

Создание файла:

1. **Flow Navigator → Add Sources → Add or create design sources** → **Next**.
2. **Create File** → выберите **Verilog**, имя файла `top.v`, **OK** → **Finish**.
3. В появившемся окне **Define Module** просто нажмите **OK** (порты допишем руками).

Минимальный пример «top для проверки светодиодов» (помогает убедиться, что плата вообще прошивается):

```verilog
module top (
    input  wire clk_50m_i,
    input  wire key1_n_i,
    output wire [3:0] led_o
);
    reg [25:0] cnt = 26'd0;
    always @(posedge clk_50m_i) cnt <= cnt + 26'd1;
    assign led_o = cnt[25:22];   // Светодиоды мигают разным темпом
endmodule
```

Этот блок просто делит частоту 50 МГц до 1 Гц и зажигает светодиоды бегущей последовательностью — отличный «hello world» перед тем, как лезть в I2C.

## 7. Block Design + Zynq PS (для пути PS+PL, продвинутый сценарий)

Этот раздел нужен, если вы хотите управлять I2C-мастером из программы на ARM, например через драйвер UIO или Linux. Если вы только начинаете — пропустите и идите в раздел 8.

1. **Flow Navigator → IP Integrator → Create Block Design** → имя `system` → **OK**.
2. В пустом холсте кликните **+ (Add IP)** → ищите `ZYNQ7 Processing System` → двойной клик.
3. Сверху появится зелёная подсказка **«Run Block Automation»** — нажмите. Vivado сам подгрузит **board-preset**? Для нашей платы пресета нет — оставьте Default, но поправьте параметры:
   - **PS-PL Configuration → AXI Non Secure Enablement → GP Master AXI Interface** → ✅ Enable.
   - **DDR Configuration** — если плата имеет DDR3 (на ZYNQ MINI он есть): выберите тип чипа из даташита.
   - **Clock Configuration → PL Fabric Clocks → FCLK_CLK0** = 50 или 100 МГц (выберите ту, под которую считали PRESCALE).
4. Добавьте свой IP:
   - **+** → введите имя модуля `i2c_master_axi` (Vivado автоматически индексирует ваши Verilog-файлы как «Module Reference»). Если не появляется — щёлкните правой кнопкой по модулю в **Sources** → **Add Module to Block Design**.
5. **Run Connection Automation** → Vivado соединит порты AXI, тактирование и сброс. На SDA/SCL связи всё ещё надо вытащить наружу:
   - правой кнопкой по `sda_io` → **Make External**, аналогично `scl_io`.
6. **Validate Design** (F6) — должно быть `Validation successful`.
7. В **Sources** правый клик по `system.bd` → **Create HDL Wrapper** → **Let Vivado manage** → **OK**.
8. Назначьте этот wrapper как **top**.

## 8. Создаём файл ограничений (XDC)

**XDC** = Xilinx Design Constraints. Это текстовый файл с командами Tcl, который говорит:

- **«какой порт где физически расположен»** (`set_property PACKAGE_PIN`),
- **«в каком стандарте напряжения он работает»** (`set_property IOSTANDARD`),
- **«какая частота на тактовом входе»** (`create_clock`).

Без XDC битстрим не соберётся: Vivado откажется наугад выбирать пины.

### 8.1 Создание файла

1. **Flow Navigator → Add Sources → Add or create constraints** → **Next**.
2. **Create File** → имя `pins.xdc` → **OK** → **Finish**.
3. В **Sources → Constraints → constrs_1 → pins.xdc** двойной клик — откроется текстовый редактор Vivado.

### 8.2 Готовый XDC под имена портов из примера

Если ваш top называется `top` и имеет порты `clk_50m_i`, `key1_n_i`, `led_o[3:0]`, **`oled_sda_io`, `oled_scl_io`, `oled_dc_o`, `oled_rst_o`** — вставьте это:

```tcl
# ---------------------------------------------------------------
# Системный такт 50 МГц с генератора X1 (PL_CLK_50M → пин K17)
# ---------------------------------------------------------------
set_property PACKAGE_PIN K17 [get_ports clk_50m_i]
set_property IOSTANDARD LVCMOS33 [get_ports clk_50m_i]
create_clock -name clk_50m -period 20.000 [get_ports clk_50m_i]

# ---------------------------------------------------------------
# Кнопка KEY1 (FPGA_PL_KEY1 → пин M20). Активный уровень — низкий.
# ---------------------------------------------------------------
set_property PACKAGE_PIN M20 [get_ports key1_n_i]
set_property IOSTANDARD LVCMOS33 [get_ports key1_n_i]

# ---------------------------------------------------------------
# Светодиоды LED1..LED4 (FPGA_PL_LED1..4)
# ---------------------------------------------------------------
set_property PACKAGE_PIN T12 [get_ports {led_o[0]}]
set_property PACKAGE_PIN U12 [get_ports {led_o[1]}]
set_property PACKAGE_PIN V12 [get_ports {led_o[2]}]
set_property PACKAGE_PIN W13 [get_ports {led_o[3]}]
set_property IOSTANDARD LVCMOS33 [get_ports {led_o[*]}]

# ---------------------------------------------------------------
# OLED-дисплей (J4) — режим I2C на модуле SSD1306
# OLED_D0 → SDA → пин E18
# OLED_D1 → SCL → пин E19
# OLED_DC          → пин F16  (логический GPIO)
# OLED_RST         → пин F17  (логический GPIO)
# ---------------------------------------------------------------
set_property PACKAGE_PIN E18 [get_ports oled_sda_io]
set_property PACKAGE_PIN E19 [get_ports oled_scl_io]
set_property PACKAGE_PIN F16 [get_ports oled_dc_o]
set_property PACKAGE_PIN F17 [get_ports oled_rst_o]
set_property IOSTANDARD LVCMOS33 [get_ports {oled_sda_io oled_scl_io oled_dc_o oled_rst_o}]

# I2C-линии — открытый сток. Внутри FPGA реализуется через oen,
# но иногда полезно явно поставить PULLUP, если внешний резистор
# на модуле слабый или отсутствует.
set_property PULLUP TRUE [get_ports oled_sda_io]
set_property PULLUP TRUE [get_ports oled_scl_io]
```

### 8.3 Как читать каждую строку

| Конструкция | Что делает |
|-------------|------------|
| `set_property PACKAGE_PIN K17 [get_ports clk_50m_i]` | «Привяжи логический порт `clk_50m_i` к физическому шарику `K17` корпуса CLG400» |
| `set_property IOSTANDARD LVCMOS33 [get_ports …]` | «Этот порт работает в стандарте 3,3 В CMOS» (соответствует напряжению банка из схемы) |
| `create_clock -name clk_50m -period 20.000 [get_ports clk_50m_i]` | «На входе `clk_50m_i` сидит периодический сигнал с периодом 20 нс ⇒ частота 50 МГц». Это нужно временному анализатору, чтобы он мог проверить, успевает ли логика |
| `set_property PULLUP TRUE …` | «Включи внутренний подтягивающий резистор к VCC». Полезно для I2C, где линии всегда подтянуты вверх |

> *Внимание:* если ваш top использует другие имена портов — поменяйте их в `[get_ports …]`. Vivado чувствителен к регистру и квадратным скобкам для битов вектора (`led_o[0]`, не `led_o(0)`).

## 9. Запускаем синтез

**Синтез** = превращение Verilog в граф из примитивов (LUT, регистры, IO-буферы). Результат — ещё не битстрим, но уже не текст.

1. В **Flow Navigator → Synthesis** нажмите **Run Synthesis**.
2. Появится диалог *Launch Runs* — оставьте всё по умолчанию (Number of jobs = столько ядер CPU, сколько у вас) → **OK**.
3. В правом верхнем углу видна полоса прогресса. Обычно занимает **30–120 секунд** для нашего проекта.
4. По завершении Vivado спросит «Synthesis Completed → Open Synthesized Design / Run Implementation / View Reports». Выберите **Open Synthesized Design**.
5. Откроется окно **Schematic** и **Reports**. Полезно посмотреть:
   - **Reports → Utilization** — сколько LUT/FF/BRAM съел дизайн;
   - **Tcl Console** на ошибки и предупреждения (`WARNING:` тоже стоит читать).

> *Если синтез не пошёл:* читайте красные сообщения в **Messages**. Чаще всего — опечатка в имени порта или пина в XDC.

## 10. Имплементация и временной анализ

**Implementation** = размещение и трассировка: Vivado физически размещает примитивы по реальным ячейкам кристалла и соединяет их.

1. **Flow Navigator → Implementation → Run Implementation** → **OK**.
2. По завершении — **Open Implemented Design**.
3. Проверьте отчёт **Reports → Timing → Report Timing Summary**:
   - **WNS (Worst Negative Slack)** — должен быть **положительным** (или 0). Если отрицательный, дизайн не успевает на 50 МГц — это редкость для I2C, но проверить надо.
   - **Setup / Hold violations** — обоих должно быть **0**.

> *Аналогия:* WNS — это «запас по времени». Если он положительный — поезд приехал раньше, всё ок. Если отрицательный — опоздал, поезд уехал без вас, дизайн не работает.

## 11. Генерируем битстрим

**Bitstream** (`*.bit`) — двоичный файл, который заливается в память конфигурации FPGA и «оживляет» вашу схему.

1. **Flow Navigator → Program and Debug → Generate Bitstream** → **OK**.
2. Время генерации — обычно меньше минуты.
3. По завершении выберите **Open Hardware Manager**.

Файл битстрима теперь лежит в `<имя_проекта>.runs/impl_1/<top>.bit`.

---

## 12. Подключаем плату и заливаем битстрим

1. Соедините плату с компьютером **USB-кабелем** на порту JTAG (на ZYNQ MINI это обычно отдельный микро-USB или type-C, в схеме это PR13/CH340/FT232HL — конкретный разъём смотрите в инструкции к плате).
2. Подайте питание (часто питание идёт по тому же USB).
3. Включите плату. Светодиод **DONE** на плате не должен гореть до прошивки.
4. В Vivado **Flow Navigator → Open Hardware Manager** → **Open Target → Auto Connect**.
5. Должно появиться устройство `xc7z010_1`. Кликните по нему правой кнопкой → **Program Device…**.
6. В поле **Bitstream file** Vivado сам подставит ваш `.bit`. Нажмите **Program**.
7. Через ~3 секунды светодиод **DONE** загорится. Если у вас в дизайне есть мигающие LED — они начнут мигать.

> *Если плата не определяется:* проверьте **Cable Drivers** (см. п. 0.2), кабель и питание. На Linux может потребоваться запуск `install_drivers` от root.

## 13. Проверка работы OLED

1. Снова **физическая проверка**: на разъёме `J4` должно быть **3,3 В** между VCC и GND. Меряйте мультиметром.
2. Подключите модуль OLED к `J4` через шлейф. Перепроверьте, что **SDA-линия модуля** идёт на `OLED_D0` (E18 у FPGA), **SCL-линия** — на `OLED_D1` (E19).
3. После прошивки в FPGA должна крутиться ваша инициализация SSD1306 + вывод тестового кадра. На дисплее появится изображение.
4. **Ничего не показывает?** Возможные причины:
   - Модуль в режиме **SPI**, а не **I2C** — проверьте перемычки **BS0–BS2** на самом модуле OLED.
   - Неверный адрес слейва — попробуйте `0x3C` и `0x3D`.
   - Не подан сигнал **`OLED_RST`** в `1` после старта (нужно подержать `0` минимум 10 мс при reset, потом `1`).
   - Не настроен прескалер: посчитайте `PRESCALE` под вашу частоту (см. раздел 14).

> *Лучший друг отладчика — осциллограф на SDA/SCL.* Если на линиях вообще нет «зубцов», значит I2C-мастер не запустился. Если есть — но дисплей молчит, проблема выше: адрес, последовательность инициализации или RST.

## 14. Прескалер I2C — формула с примерами

Регистр `PRESCALE` ядра задаёт делитель тактовой частоты. Полный период SCL = **4 такта** `ena_i`, а `ena_i` импульсирует раз в `PRESCALE+1` тактов системной частоты:

\[
f_{\mathrm{SCL}} = \frac{f_{\mathrm{clk}}}{4 \cdot (\mathrm{PRESCALE} + 1)}
\]

Решая обратно:

\[
\mathrm{PRESCALE} = \frac{f_{\mathrm{clk}}}{4 \cdot f_{\mathrm{SCL}}} - 1
\]

| Системный `clk` | Желаемая SCL | Расчёт | Значение `PRESCALE` |
|------------------|--------------|--------|---------------------|
| 50 МГц | 100 кГц | 50000000/(4·100000)−1 | **124** |
| 50 МГц | 400 кГц | 50000000/(4·400000)−1 | **30** (округление) |
| 100 МГц (FCLK0 от PS) | 100 кГц | 100000000/(4·100000)−1 | **249** |
| 100 МГц | 1 МГц | 100000000/(4·1000000)−1 | **24** |

⚠️ SSD1306 официально поддерживает **до 400 кГц** в режиме I2C; не разгоняйте выше без необходимости.

## 15. Часто задаваемые вопросы (FAQ)

**В: Можно ли использовать переключатель Boards вместо Parts при создании проекта?**  
О: Если у вас есть `*.xml`-файл этой платы (например, скачанный с GitHub `alinxalinx/AX7010_2023.1`), да. Для чистого старта удобнее **Parts** — без зависимостей от файлов производителя.

**В: Почему именно `LVCMOS33`, а не `LVCMOS18`?**  
О: По схеме *第二部分* банк `BANK 34/35 (FPGA)` запитан напряжением **3,3 В**. Если задать `LVCMOS18`, Vivado откажется размещать порт: «conflict in IO standard».

**В: Можно ли поставить несколько `IOSTANDARD` для одного порта?**  
О: Нет. Один порт — один стандарт. Внутри банка можно смешивать разные стандарты только в пределах совместимых с напряжением банка.

**В: А что насчёт `OLED_DC` — куда его подключать?**  
О: В *чистом* I2C-режиме SSD1306 у некоторых модулей он не используется (фиксирован на модуле резистором), у некоторых задаёт младший бит I2C-адреса (0x3C ↔ 0x3D). Если он заведён на разъём `J4`, держите его в RTL как обычный output, ставьте уровень по даташиту вашего модуля.

**В: Откуда брать порядок шагов SSD1306-инициализации?**  
О: Из даташита SSD1306 (раздел *Power-On Sequence*). Готовая последовательность есть в `quartus_ssd1306/src/ssd1306_ctrl.v` этого репозитория — её можно перенести в Vivado-проект «как есть».

**В: Почему я вижу `WARNING: Bus skew constraint not met`?**  
О: Для I2C на 100 кГц это нестрашно — шина медленная. Главное, чтобы Setup/Hold = 0.

## 16. Краткий чек-лист на одну страницу

1. Установить Vivado 2025.2 + cable drivers.
2. Create Project → Parts → `xc7z010clg400-1`.
3. Add Sources: `i2c_master_top.v`, `i2c_master_axi.v`, `i2c_master_core.v`.
4. (Опц.) Block Design + Zynq PS → AXI GP Master + Module Reference на `i2c_master_axi`.
5. Add Sources → Constraints → `pins.xdc` (по таблице в разделе 8.2).
6. Run Synthesis → Run Implementation → Generate Bitstream.
7. Open Hardware Manager → Auto Connect → Program Device.
8. Подключить OLED к `J4`, проверить 3,3 В, сравнить SDA/SCL.
9. На дисплее появилось изображение → готово!
