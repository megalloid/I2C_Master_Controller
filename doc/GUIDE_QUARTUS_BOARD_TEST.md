# Пошаговое руководство: аппаратный тест I2C EEPROM на FPGA Cyclone IV

## Введение

В этом руководстве мы пройдём весь путь проектирования аппаратного теста для I2C EEPROM на FPGA — от анализа платы и протокола до работающей прошивки. Вы узнаете, как спроектировать систему, которая по нажатию кнопки записывает данные в микросхему памяти EEPROM по шине I2C, читает их обратно и показывает результат на 7-сегментном дисплее и светодиодах.

**Целевая аудитория:** начинающие FPGA-разработчики, знакомые с основами Verilog.

**Что мы создадим:**
- Самодостаточный проект для Quartus без использования процессора
- Тестовый контроллер, который работает автономно в FPGA
- Визуальную индикацию результатов на дисплее и LED

**Плата:** ALINX AX301 v2 (EP4CE6F17C8, Cyclone IV E)

---

## Шаг 0. Анализ задачи — что нам нужно сделать

Прежде чем писать код, нужно чётко сформулировать задачу:

1. **Записать** байт данных в EEPROM по шине I2C
2. **Подождать** пока EEPROM завершит внутреннюю запись (~5 мс)
3. **Прочитать** записанный байт обратно
4. **Сравнить** записанное с прочитанным
5. **Показать** результат человеку

Повторить это для нескольких тестовых паттернов и адресов.

**Ключевой вопрос для новичка:** почему мы делаем это в FPGA, а не на микроконтроллере? Потому что наша цель — протестировать *сам I2C контроллер*, реализованный как RTL-блок в FPGA. Нам нужно убедиться, что аппаратная логика работает корректно на реальном кремнии, а не только в симуляции.

## Шаг 1. Изучение платы — какие ресурсы у нас есть

Это **самый важный** первый шаг в любом FPGA-проекте. Вы не можете написать ни строчки кода, пока не знаете:

- Какая FPGA стоит на плате (семейство, корпус, объём)
- Какая тактовая частота
- Какая периферия доступна и на каких пинах

### Откуда брать информацию

1. **Схематик платы** (PDF) — основной источник истины
2. **Референсные проекты** производителя (`.qsf` файлы с pin assignments)
3. **User manual** платы

Для нашей платы ALINX AX301 информация из `.qsf` файлов:

### Параметры FPGA

```
Семейство:     Cyclone IV E
Устройство:    EP4CE6F17C8
Корпус:        256-pin FBGA
Логические LE: ~6000
Скорость:      Grade 8
```

### Тактовый генератор

```
Частота: 50 МГц
Пин:     PIN_E1
Имя:     clk
```

**Почему это важно:** от тактовой частоты зависят ВСЕ временные расчёты в проекте — скорость I2C, задержки, частота обновления дисплея.

### Доступная периферия

| Периферия | Количество | Активный уровень | Пины |
|-----------|-----------|-------------------|------|
| Светодиоды | 4 | Active-high (1 = горит) | E10, F9, C9, D9 |
| Кнопки | 3 + reset | Active-low (0 = нажата) | M15, M16, E16, N13(rst) |
| 7-сегментный дисплей | 6 разрядов | Active-low (0 = сегмент горит) | seg_sel + seg_data |
| I2C EEPROM (24LC04) | 1 | Open-drain | SCL=D1, SDA=E6 |

### Критическое наблюдение о конфликте пинов

Изучая `.qsf` файлы разных демо-проектов, обнаруживается важная вещь: в проекте `seg_test` для 8-го разряда дисплея (`seg_sel[7:6]`) используются пины **D1 и E6** — те же самые, что для **I2C SCL и SDA**!

```
seg_sel[6] → PIN_E6  = i2c_sda  ← КОНФЛИКТ!
seg_sel[7] → PIN_D1  = i2c_scl  ← КОНФЛИКТ!
```

**Вывод:** если мы используем I2C, у нас доступны только **6 разрядов** дисплея, а не 8. Это архитектурное ограничение платы, которое нужно учесть сразу.

## Шаг 2. Изучение протокола I2C — как разговаривать с EEPROM

### Физический уровень: open-drain

I2C — это двухпроводная шина:
- **SCL** (Serial Clock) — тактовый сигнал, генерируется мастером
- **SDA** (Serial Data) — данные, двунаправленная линия

Обе линии работают в режиме **open-drain** (открытый сток):

```
     VCC (3.3V)
      │
     ┌┴┐
     │R│ pull-up 4.7 кОм
     └┬┘
      ├──────── Шина SDA/SCL
      │
  ┌───┴───┐
  │ FPGA  │  Может только ПРИТЯНУТЬ К ЗЕМЛЕ или ОТПУСТИТЬ
  │ (OD)  │  Не может выдать единицу — за это отвечает pull-up
  └───────┘
```

**Важно для FPGA:** мы не выдаём `1` на линию. Мы либо выдаём `0` (притягиваем к земле), либо переводим пин в состояние `Z` (высокий импеданс), и внешний резистор подтягивает линию к `VCC`.

В Verilog это выглядит так:

```verilog
// oen = 1 → отпустить линию (Z, подтянется к VCC через pull-up)
// oen = 0 → притянуть к земле (выдать 0)
assign i2c_scl = scl_oen ? 1'bz : 1'b0;
assign i2c_sda = sda_oen ? 1'bz : 1'b0;
```

### Условия START и STOP

I2C-транзакция начинается с условия **START** и заканчивается **STOP**:

```
        START                         STOP
SDA: ───╲                               ╱───
         ╲__                          __╱
SCL: ─────────╲    ...данные...   ╱─────────
               ╲_________________╱

START: SDA падает, пока SCL = 1
STOP:  SDA растёт, пока SCL = 1
```

Во время передачи данных SDA может меняться **только когда SCL = 0**. Если SDA меняется при SCL = 1 — это либо START, либо STOP.

### Формат транзакции

Каждый байт передаётся по 8 бит (MSB first), за ним следует 1 бит подтверждения (ACK/NACK):

```
    S | A6 A5 A4 A3 A2 A1 A0 RW | ACK | D7 D6 D5 D4 D3 D2 D1 D0 | ACK | P
    ↑                            ↑                                  ↑     ↑
  START    Адрес + R/W бит     Slave    Данные                    Slave  STOP
                                 подтверждает                    подтверждает
```

- **ACK (acknowledge)** = slave притягивает SDA к 0 → «я тут, принял»
- **NACK** = SDA остаётся 1 → «не получил» или «хватит, больше не надо»

### Протокол для EEPROM 24LC04

**24LC04** — это I2C EEPROM на 512 байт (2 блока по 256).

**Адрес устройства:** `1010 B0 XX R/W` где B0 — номер блока (0 или 1).

Для блока 0:
- Запись: `0xA0` (1010 000 **0**)
- Чтение: `0xA1` (1010 000 **1**)

#### Операция записи байта

```
[START] [0xA0] [ACK] [addr] [ACK] [data] [ACK] [STOP]
   ↓       ↓            ↓            ↓
 Начало  "Я хочу      Куда         Что
         писать в      писать       писать
         EEPROM"
```

После STOP нужно подождать **~5 мс** — EEPROM внутренне записывает данные во flash-ячейки. В это время он не отвечает на запросы.

#### Операция чтения байта (random read)

Чтение сложнее — нужен **Repeated START**:

```
[START] [0xA0] [ACK] [addr] [ACK] [RESTART] [0xA1] [ACK] [data] [NACK] [STOP]
   ↓       ↓            ↓             ↓         ↓            ↓       ↓
 Начало  "Запись"     Адрес       Повторный   "Чтение"    EEPROM  "Спасибо,
                      ячейки       старт       из EEPROM   шлёт     хватит"
                                               (без                 (NACK =
                                               потери               последний
                                               шины)                байт)
```

**Почему Repeated START, а не два отдельных цикла?** Потому что между STOP и новым START другой мастер может захватить шину. Repeated START атомарно переключает направление, не отпуская шину.

**Почему NACK перед STOP при чтении?** Это сигнал EEPROM: «последний байт, больше не посылай». Если послать ACK — EEPROM начнёт выдавать следующий байт.

## Шаг 3. Архитектурное проектирование — декомпозиция системы

Теперь, когда мы понимаем задачу, плату и протокол, нужно разбить систему на модули. Это **ключевой навык** FPGA-разработчика — правильная декомпозиция.

### Принцип: один модуль — одна ответственность

```
┌─────────────────────────────────────────────────────────┐
│                    i2c_test_top                          │
│                                                         │
│  ┌───────────┐   ┌───────────────┐   ┌──────────────┐  │
│  │ Прескалер │──►│ i2c_master    │   │ i2c_test     │  │
│  │           │   │ _core         │◄─►│ _ctrl        │  │
│  └───────────┘   └──────┬────────┘   └──┬───────────┘  │
│                         │               │    │          │
│  ┌───────────┐          │          ┌────┘    │          │
│  │ax_debounce│──────────┼──────────┘    ┌────┘          │
│  └───────────┘          │               │               │
│                    SDA/SCL         ┌────┴─────┐         │
│                         │         │  seg_scan │──►7-seg │
│                         │         └──────────┘         │
│                         │                          LED──►│
└─────────────────────────┼──────────────────────────────┘
                          │
                    ┌─────┴─────┐
                    │  24LC04   │
                    │  EEPROM   │
                    └───────────┘
```

**5 модулей, каждый с чёткой задачей:**

| Модуль | Ответственность | Сложность |
|--------|----------------|-----------|
| `i2c_test_top` | Связать всё вместе, содержит прескалер | Простой |
| `i2c_master_core` | Физический уровень I2C (уже готов) | Сложный (но готовый) |
| `i2c_test_ctrl` | Тестовый сценарий (FSM) | Средний |
| `seg_scan` | Мультиплексирование 7-сегментного дисплея | Простой |
| `ax_debounce` | Подавление дребезга кнопки | Простой |

### Почему именно так?

1. **`i2c_master_core` уже существует** — мы его тестируем, а не пишем заново. Он принимает команды (START, WRITE, READ, STOP) и генерирует I2C-сигналы.

2. **`i2c_test_ctrl` — это «мозг» проекта.** Это конечный автомат (FSM), который знает: «сначала пошли START, потом адрес, потом данные...» Он командует ядром I2C.

3. **`seg_scan` — чистая «механика» дисплея.** Он ничего не знает о тестах — просто берёт 6 наборов сегментных данных и по очереди выводит на дисплей.

4. **`ax_debounce` — решает физическую проблему.** Механическая кнопка при нажатии генерирует серию быстрых замыканий/размыканий (дребезг). Без подавления одно нажатие может быть воспринято как 10-50 нажатий.

## Шаг 4. Прескалер — задаём скорость I2C

### Зачем нужен прескалер

Наше I2C ядро (`i2c_master_core`) тактируется основным клоком 50 МГц, но работает медленнее — получает периодические «тики» (сигнал `ena_i`). Каждый тик продвигает внутренний конечный автомат на один шаг.

I2C-бит состоит из **4 фаз** (четверть периода SCL каждая):

```
SCL: ──┐     ┌─────┐     ┌──
       │     │     │     │
       └─────┘     └─────┘
       ↑     ↑     ↑     ↑
     Фаза0 Фаза1 Фаза2 Фаза3
     (SCL↓) (SCL↑) (SCL↑) (SCL↓)
```

Значит: **частота тиков = 4 × частота SCL**.

### Формула

```
SCL = f_clk / (4 × (PRESCALE + 1))
```

Для Standard Mode (100 кГц):

```
PRESCALE = f_clk / (4 × SCL) - 1 = 50 000 000 / (4 × 100 000) - 1 = 124
```

Проверка: 50 000 000 / (4 × 125) = 100 000 Гц ✓

### Реализация

```verilog
localparam [15:0] PRESCALE = 16'd124;

reg [15:0] pre_cnt;
reg        core_ena;

always @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
        pre_cnt  <= 16'd0;
        core_ena <= 1'b0;
    end else if (pre_cnt == 16'd0) begin
        pre_cnt  <= PRESCALE;       // Перезагрузить счётчик
        core_ena <= 1'b1;           // Один такт — тик!
    end else begin
        pre_cnt  <= pre_cnt - 16'd1; // Считаем вниз
        core_ena <= 1'b0;            // Тик только один такт
    end
end
```

**Как это работает:**
1. Счётчик считает от 124 до 0
2. Когда достигает 0 — выдаёт одноцикловый импульс `core_ena = 1`
3. Перезагружается и начинает заново

Результат: импульс каждые 125 тактов = 400 кГц = 4 × 100 кГц SCL.

## Шаг 5. Подавление дребезга кнопки

### Проблема

Когда вы нажимаете механическую кнопку, контакт не замыкается мгновенно. В течение ~10-20 мс контакт «прыгает»:

```
Реальный сигнал кнопки:
────┐ ┌┐ ┌┐ ┌─────────────────────   (нажата)
    └─┘└─┘└─┘

Желаемый сигнал:
────┐
    └─────────────────────────────   (чистый переход)
```

При 50 МГц эти «прыжки» выглядят как десятки нажатий.

### Решение: цифровой debouncer

Алгоритм:
1. Засечь изменение уровня на входе
2. Запустить таймер (20 мс)
3. Если за 20 мс вход не менялся — принять новое значение
4. Если вход снова изменился — сбросить таймер

```verilog
// Таймер: 20 мс × 1000 мкс × 50 МГц = 1 000 000 тактов
localparam TIMER_MAX_VAL = 20 * 1000 * 50;  // = 1_000_000
```

Модуль `ax_debounce` выдаёт три полезных сигнала:
- `button_out` — стабильное состояние кнопки
- `button_negedge` — одноцикловый импульс при **нажатии** (переход 1→0)
- `button_posedge` — одноцикловый импульс при **отпускании** (переход 0→1)

Нам нужен `button_negedge` — момент нажатия кнопки = команда «старт».

## Шаг 6. 7-сегментный дисплей — динамическая индикация

### Принцип работы

На плате 6 отдельных 7-сегментных индикаторов, но управляются они по принципу **мультиплексирования** — в каждый момент времени горит **только один** индикатор. Переключение происходит так быстро (>100 Гц), что глаз видит все 6 разрядов одновременно.

```
Время →
           ┌──┐      ┌──┐      ┌──┐
Разряд 0:  │ON│      │  │      │  │  ...
           └──┘      └──┘      └──┘
               ┌──┐      ┌──┐
Разряд 1:  │  ││ON│  │  ││  │  ...
               └──┘      └──┘
                    ┌──┐
Разряд 2:  │  ││  ││ON│  ...
                    └──┘
...и так далее для всех 6 разрядов
```

### Управляющие сигналы

- **`seg_sel[5:0]`** — выбор разряда (active-low: `0` = разряд активен)
- **`seg_data[7:0]`** — какие сегменты горят (active-low: `0` = сегмент горит)

Пример: включить только разряд 0, показать цифру «3»:
```
seg_sel  = 6'b111_110   (активен разряд 0)
seg_data = {1'b1, 7'b011_0000}   (сегменты a,b,c,d,g горят)
```

### Кодировка сегментов

Расположение сегментов стандартное:

```
   aaaa
  f    b
  f    b
   gggg
  e    c
  e    c
   dddd  .dp
```

Битовое поле: `{dp, g, f, e, d, c, b, a}` — бит 7 = dp, бит 0 = a.

Active-low означает `0` = горит:

```
Цифра 0: сегменты a,b,c,d,e,f → 1_000_0000 = 0x40 (g и dp выключены)
Цифра 1: сегменты b,c         → 1_111_1001 = 0x79
...
```

### Частота сканирования

Для отсутствия мерцания нужно обновлять каждый разряд > 100 раз/сек. С 6 разрядами:

```
Частота переключения = 200 Гц × 6 = 1200 Гц
Период на разряд = 50 000 000 / 1200 ≈ 41 666 тактов
```

### Реализация

```verilog
localparam SCAN_MAX = CLK_FREQ / (SCAN_FREQ * 6) - 1;  // ≈ 41 666

reg [31:0] scan_cnt;
reg [2:0]  scan_idx;  // 0..5

always @(posedge clk) begin
    if (scan_cnt >= SCAN_MAX) begin
        scan_cnt <= 0;
        scan_idx <= (scan_idx == 5) ? 0 : scan_idx + 1;
    end else
        scan_cnt <= scan_cnt + 1;
end

always @(posedge clk) begin
    case (scan_idx)
        0: begin seg_sel <= 6'b111_110; seg_data <= digit_0_data; end
        1: begin seg_sel <= 6'b111_101; seg_data <= digit_1_data; end
        // ...и так для всех 6 разрядов
    endcase
end
```

**Принцип:** счётчик `scan_cnt` считает до ~41666, потом переключает `scan_idx` на следующий разряд. В зависимости от `scan_idx` выбирается один из 6 наборов данных.

## Шаг 7. Тестовый контроллер — сердце проекта

Это самый сложный модуль. Он управляет всем процессом тестирования.

### Концепция: двухуровневый конечный автомат

**Уровень 1 (Main FSM)** — «что мы делаем»:

```
IDLE → PREP → WR_SEQ → WR_DELAY → RD_SEQ → VERIFY → SHOW → NEXT → SUMMARY
  ↑                                                             │
  └─────────────────────────────────────────────────────────────┘
                     (следующий тест или конец)
```

**Уровень 2 (Sub FSM)** — «как мы общаемся с I2C ядром»:

```
ISSUE → ACCEPT → DONE → (переход к следующему шагу)
```

### Зачем два уровня?

I2C ядро имеет протокол «рукопожатия» (handshake):
1. Контроллер выставляет команду (`cmd_valid = 1`)
2. Ядро принимает (`ready` падает в 0)
3. Ядро выполняет (это занимает много тактов)
4. Ядро завершает (`ready` возвращается в 1)

Этот протокол одинаков для КАЖДОЙ команды (START, WRITE, READ, STOP). Вместо того чтобы дублировать код, мы выносим его в под-состояния:

```verilog
// Под-состояния (одинаковые для всех команд)
SS_ISSUE:  // Выдать команду, пока ядро готово
    if (ready) begin
        cmd_valid <= 1;
        cmd <= нужная_команда;
        din <= нужные_данные;
        sub <= SS_ACCEPT;
    end

SS_ACCEPT:  // Ждём, пока ядро примет команду
    if (!ready) begin
        cmd_valid <= 0;
        sub <= SS_DONE;
    end

SS_DONE:  // Ждём завершения, проверяем результат
    if (ready) begin
        // Команда выполнена, можно проверять ACK/данные
        // Переходим к следующему шагу
    end
```

### Таблица команд записи

Запись байта в EEPROM — это **5 I2C-команд** подряд:

```
Step │ Команда     │ Данные     │ Проверка ACK │ Смысл
─────┼─────────────┼────────────┼──────────────┼──────────────────────
  0  │ START       │ —          │ Нет          │ Начало транзакции
  1  │ WRITE       │ 0xA0       │ Да           │ Адрес slave (запись)
  2  │ WRITE       │ reg_addr   │ Да           │ Адрес ячейки EEPROM
  3  │ WRITE       │ data       │ Да           │ Данные для записи
  4  │ STOP        │ —          │ Нет          │ Конец транзакции
```

### Таблица команд чтения

Чтение байта — **7 I2C-команд**:

```
Step │ Команда     │ Данные     │ Проверка ACK │ Смысл
─────┼─────────────┼────────────┼──────────────┼──────────────────────
  0  │ START       │ —          │ Нет          │ Начало
  1  │ WRITE       │ 0xA0       │ Да           │ Адрес slave (запись)
  2  │ WRITE       │ reg_addr   │ Да           │ Адрес ячейки
  3  │ RESTART     │ —          │ Нет          │ Повторный старт
  4  │ WRITE       │ 0xA1       │ Да           │ Адрес slave (чтение)
  5  │ READ        │ NACK=1     │ Нет          │ Читаем байт (NACK=последний)
  6  │ STOP        │ —          │ Нет          │ Конец
```

### Реализация через «пошаговый исполнитель»

Вместо того чтобы создавать отдельное состояние FSM для каждого шага, мы используем **счётчик шагов** (`step`) и **комбинаторные мультиплексоры**, которые выбирают команду:

```verilog
// Мультиплексор команд записи (комбинаторная логика)
always @(*) begin
    case (step)
        0: begin wr_cmd = START; wr_din = 8'h00;    end
        1: begin wr_cmd = WRITE; wr_din = 8'hA0;    end  // slave addr
        2: begin wr_cmd = WRITE; wr_din = tst_addr;  end  // reg addr
        3: begin wr_cmd = WRITE; wr_din = tst_data;  end  // data
        4: begin wr_cmd = STOP;  wr_din = 8'h00;     end
    endcase
end
```

Это **паттерн «микропрограмма»** — таблица команд, индексируемая счётчиком. Очень мощный приём для проектирования FSM.

### Проверка ACK

После каждого WRITE мы проверяем, ответил ли slave сигналом ACK:

```verilog
if (wr_check_ack && rx_ack) begin
    // rx_ack = 1 означает NACK — slave не ответил!
    test_error <= 1'b1;
    state <= ST_ERR_STOP;  // Послать STOP и пометить ошибку
end
```

**Типичные причины NACK:**
- EEPROM не установлен на плате
- Неверный адрес slave
- EEPROM ещё занят предыдущей записью (не выдержали 5 мс)
- Нет pull-up резисторов на шине

### Задержка записи EEPROM

После STOP (конец записи) EEPROM нужно время на внутреннюю запись. Datasheet 24LC04 говорит: максимум 5 мс.

```verilog
// 6 мс = 300 000 тактов при 50 МГц (с запасом)
parameter WR_TICKS = 300_000;

ST_WR_DELAY: begin
    if (delay_cnt >= WR_TICKS)
        state <= ST_RD_SEQ;   // Переходим к чтению
    else
        delay_cnt <= delay_cnt + 1;
end
```

### Верификация

После чтения сравниваем:

```verilog
ST_VERIFY: begin
    if (rd_data != tst_data)
        test_error <= 1'b1;   // Несовпадение!
    state <= ST_SHOW;
end
```

### Тестовые паттерны (Test ROM)

Выбираем 4 паттерна, которые покрывают разные случаи:

```verilog
case (test_idx)
    0: begin addr = 8'h00; data = 8'hA5; end  // Чередующиеся биты
    1: begin addr = 8'h01; data = 8'h5A; end  // Инверсия предыдущего
    2: begin addr = 8'h10; data = 8'hFF; end  // Все единицы
    3: begin addr = 8'h11; data = 8'h00; end  // Все нули
endcase
```

**Почему именно эти?**
- `0xA5` = `10100101` — чередование бит, ловит залипание соседних линий
- `0x5A` = `01011010` — инверсия, проверяет что можно переписать данные
- `0xFF` и `0x00` — граничные значения, ловят проблемы с pull-up/pull-down

## Шаг 8. Управление дисплеем из контроллера

### Проблема: отображение hex-данных и спецсимволов

На дисплее нужно показать:
- Цифры 0-9 и A-F (hex)
- Символ `-` (тире) — тест в процессе
- Символ `P` — Pass (пройден)
- Символ `F` — Fail (провален)
- Пробел (пустой разряд)

### Решение: функция-конвертер + константы

Для hex-цифр используем функцию:

```verilog
function [7:0] seg_hex;
    input [3:0] h;
    // Возвращает 8 бит: {DP_off, gfedcba}
    // Segment ON = 0, OFF = 1 (active-low)
endfunction
```

Для спецсимволов — константы:

```verilog
localparam [7:0] S_BLANK = 8'hFF;  // Всё выключено
localparam [7:0] S_DASH  = 8'hBF;  // Только сегмент g (тире)
localparam [7:0] S_P     = 8'h8C;  // a, b, e, f, g
localparam [7:0] S_F     = 8'h8E;  // a, e, f, g
```

### Формат дисплея

```
Позиция: [5] [4] [3] [2] [1] [0]    (5 = левый, 0 = правый)

IDLE:     [-] [-] [-] [-] [-] [-]

Тест:     [N] [-] [W] [W] [-] [-]    (N=номер, WW=записанные данные)

Результат:[N] [P] [W] [W] [R] [R]    (P/F, RR=прочитанные данные)

Итог:     [P] [n] [ ] [F] [n] [ ]    (n=количество)
```

Контроллер напрямую выдаёт 8-битные данные для каждого разряда:

```verilog
// Пример: показать результат теста
seg5 <= seg_hex(test_num);      // Номер теста
seg4 <= test_error ? S_F : S_P; // Pass или Fail
seg3 <= seg_hex(wr_data[7:4]);  // Старший hex-ниббл записанного
seg2 <= seg_hex(wr_data[3:0]);  // Младший hex-ниббл записанного
seg1 <= seg_hex(rd_data[7:4]);  // Старший hex-ниббл прочитанного
seg0 <= seg_hex(rd_data[3:0]);  // Младший hex-ниббл прочитанного
```

## Шаг 9. Top-level модуль — собираем всё вместе

Top-level — это «сборочный цех», где все модули соединяются проводами.

### Принцип: минимум логики в top-level

В top-level должно быть:
1. Инстанциации подмодулей
2. Проводные соединения (`wire`)
3. Простая логика, которая не заслуживает отдельного модуля (прескалер, tri-state буферы)

### Порты модуля = физические пины FPGA

```verilog
module i2c_test_top (
    input  wire       clk,       // ← Физический пин E1 (осциллятор 50 МГц)
    input  wire       rst_n,     // ← Физический пин N13 (кнопка reset)
    input  wire       key1,      // ← Физический пин M15 (кнопка start)
    output wire [3:0] led,       // → Физические пины E10,F9,C9,D9
    output wire [5:0] seg_sel,   // → Физические пины N9..M11
    output wire [7:0] seg_data,  // → Физические пины R14..R16
    inout  wire       i2c_sda,   // ↔ Физический пин E6 (bidirectional!)
    inout  wire       i2c_scl    // ↔ Физический пин D1 (bidirectional!)
);
```

Обратите внимание на `inout` — это **двунаправленные** пины. FPGA может и читать с них, и записывать. Для I2C это необходимо: мастер и slave по очереди управляют линиями SDA/SCL.

### Соединение модулей

Каждый модуль «видит» только свои входы и выходы. Задача top-level — правильно их связать:

```
[Кнопка] → [Debounce] → btn_negedge → [Test Ctrl] → cmd/din → [I2C Core] → SDA/SCL
                                            ↓
                                       seg_dig[5:0] → [Seg Scan] → seg_sel/seg_data
                                            ↓
                                          led[3:0]
```

## Шаг 10. Quartus-проект — привязка к железу

### Файл .qsf — «паспорт» проекта

Этот файл содержит **критически важную** информацию:

**1. Какая FPGA:**
```tcl
set_global_assignment -name FAMILY "Cyclone IV E"
set_global_assignment -name DEVICE EP4CE6F17C8
```

**2. Какие файлы компилировать:**
```tcl
set_global_assignment -name VERILOG_FILE src/i2c_test_top.v
set_global_assignment -name VERILOG_FILE src/i2c_test_ctrl.v
set_global_assignment -name VERILOG_FILE ../rtl/i2c_master_core.v
```

**3. Стандарт ввода/вывода для каждого пина:**
```tcl
set_instance_assignment -name IO_STANDARD "3.3-V LVTTL" -to clk
set_instance_assignment -name IO_STANDARD "3.3-V LVTTL" -to i2c_sda
```

**4. Физическая привязка пинов (pin assignment):**
```tcl
set_location_assignment PIN_E1  -to clk
set_location_assignment PIN_M15 -to key1
set_location_assignment PIN_D1  -to i2c_scl
set_location_assignment PIN_E6  -to i2c_sda
```

**Внимание!** Ошибка в pin assignment может **повредить FPGA** (если подключить выход к питанию) или просто «ничего не работает» (если перепутать пины). Всегда проверяйте дважды по схематику платы.

### Файл .sdc — временные ограничения

SDC (Synopsys Design Constraints) говорит Quartus о частотах и временных требованиях:

```tcl
# Тактовый сигнал 50 МГц (период 20 нс)
create_clock -name clk -period 20.000 [get_ports {clk}]

# I2C медленный, LED/7-seg не критичны — ослабляем требования
set_false_path -from [get_ports {i2c_sda i2c_scl key1 rst_n}]
set_false_path -to   [get_ports {i2c_sda i2c_scl led[*] seg_sel[*] seg_data[*]}]
```

`set_false_path` сообщает: «не проверяй timing на этих путях». I2C работает на 100 кГц — при 50 МГц тактовой задержки FPGA (единицы наносекунд) ничтожны по сравнению с периодом I2C (10 мкс).

## Шаг 11. Верификация перед заливкой в FPGA

### Синтаксическая проверка (iverilog)

Перед открытием Quartus стоит проверить синтаксис:

```bash
iverilog -g2012 -Wall -o /dev/null \
    rtl/i2c_master_core.v \
    quartus/src/*.v
```

Если выхлоп пустой — ошибок нет.

### Lint (Verilator)

Verilator находит потенциальные проблемы, которые iverilog пропускает:

```bash
verilator --lint-only -Wall --top-module i2c_test_top \
    rtl/i2c_master_core.v \
    quartus/src/*.v
```

Типичные находки:
- **WIDTHEXPAND** — несовпадение разрядности (опасно, может обрезать данные)
- **UNUSEDSIGNAL** — неиспользованный сигнал (обычно безобидно)
- **BLKSEQ** — blocking assignment в sequential-блоке (не всегда ошибка)

### Симуляция (при наличии тестбенча)

В идеале — написать тестбенч с моделью EEPROM и прогнать все сценарии:

```bash
iverilog -g2012 -o sim.vvp rtl/*.v quartus/src/*.v tb/i2c_slave_model.sv tb/board_tb.sv
vvp sim.vvp
gtkwave sim.vcd
```

## Шаг 12. Компиляция и загрузка в FPGA

### Компиляция в Quartus

```bash
cd quartus/
quartus_sh --flow compile i2c_test_top
```

Процесс:
1. **Analysis & Synthesis** — Quartus читает Verilog, строит логическую схему
2. **Fitter** — размещает логику в LE FPGA и прокладывает соединения
3. **Assembler** — генерирует файл прошивки (`.sof`)
4. **Timing Analysis** — проверяет, успевает ли логика за тактовую частоту

### Загрузка через JTAG

```bash
quartus_pgm -m jtag -o "P;output_files/i2c_test_top.sof"
```

Или через GUI: **Tools → Programmer → Start**

**Внимание:** `.sof` — это SRAM-прошивка, она **теряется** при выключении питания. Для постоянной прошивки нужно конвертировать в `.jic` и записать во flash.

## Шаг 13. Тестирование на плате

### Сценарий «всё работает»

1. Подключить USB-Blaster к плате
2. Загрузить прошивку
3. Дисплей показывает `------`
4. Нажать key1
5. По очереди появляются: `1PA5A5`, `2P5A5A`, `3PFFFF`, `4P0000`
6. Все 4 LED горят
7. Итог: `P4  F0`

### Сценарий «EEPROM не отвечает»

1. Дисплей: `1F  EE` (test 1 fail, read data = 0xEE = error marker)
2. LED[0] не горит
3. Итог: `P0  F4`
4. **Действие:** проверить наличие EEPROM на плате, pull-up резисторы

### Сценарий «данные не совпадают»

1. Дисплей: `1FA5xx` где `xx ≠ A5`
2. **Действие:** проверить осциллографом сигналы SDA/SCL, искать проблемы с timing

## Итоговая карта проектирования

```
1. Анализ задачи         → ЧТО делаем
2. Изучение платы        → НА ЧЁМ делаем (пины, частоты, конфликты)
3. Изучение протокола    → КАК общаемся с устройством
4. Декомпозиция          → Разбиваем на модули
5. Прескалер             → Задаём скорость I2C
6. Debouncer             → Фильтруем дребезг кнопки
7. Дисплей               → Визуализация результатов
8. Тестовый контроллер   → FSM — главная логика
9. Top-level             → Соединяем всё вместе
10. Quartus-проект       → Привязка к физическим пинам
11. Верификация          → Lint + симуляция перед заливкой
12. Компиляция/загрузка  → Создание прошивки
13. Тестирование         → Проверка на реальном железе
```

Каждый последующий шаг опирается на предыдущие. Пропуск шагов (например, начать писать код без изучения платы) — главная ошибка новичков.
